## 1. Dedup Internals

The chunk store lives under `System Volume Information\Dedup\` and consists of:

| Component | File Type | Purpose |
|---|---|---|
| **Container files** | `*.ddc` | Store the actual unique data chunks in compressed form |
| **Stream map files** | `*.dds` | Map each file's logical byte ranges to chunk references |
| **Hotspot files** | `*.ddh` | Cache frequently accessed chunks for faster recall |
| **Chunk hash index** | In-memory + on-disk | Maps SHA-256 hashes to container locations for dedup lookup |

**Variable-size chunking (Rabin fingerprinting):**
- Unlike fixed-size chunking, Rabin fingerprinting finds chunk boundaries based on
  content patterns, making dedup resilient to insertions and deletions.
- The algorithm slides a window across the data and computes a rolling hash.
- When the hash matches a boundary condition, a chunk boundary is created.
- Result: chunk sizes vary between **32 KB and 128 KB** (average ~64 KB).

**Hash algorithm:**
- SHA-256 truncated вҖ” each chunk is hashed with SHA-256 and the hash is truncated
  for storage efficiency while maintaining collision resistance.
- Hash collisions are astronomically unlikely but the system uses full content
  verification as a safety net for critical operations.

## 2. Dedup on ReFS vs NTFS

| Feature | NTFS | ReFS (Server 2019+) |
|---|---|---|
| Dedup supported | вң… Yes (Server 2012+) | вң… Yes (Server 2019+) |
| Usage types | Default, HyperV, Backup | Default, HyperV, Backup |
| Integrity streams | вқҢ No | вң… Yes вҖ” built-in checksumming |
| Block cloning | вқҢ No | вң… Yes вҖ” instant file copies |
| Max volume size tested | 64 TB | 64 TB (dedup), volume itself up to PB scale |
| Compression | вң… NTFS + dedup compression | вң… Dedup compression only |
| Encryption (EFS) | вң… Supported | вқҢ Not supported |

**Workload recommendations:**
- **Hyper-V / VDI on S2D:** Prefer ReFS вҖ” block cloning + dedup + integrity streams.
- **General file servers:** NTFS is mature and well-tested; ReFS if you need integrity.
- **Backup targets:** Either works; ReFS avoids bit-rot with integrity streams.

> вҡ пёҸ ReFS dedup requires Windows Server 2019 or later. On Server 2016, ReFS volumes
> cannot be dedup-enabled.

## 3. Dedup Performance Impact

**CPU and memory during optimization:**
- Optimization jobs use **low-priority I/O** and background CPU scheduling.
- Default: jobs consume вүӨ 25 % of CPU and limited memory.
- Impact is generally minimal for off-hours processing, but noticeable on busy volumes.

**Tuning parameters:**

```powershell
# Run optimization with higher resource allocation (percentage values)
Start-DedupJob -Volume D: -Type Optimization -Memory 50 -Cores 75

# Set volume-level throughput optimization
Set-DedupVolume -Volume D: -OptimizePartialFiles $true

# Schedule optimization for off-peak hours only
Set-DedupSchedule -Name "NightlyOptimization" -Type Optimization `
    -Start "22:00" -DurationHours 8 -Days Monday,Tuesday,Wednesday,Thursday,Friday
```

**Best practices:**
- Schedule full optimization during **off-peak hours** (nights, weekends).
- On volumes with heavy daytime I/O, limit `-Cores` to 25 and `-Memory` to 25.
- Monitor with `Get-DedupJob` and performance counters under
  `Dedup\Optimization Jobs Running`, `Dedup\Optimization Rate MB/s`.
- For backup volumes, set `-MinimumFileAge 0` вҖ” backup files are often immediately
  eligible for dedup.

## 4. FSRM Classification Infrastructure

FSRM can automatically **classify** files by assigning metadata properties to them
based on rules. This powers dynamic access control and automated file management.

**Classification rule types:**

| Classifier | How It Works | Example |
|---|---|---|
| **Content Classifier** | Scans file contents for patterns (regex, strings) | Mark files containing "Confidential" as `Impact = High` |
| **Folder Classifier** | Assigns properties based on file location | Files in `\Finance\` get `Department = Finance` |
| **Custom Classifier** | Runs a custom PowerShell script or COM object | Classify by file age, owner, or external DB lookup |

**Setting up classification:**

```powershell
# Step 1: Define a classification property
New-FsrmClassificationPropertyDefinition -Name "Confidentiality" `
    -Type SingleChoice -PossibleValue @("Public","Internal","Confidential")

# Step 2: Create a content classification rule
New-FsrmClassificationRule -Name "Detect Confidential Content" `
    -Namespace "D:\Shares" -Property "Confidentiality" -PropertyValue "Confidential" `
    -ClassificationMechanism "Content Classifier" `
    -ContentRegularExpression "SSN|Social Security|Confidential"

# Step 3: Run classification
Start-FsrmClassification
```

**Integration with Dynamic Access Control (DAC):**
- Classification properties become resource properties in Active Directory.
- Central Access Policies reference these properties for conditional access.
- Example: only allow Finance group access to files classified `Department = Finance`.

## 5. FSRM File Management Tasks

Automate file lifecycle operations based on classification, age, or last access time.

**Task types:**
- **File Expiration** вҖ” move files to an expiration directory after N days.
- **Custom Command** вҖ” run a script or executable against matching files.

```powershell
# Create a file management task: expire files older than 365 days
New-FsrmFileManagementJob -Name "Archive Old Files" `
    -Namespace "D:\Shares\Projects" `
    -Action @{ Type = "Expiration"; ExpirationDirectory = "D:\Archive" } `
    -Condition @{ Property = "File.DateLastAccessed"; Value = "365"; Type = "DateOlderThan" } `
    -Schedule (New-FsrmScheduledTask -Weekly -At "02:00" -DaysOfWeek Saturday)

# List existing file management tasks
Get-FsrmFileManagementJob

# Run a file management task immediately
Start-FsrmFileManagementJob -Name "Archive Old Files"
```

**Use cases:**
- Move files not accessed in 1 year to cold storage.
- Delete temp files older than 30 days.
- Run a custom script to encrypt files classified as `Confidentiality = High`.

## 6. Advanced File Screens and Reporting

**File screen exceptions** allow specific subdirectories to bypass a parent screen:

```powershell
# Allow .exe files in a tools subdirectory despite a parent block
New-FsrmFileScreenException -Path "D:\Shares\IT\Tools" `
    -IncludeGroup "Executable Files"
```

**Custom file groups** for granular control:

```powershell
# Create a custom file group for database files
New-FsrmFileGroup -Name "Database Files" `
    -IncludePattern @("*.mdf","*.ldf","*.ndf","*.bak")
```

**Storage reports** вҖ” generate on-demand or scheduled reports:

| Report Type | What It Shows |
|---|---|
| Duplicate Files | Files with identical content (by hash) |
| Files by File Group | Breakdown by file type categories |
| Large Files | Files exceeding a size threshold |
| Least Recently Accessed | Files not accessed within N days |
| Quota Usage | Current usage vs. quota limits |

```powershell
# Generate an on-demand storage report
New-FsrmStorageReport -Name "Monthly File Audit" `
    -Namespace "D:\Shares" `
    -ReportType @("LargeFiles","DuplicateFiles","FilesByFileGroup") `
    -LargeFileMinimum 100MB

# View scheduled reports
Get-FsrmStorageReport
```

## 7. Capacity Planning with Dedup

**DDPEval.exe (Data Deduplication Evaluation Tool):**

This standalone tool estimates dedup savings **without enabling dedup** on the volume.

```powershell
# Run dedup evaluation on a data volume
DDPEval.exe D:\Shares

# Output includes:
#   - Total data size
#   - Estimated dedup savings (bytes and percentage)
#   - Number of files evaluated
#   - Chunk count and average chunk size
```

**Per-workload savings expectations:**

| Workload | Expected Savings | Notes |
|---|---|---|
| General file shares | 50вҖ“70 % | Higher with many similar Office documents |
| VDI (100 similar VMs) | 80вҖ“95 % | OS files deduplicate extremely well |
| Backup (daily fulls) | 50вҖ“80 % | Each full contains mostly unchanged data |
| Software distribution | 60вҖ“80 % | Multiple versions of same installers |
| Already-compressed data | 0вҖ“5 % | ZIP, JPEG, MP4 вҖ” virtually no benefit |

**Baseline measurement workflow:**

1. Run `DDPEval.exe` on the target volume before enabling dedup.
2. Enable dedup and let optimization run for at least one full cycle.
3. Compare with `Get-DedupStatus` вҖ” verify savings match the estimate.
4. Monitor over 30 days вҖ” savings improve as more data is processed.
5. Plan capacity based on `SavingsRate` and projected data growth.
