**⏱ Estimated: 10 minutes | 🎯 Goal: Diagnose dedup failures, FSRM quota issues, and file screen bypasses**

---

## 1. Dedup Job Failures

When optimization is not running or producing poor results, check these areas:

**Optimization not running:**

| Symptom | Cause | Fix |
|---|---|---|
| Job never starts | Schedule disabled or misconfigured | `Get-DedupSchedule` — verify `Enabled = True` |
| Job starts but skips volume | Volume not eligible (boot, system, CSV non-S2D) | `Get-DedupVolume` — confirm `Enabled = True` |
| Files not being optimized | MinimumFileAge not met (default = 3 days) | `Set-DedupVolume -Volume D: -MinimumFileAge 0` to reduce |
| Job completes with 0 savings | Files too small (< 32 KB) or already compressed | Check workload type — `.zip`, `.jpg`, `.mp4` don't dedup well |

**Low savings:**
- Small files (< 32 KB) are **not chunked** — they stay as-is.
- Already-compressed data (ZIP, JPEG, encrypted) yields near-zero dedup.
- Wrong usage type — VDI workload on a `Default` type misses optimization.

```powershell
# Diagnose current job status
Get-DedupJob | Format-List *

# Check volume status and savings
Get-DedupStatus -Volume D: | Select-Object Volume, OptimizedFilesCount,
    OptimizedFilesSavingsRate, LastOptimizationResult

# Check event logs for errors
Get-WinEvent -LogName "Microsoft-Windows-Deduplication/Operational" -MaxEvents 20
```

## 2. Dedup Corruption and Repair

**Symptoms of chunk store corruption:**
- Users get **access errors** when opening deduplicated files
- Event ID 6153 or 6159 in the Deduplication event log
- `Get-DedupStatus` shows `LastScrubResult` as non-zero

**Repair steps:**

```powershell
# Step 1: Run integrity scrubbing to detect corruption
Start-DedupJob -Volume D: -Type Scrubbing -Full

# Step 2: Check scrub results
Get-DedupStatus -Volume D: | Select-Object LastScrubResult, CorruptionCount

# Step 3: If corruption found, dedup will auto-repair where possible
# For severe corruption, re-optimize affected files:
Start-DedupJob -Volume D: -Type Optimization -Full
```

> ⚠️ **Critical:** If the chunk store is severely corrupted and files cannot be read,
> the only recovery option may be restoring from backup. Always ensure dedup volumes
> are included in your backup strategy.

## 3. Volume Full with Dedup

A dedup-enabled volume can still fill up. Common causes:

| Cause | Explanation | Resolution |
|---|---|---|
| GC not keeping up | Deleted/modified files leave orphan chunks | Run `Start-DedupJob -Volume D: -Type GarbageCollection -Full` |
| Chunk store metadata growth | Hash tables and stream maps consume space | Monitor with `Get-DedupVolume` — check `ChunkStoreSize` |
| Optimization temp space | Jobs need temporary scratch space during processing | Ensure 5–10 % free space for optimization to run |
| New data arriving faster than optimization | Optimization backlog grows | Increase throughput: `Start-DedupJob -Volume D: -Type Optimization -Memory 50 -Cores 50` |

```powershell
# Monitor volume dedup space usage
Get-DedupVolume -Volume D: | Select-Object Volume, SavedSpace, FreeSpace,
    UnoptimizedSize, SavingsRate

# Check if GC is overdue
Get-DedupStatus -Volume D: | Select-Object LastGarbageCollectionTime,
    LastGarbageCollectionResult
```

## 4. FSRM Quota Not Enforcing

**Common reasons quotas don't enforce:**

1. **Soft quota mistaken for hard quota** — soft quotas only alert; they never block writes.
   ```powershell
   # Check quota type
   Get-FsrmQuota -Path "D:\Shares\Data" | Select-Object Path, Template, SoftLimit
   # SoftLimit = True means it is a soft (monitoring-only) quota
   ```

2. **Quota not applied to existing data** — quotas track usage from the point of creation.
   Existing data is counted, but the quota won't retroactively block until a new write pushes
   usage over the limit.

3. **Auto-apply vs manual mismatch** — auto-apply quotas only create quotas for **new**
   subfolders. Pre-existing subfolders need manual quota assignment.
   ```powershell
   # Scan existing subfolders and apply quotas
   Update-FsrmAutoQuota -Path "D:\UserFolders"
   ```

4. **FSRM service not running:**
   ```powershell
   Get-Service srmsvc | Select-Object Status, StartType
   Start-Service srmsvc
   ```

## 5. File Screen Bypass Issues

When files are getting past a file screen:

- **Admin exemptions** — by default, administrators may be exempt from active file screens.
  Check `Get-FsrmSetting` for `AdminNotificationEmail` and exemption settings.
- **Inherited exceptions** — a file screen exception on a parent folder overrides child screens.
  ```powershell
  Get-FsrmFileScreenException | Format-List Path, IncludeGroup
  ```
- **Scope is directory-level only** — file screens apply to the specified folder and subfolders,
  not the entire volume. Verify the screen path matches where users are writing.
- **File renamed after save** — user saves as `.txt` then renames to `.exe`. File screens only
  check at write time, not on rename. Use passive screening + monitoring for this pattern.

## 6. FSRM Notification Issues

Notifications not arriving? Walk through this checklist:

```powershell
# Step 1: Verify SMTP configuration
Get-FsrmSetting | Select-Object SmtpServer, AdminEmailAddress, FromEmailAddress

# Step 2: Set or update SMTP server
Set-FsrmSetting -SmtpServer "smtp.contoso.com" -AdminEmailAddress "admin@contoso.com" `
    -FromEmailAddress "fsrm@contoso.com"

# Step 3: Send a test notification
Send-FsrmTestEmail -ToEmailAddress "admin@contoso.com"
```

**Other notification types:**
- **Event Log** — always works; check Application log for SRMSVC source events.
- **Command Actions** — verify the command path exists and the FSRM service account has
  execute permissions.
- **Notification limits** — FSRM has a minimum notification interval (default 60 min) to
  prevent alert floods. Adjust with `Set-FsrmSetting -NotificationLimit 30`.

## 7. Diagnostic Commands Reference

```powershell
# ── Dedup Diagnostics ──
Update-DedupStatus -Volume D:                       # Force status refresh
Get-DedupStatus -Volume D:                          # Comprehensive status
Get-DedupJob                                        # Running/queued jobs
Get-DedupVolume -Volume D:                          # Volume config and savings
Measure-DedupFileMetadata -Volume D: -Path "D:\Shares"  # Per-folder dedup metadata size

# ── FSRM Diagnostics ──
Get-FsrmQuota -Path "D:\Shares\*"                   # Quotas under a path
Get-FsrmFileScreen                                   # All file screens
Get-FsrmFileScreenException                          # All exceptions
Get-FsrmStorageReport                                # Scheduled reports
Get-FsrmSetting                                      # Global FSRM config
```
