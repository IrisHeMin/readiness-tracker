## 📌 Live Migration Internals: Memory Transfer Deep Dive

Understanding the internal mechanics of how memory is transferred is critical for optimizing large-scale migrations and troubleshooting edge cases.

### Iterative Memory Copy Algorithm

```
┌─────────────────────────────────────────────────────────────────────────┐
│            ITERATIVE PRE-COPY MEMORY TRANSFER                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  Iteration 0 (Initial Copy):                                            │
│  ┌───────────────────────────────────────────────────┐                   │
│  │ ████████████████████████████████████████████████████│ All pages        │
│  └───────────────────────────────────────────────────┘ (e.g., 8 GB)     │
│                                                                          │
│  Iteration 1 (Dirty pages since Iteration 0):                           │
│  ┌───────────────────────────────────────────────────┐                   │
│  │ ██░░░██░░░░██░░░░░░██░░░██░░░░░░░░░██░░░░░░░░░░░░│ ~40% dirty       │
│  └───────────────────────────────────────────────────┘ (~3.2 GB)        │
│                                                                          │
│  Iteration 2 (Dirty pages since Iteration 1):                           │
│  ┌───────────────────────────────────────────────────┐                   │
│  │ ░░░░░██░░░░░░░░░░░░░░░░░██░░░░░░░░░░░░░░░░░░░░░░│ ~15% dirty       │
│  └───────────────────────────────────────────────────┘ (~1.2 GB)        │
│                                                                          │
│  Iteration N (Converged — dirty set small enough):                      │
│  ┌───────────────────────────────────────────────────┐                   │
│  │ ░░░░░░░░░░░░█░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░│ < threshold      │
│  └───────────────────────────────────────────────────┘                   │
│                                                                          │
│  ► BLACKOUT: Pause VM → Transfer final dirty set → Resume on dest       │
│                                                                          │
│  Legend: ██ = Dirty (must transfer)  ░░ = Clean (skip)                   │
└─────────────────────────────────────────────────────────────────────────┘
```

### Dirty Page Tracking Mechanism

The hypervisor uses **Second Level Address Translation (SLAT)** — Intel EPT or AMD RVI — to track which memory pages are modified by the guest:

1. Before each iteration, the hypervisor marks all guest page table entries as **read-only** in the EPT/RVI tables
2. When the guest writes to a page, it triggers an **EPT violation** (page fault at hypervisor level)
3. The hypervisor records the page as dirty in a **bitmap** and allows the write to proceed
4. At the end of the iteration, only pages in the dirty bitmap are transferred

> 💡 **Performance Implication:** Very write-heavy VMs (databases, in-memory caches) generate high dirty rates, making convergence harder. This is where compression and RDMA become critical.

### Convergence Mechanisms

When the dirty rate exceeds the network transfer rate, Hyper-V employs two strategies:

| Mechanism | How It Works | Available Since | Trade-off |
|---|---|---|---|
| **Memory Compression** | Dirty pages are compressed before transfer; ~50–70% reduction for typical workloads | Server 2012 R2 | Slight CPU overhead on source |
| **CPU Throttling** | Source VM's virtual CPU is progressively slowed (up to 90% throttle) to reduce dirty rate | Server 2016 | VM performance degrades during migration |

```
  Convergence Timeline:
  
  Time ──────────────────────────────────────────────────────►
  
  Dirty Rate:    ████████████████████████████
                 ████████████████████████
                 █████████████████████         ◄── Compression kicks in
                 ████████████████
                 ██████████████                ◄── CPU throttle begins
                 ████████
                 ██████
                 ████                          ◄── Below threshold
                 ██ → BLACKOUT → Complete!
  
  Transfer Rate: ─────────────────────────── (constant, limited by network)
```

## 📌 Transfer Protocols: SMB vs TCP

Hyper-V supports two transport protocols for Live Migration data transfer:

### Comparison

| Feature | TCP (Default pre-2016) | SMB 3.x (Default 2016+) |
|---|---|---|
| **Protocol** | Raw TCP on port 6600 | SMB over port 445 |
| **RDMA Support** | ❌ No | ✅ Yes (SMB Direct) |
| **Multichannel** | ❌ No | ✅ Yes (multiple NICs aggregated) |
| **Encryption** | ❌ No (plaintext) | ✅ Optional (AES-128-GCM) |
| **CPU Overhead** | Moderate | Low (especially with RDMA) |
| **Best For** | Simple environments, legacy | Modern datacenters, performance-critical |

### Configuring Transport

```powershell
# Check current setting
Get-VMHost | Select-Object VirtualMachineMigrationPerformanceOption
# Values: TCPIP, Compression, SMB

# Set to SMB (enables RDMA if hardware supports it)
Set-VMHost -VirtualMachineMigrationPerformanceOption SMB

# Set to Compression over TCP
Set-VMHost -VirtualMachineMigrationPerformanceOption Compression

# Verify RDMA is active on NICs
Get-NetAdapterRdma | Select-Object Name, Enabled, Operational

# Verify SMB Direct is working
Get-SmbClientNetworkInterface | Where-Object RdmaCapable -eq $true
Get-SmbServerNetworkInterface | Where-Object RdmaCapable -eq $true

# Check SMB Multichannel connections during migration
Get-SmbMultichannelConnection -IncludeNotSelected |
    Format-Table ClientInterface, ServerInterface, Selected, ClientRdmaCapable
```

## 📌 Kerberos Constrained Delegation Setup

For production environments where migration is initiated from remote tools (SCVMM, WAC, remote PowerShell), Kerberos constrained delegation is essential.

### Step-by-Step Configuration

```powershell
# Run on Domain Controller or machine with AD PowerShell module

# Step 1: Get all Hyper-V host computer objects
$hvHosts = @("HV-Node01", "HV-Node02", "HV-Node03")

# Step 2: For each host, allow delegation to all OTHER hosts
foreach ($source in $hvHosts) {
    $delegationTargets = @()
    foreach ($dest in $hvHosts | Where-Object { $_ -ne $source }) {
        $fqdn = "$dest.contoso.com"
        $delegationTargets += @(
            "Microsoft Virtual System Migration Service/$fqdn",
            "Microsoft Virtual System Migration Service/$dest",
            "cifs/$fqdn",
            "cifs/$dest"
        )
    }

    # Apply delegation
    Set-ADComputer $source -Add @{
        'msDS-AllowedToDelegateTo' = $delegationTargets
    }

    # Enable "Trust this computer for delegation to specified services only"
    # Using "Any authentication protocol" (Protocol Transition)
    Set-ADAccountControl -Identity "$source$" -TrustedToAuthForDelegation $true

    Write-Host "Configured delegation for $source → $($delegationTargets -join ', ')"
}

# Step 3: Verify configuration
foreach ($host in $hvHosts) {
    Write-Host "`n=== $host ===" -ForegroundColor Cyan
    Get-ADComputer $host -Properties msDS-AllowedToDelegateTo,
        TrustedToAuthForDelegation |
        Select-Object Name, TrustedToAuthForDelegation,
        @{N='DelegateTo'; E={$_.'msDS-AllowedToDelegateTo' -join "`n"}}
}

# Step 4: Set Hyper-V hosts to use Kerberos
foreach ($host in $hvHosts) {
    Invoke-Command -ComputerName $host -ScriptBlock {
        Set-VMHost -VirtualMachineMigrationAuthenticationType Kerberos
    }
}
```

### Delegation Requirements Summary

| Component | Service Principal | Purpose |
|---|---|---|
| **Migration Service** | `Microsoft Virtual System Migration Service/<host>` | Authorize memory + VM state transfer |
| **CIFS/SMB** | `cifs/<host>` | Authorize VHD and storage transfer |
| **Direction** | Source → Destination (both directions for bi-directional migration) | |

> ⚠️ **Gotcha:** If you add new hosts to the cluster, you must update delegation for ALL existing hosts to include the new host, and configure the new host to delegate to all others. This is O(n²) — automation is essential.

## 📌 Cross-Version Migration Compatibility

Live Migration supports moving VMs between different Windows Server versions within certain constraints:

### Compatibility Matrix

| Source Host | Destination Host | Supported? | Notes |
|---|---|---|---|
| Server 2016 | Server 2016 | ✅ Yes | Same version |
| Server 2016 | Server 2019 | ✅ Yes | Older → Newer works |
| Server 2019 | Server 2016 | ⚠️ Depends | Only if VM config version ≤ 8.0 |
| Server 2019 | Server 2019 | ✅ Yes | Same version |
| Server 2019 | Server 2022 | ✅ Yes | Older → Newer works |
| Server 2022 | Server 2019 | ⚠️ Depends | Only if VM config version ≤ 9.0 |
| Server 2022 | Server 2022 | ✅ Yes | Same version |
| Server 2022 | Server 2025 | ✅ Yes | Older → Newer works |

### VM Configuration Version Reference

| VM Config Version | Introduced With | Features Added |
|---|---|---|
| 5.0 | Server 2012 R2 | Base version |
| 7.0 | Server 2016 | Secure Boot for Linux, Production Checkpoints |
| 8.0 | Server 2016 (1709) | Persistent memory, vTPM |
| 9.0 | Server 2019 | Battery status, device pass-through |
| 9.2 | Server 2019 (1903) | Enhanced session mode |
| 10.0 | Server 2022 | Nested virtualization improvements |
| 11.0 | Server 2025 | Latest features |


### Measuring Migration Performance

```powershell
# Comprehensive migration performance test
function Measure-LiveMigration {
    param(
        [string]$VMName,
        [string]$DestinationHost
    )

    $vm = Get-VM -Name $VMName
    $memGB = [math]::Round($vm.MemoryAssigned / 1GB, 2)

    Write-Host "Migrating $VMName ($memGB GB memory) to $DestinationHost..." -ForegroundColor Cyan
    $stopwatch = [System.Diagnostics.Stopwatch]::StartNew()

    Move-VM -Name $VMName -DestinationHost $DestinationHost

    $stopwatch.Stop()
    $seconds = $stopwatch.Elapsed.TotalSeconds
    $throughputGbps = ($memGB * 8) / $seconds  # Rough estimate

    Write-Host "`nMigration Complete!" -ForegroundColor Green
    Write-Host "  Duration:    $([math]::Round($seconds, 1)) seconds"
    Write-Host "  Memory:      $memGB GB"
    Write-Host "  Throughput:  ~$([math]::Round($throughputGbps, 2)) Gbps (estimated)"
}

# Configure RDMA/SMB Direct for maximum performance
Set-VMHost -VirtualMachineMigrationPerformanceOption SMB

# Enable Jumbo Frames on migration NICs
Set-NetAdapterAdvancedProperty -Name "Migration-NIC" `
    -RegistryKeyword "*JumboPacket" -RegistryValue 9014

# Increase simultaneous migration limit
Set-VMHost -MaximumVirtualMachineMigrations 4
# For 10 GbE: 2-4 simultaneous is typical
# For 25+ GbE with RDMA: 4-8 can work well
```

## 📌 Edge Cases and Limitations

### SR-IOV (Single Root I/O Virtualization)

VMs with SR-IOV enabled network adapters **can** be Live Migrated, but with a caveat:

```
  During Migration with SR-IOV:
  
  Source Host                          Destination Host
  ┌────────────┐                      ┌────────────┐
  │ VM         │                      │ VM         │
  │ ┌────────┐ │                      │ ┌────────┐ │
  │ │ SR-IOV │ │  ──── Migration ──►  │ │Synthetic│ │ ← Temporary fallback!
  │ │  VF    │ │                      │ │  NIC    │ │
  │ └────────┘ │                      │ └────────┘ │
  └─────┬──────┘                      └─────┬──────┘
        │                                    │
  Physical NIC                         Physical NIC
  (VF released)                        (VF assigned after migration)
  
  Post-Migration: VM automatically switches back to SR-IOV VF
  Brief network blip (~1-2 seconds) during SR-IOV re-attach
```

```powershell
# Check if SR-IOV is enabled on a VM's NIC
Get-VMNetworkAdapter -VMName "MyVM" | Select-Object Name, IovWeight,
    IovQueuePairsRequested, IovInterruptModeration

# SR-IOV falls back to synthetic during migration automatically
# No special config needed — but test that applications tolerate the brief blip
```

### Discrete Device Assignment (DDA) / vGPU

| Feature | Live Migration Support |
|---|---|
| SR-IOV NIC | ✅ Yes (falls back to synthetic temporarily) |
| DDA GPU (pass-through) | ❌ **No** — VM must be stopped |
| GPU-P (GPU Partitioning / vGPU) | ✅ Yes (Server 2025+ with supported GPUs) |
| DDA NVMe | ❌ **No** — VM must be stopped |
| vTPM | ✅ Yes (keys stored in VM config, transfers with it) |
| Persistent Memory (PMem) | ⚠️ Depends on configuration |

### VMs with Very Large Memory (1+ TB)

```powershell
# For very large VMs, pre-migration checks are critical
# Estimate migration time: Memory(GB) / NetworkSpeed(GB/s)
```

## 📌 Network Trace Troubleshooting

When migration issues are not obvious from event logs, packet captures reveal protocol-level problems.

### Capturing Migration Traffic

```powershell
# Method 1: Built-in packet capture (pktmon — Server 2019+)
pktmon filter add -t TCP -p 6600     # Live Migration TCP
pktmon filter add -t TCP -p 445      # SMB-based migration
pktmon start --capture --pkt-size 0  # Full packet capture

# ... reproduce migration issue ...

pktmon stop
pktmon etl2pcap pktmon.etl --out migration-trace.pcapng

# Method 2: netsh trace (all versions)
netsh trace start capture=yes tracefile=C:\migration-trace.etl `
    provider=Microsoft-Windows-Hyper-V-VMMS `
    provider=Microsoft-Windows-SMBClient `
    maxsize=1024

# ... reproduce issue ...
netsh trace stop

# Method 3: Performance Monitor for detailed migration metrics
logman create counter "MigrationPerf" `
    -c "\Hyper-V VM Live Migration\*" `
       "\SMB Client Shares\*" `
       "\RDMA Activity\*" `
       "\Network Interface(*)\*" `
    -si 1 -f csv -o "C:\MigrationPerf.csv"

logman start "MigrationPerf"
# ... perform migration ...
logman stop "MigrationPerf"
```

### What to Look For in Traces

| Symptom | What to Check in Capture |
|---|---|
| Migration never starts | TCP SYN to port 6600 — is it reaching destination? Firewall blocking? |
| Auth failure | Look for Kerberos TGS-REQ/REP or NTLM challenge — are SPNs correct? |
| Slow transfer | Check for TCP retransmissions, window size reductions, or MTU mismatches |
| SMB errors | SMB2 NEGOTIATE, SESSION_SETUP responses — check STATUS codes |
| RDMA not used | Verify iWARP MPA handshake or RoCE RDMA CM messages are present |

## 📌 Advanced Cluster-Aware Migration

In Failover Cluster environments, the Cluster service orchestrates migrations with additional intelligence:

```powershell
# Cluster-aware Live Migration (preferred in cluster environments)
Move-ClusterVirtualMachineRole -Name "MyVM" -Node "HV-Node02" `
    -MigrationType Live

# Drain a node (migrate ALL VMs off a host for maintenance)
Suspend-ClusterNode -Name "HV-Node01" -Drain -Wait
# This uses cluster placement logic: anti-affinity, preferred owners, load balancing

# Resume node after maintenance
Resume-ClusterNode -Name "HV-Node01" -Failback Immediate

# Check cluster migration settings
Get-ClusterResourceType "Virtual Machine" | Get-ClusterParameter |
    Where-Object { $_.Name -match "Migration" }

# View migration queue and history
Get-ClusterLog -Node "HV-Node01" -TimeSpan 60 |
    Select-String "migration|move|drain" -Context 2
```
