## рҹ“Ң What is Live Migration?

Live Migration is a Hyper-V feature that allows you to **move a running virtual machine from one physical host to another with zero downtime**. The VM stays online and responsive throughout the process вҖ” end users experience no disconnection or service interruption.

### Migration Types Comparison

| Feature | Live Migration | Quick Migration | Storage Migration |
|---|---|---|---|
| **VM State** | Running (no downtime) | Saved вҶ’ Moved вҶ’ Resumed | Running (no downtime) |
| **What Moves** | Memory + VM state (optionally storage) | Saved state file + config | Virtual hard disks only |
| **Downtime** | Milliseconds (blackout window) | Seconds to minutes | None (storage only) |
| **Requires Cluster** | No (Shared Nothing) or Yes | Yes (Failover Cluster) | No |
| **Use Case** | Planned maintenance, load balancing | Fallback when Live fails | Move VHDs between storage |
| **PowerShell** | `Move-VM` | `Move-ClusterVirtualMachineRole -MigrationType Quick` | `Move-VMStorage` |

> рҹ’Ў **Key Distinction:** Live Migration moves the *compute* (running VM). Storage Migration moves the *storage* (VHDs). You can also do both simultaneously вҖ” this is called **Shared Nothing Live Migration**.

### Phase Details

| Phase | What Happens | Duration |
|---|---|---|
| **1. Setup** | Destination creates shadow VM, validates compatibility | Seconds |
| **2. Memory Transfer** | Source sends memory pages iteratively; dirty pages re-sent each round | Seconds to minutes (depends on memory size + change rate) |
| **3. Convergence** | If dirty rate is too high, Hyper-V throttles the VM CPU to reduce dirtying | Automatic вҖ” kicks in when iterations aren't converging |
| **4. Blackout Window** | VM paused on source, final state transferred, VM resumed on destination | Typically < 1 second (often < 10 ms on modern hardware) |
| **5. Cleanup** | Source removes shadow VM, ARP broadcast updates network switches | Milliseconds |

> рҹ’Ў **Convergence:** Starting with Windows Server 2012 R2, Hyper-V can **compress** memory pages during transfer. Windows Server 2016+ added **SMB Direct (RDMA)** support for even faster transfers. If the dirty page rate exceeds transfer speed, the hypervisor throttles the source VM's CPU to let the transfer catch up.

## рҹ“Ң Prerequisites

Before Live Migration will work, these conditions **must** be met:

### Infrastructure Requirements

| Requirement | Details |
|---|---|
| **Processor Compatibility** | Same processor manufacturer (IntelвҶ”Intel, AMDвҶ”AMD). Use Processor Compatibility mode for different generations. |
| **Active Directory** | Both hosts must be in the **same AD domain** (or trusting domains) |
| **Hyper-V Role** | Installed on both source and destination |
| **Network** | At least one network reachable between both hosts |
| **VM Configuration Version** | Destination must support the VM's config version |
| **Sufficient Resources** | Destination must have enough memory, CPU, and storage |
| **Virtual Switch** | Destination must have a virtual switch with the **same name** as the source |

### Authentication Methods

| Method | How It Works | Pros | Cons |
|---|---|---|---|
| **CredSSP** | Credentials delegated directly from managing machine to source to destination | Simple to configure | Requires console/RDP session on source host (no remote 3rd machine) |
| **Kerberos** | Constrained delegation via AD service accounts | Works from any management machine (3rd party remote) | Requires AD delegation setup |

> вҡ пёҸ **Common Gotcha:** If you initiate Live Migration from a **remote management machine** (not logged into the source host), CredSSP will **fail** because of the double-hop problem. You need **Kerberos constrained delegation** for remote management scenarios.

## рҹ“Ң Key Settings and Configuration

### Enable Live Migration on a Host

```powershell
# Enable Live Migration
Enable-VMMigration

# Configure allowed networks for migration traffic
Add-VMMigrationNetwork -Subnet "10.0.100.0/24" -Priority 1

# Set simultaneous migration limit (default: 2)
Set-VMHost -MaximumVirtualMachineMigrations 4

# Set authentication protocol
Set-VMHost -VirtualMachineMigrationAuthenticationType Kerberos

# Set performance option (compression is default in 2016+)
Set-VMHost -VirtualMachineMigrationPerformanceOption Compression
# Options: TCPIP, Compression, SMB
```

### View Current Configuration

```powershell
# Check Live Migration settings on a host
Get-VMHost | Select-Object Name,
    VirtualMachineMigrationEnabled,
    MaximumVirtualMachineMigrations,
    VirtualMachineMigrationAuthenticationType,
    VirtualMachineMigrationPerformanceOption

# List configured migration networks
Get-VMMigrationNetwork

# Check if a specific VM can be migrated
Test-VMMigration -Name "MyVM" -DestinationHost "HV-Node02"
```

### Perform a Live Migration

```powershell
# Basic Live Migration (shared storage вҖ” move compute only)
Move-VM -Name "MyVM" -DestinationHost "HV-Node02"

# Shared Nothing Live Migration (move compute + storage)
Move-VM -Name "MyVM" -DestinationHost "HV-Node02" `
    -IncludeStorage `
    -DestinationStoragePath "D:\VMs\MyVM"

# Move VM storage only (VM stays on same host)
Move-VMStorage -VMName "MyVM" -DestinationStoragePath "E:\NewStorage\MyVM"

# Move specific VHDs to different locations
Move-VM -Name "MyVM" -DestinationHost "HV-Node02" `
    -VirtualMachinePath "D:\VMs" `
    -VhdDestinationPath "E:\VHDs"
```

## вң… L100 Checkpoint вҖ” You Should Now Know:

1. **Live Migration** moves a running VM between hosts with near-zero downtime; **Quick Migration** saves and restores state (brief downtime); **Storage Migration** moves only VHDs
2. Live Migration works by iteratively copying memory pages, then performing a brief blackout window to transfer final state
3. **Shared Nothing** Live Migration moves both compute and storage вҖ” no shared infrastructure needed
4. Both hosts must be in the **same AD domain**, have **compatible processors**, matching **virtual switch names**, and sufficient resources
5. **CredSSP** is simpler but fails on remote 3rd-machine management; **Kerberos constrained delegation** is needed for remote administration
6. Use `Move-VM` to migrate, `Test-VMMigration` to validate, and a **dedicated migration network** for best performance
7. The convergence mechanism (CPU throttling + memory compression) ensures migration completes even under heavy workloads
