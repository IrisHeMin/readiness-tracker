# L200: Live Migration Troubleshooting
**⏱ Estimated: 10 minutes | 🎯 Goal: Diagnose and resolve migration failures**

---

## 📌 Common Failure Scenarios Overview

When a Live Migration fails, the VM remains running on the source host — no data is lost. The key is to identify *why* the migration was rejected or failed mid-transfer.

```
┌─────────────────────────────────────────────────────────────────────┐
│              LIVE MIGRATION FAILURE DECISION TREE                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Migration Failed                                                    │
│  │                                                                   │
│  ├─► "Not compatible" error?                                         │
│  │   ├─► Processor mismatch (Intel vs AMD, or generation)            │
│  │   ├─► VM configuration version too new for destination            │
│  │   └─► Missing virtual switch name on destination                  │
│  │                                                                   │
│  ├─► "Access denied" / auth error?                                   │
│  │   ├─► CredSSP double-hop problem (remote management)              │
│  │   ├─► Kerberos delegation not configured                          │
│  │   └─► Computer account missing delegation permissions             │
│  │                                                                   │
│  ├─► Migration starts but fails mid-transfer?                        │
│  │   ├─► Network connectivity lost between hosts                     │
│  │   ├─► Insufficient memory on destination                          │
│  │   ├─► Timeout — dirty page rate exceeds network bandwidth         │
│  │   └─► Storage path not accessible on destination                  │
│  │                                                                   │
│  └─► Migration succeeds but VM has issues after?                     │
│      ├─► Network connectivity lost (VLAN / switch mismatch)          │
│      ├─► Time sync issues (integration services)                     │
│      └─► Application-level clustering failover triggered             │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## 📌 Scenario 1: Processor Compatibility Failures

### Symptoms
- Error: *"The virtual machine cannot be moved to the destination computer. The hardware on the destination computer is not compatible with the hardware requirements of this virtual machine."*

### Diagnosis

```powershell
# Check processor info on both hosts
Get-WmiObject -Class Win32_Processor | Select-Object Name, Caption, NumberOfCores

# Compare VM configuration version
Get-VM -Name "MyVM" | Select-Object Name, Version

# Check supported VM config versions on destination
Get-VMHostSupportedVersion -ComputerName "HV-Node02"

# Check if Processor Compatibility mode is enabled
Get-VMProcessor -VMName "MyVM" | Select-Object CompatibilityForMigrationEnabled
```

### Resolution

| Issue | Fix |
|---|---|
| Different processor generations (same vendor) | Enable Processor Compatibility: `Set-VMProcessor -VMName "MyVM" -CompatibilityForMigrationEnabled $true` (requires VM shutdown) |
| Intel vs AMD | **Cannot** Live Migrate between vendors — must use Quick Migration or export/import |
| VM config version too high | **Cannot** downgrade — destination host must be upgraded |
| VM config version too low | Upgrade on destination: `Update-VMVersion -VMName "MyVM"` (one-way, irreversible!) |

> ⚠️ **Warning:** `Update-VMVersion` is **irreversible**. Once upgraded, the VM cannot be moved back to an older host. Always verify the cluster is fully upgraded first.

## 📌 Scenario 2: Authentication & Delegation Failures

### Symptoms
- Error: *"The Virtual Machine Management Service failed to authenticate the connection for a Virtual Machine migration at the source host."*
- Error: *"Access is denied"* or *"CredSSP authentication failed"*
- Migration works from console/RDP on source host but **fails from remote management machine**

### The Double-Hop Problem Explained

```
  SCENARIO: CredSSP — Works from Source Host
  ┌──────────┐    Hop 1 (OK)     ┌──────────┐
  │ Admin on │ ──────────────►   │  Dest    │
  │ Source   │   Credentials     │  Host    │
  │ Host     │   delegated       │          │
  └──────────┘                   └──────────┘

  SCENARIO: CredSSP — FAILS from Remote Machine (Double Hop)
  ┌──────────┐    Hop 1          ┌──────────┐    Hop 2 (FAIL!)
  │ Admin    │ ──────────────►  │  Source  │ ──────────X
  │ Worksta- │   Credentials    │  Host    │   Cannot forward
  │ tion     │   sent here      │          │   credentials again
  └──────────┘                  └──────────┘

  SCENARIO: Kerberos Constrained Delegation — Works from Anywhere
  ┌──────────┐    Hop 1          ┌──────────┐    Hop 2 (OK!)
  │ Admin    │ ──────────────►  │  Source  │ ──────────────►  ┌──────────┐
  │ Worksta- │   Kerberos       │  Host    │   Delegation     │  Dest    │
  │ tion     │   ticket         │          │   via AD (S4U)   │  Host    │
  └──────────┘                  └──────────┘                  └──────────┘
```

### Diagnosis

```powershell
# Check current authentication setting
Get-VMHost | Select-Object VirtualMachineMigrationAuthenticationType

# Test CredSSP from source host (should succeed)
Invoke-Command -ComputerName "HV-Node01" -ScriptBlock {
    Test-VMMigration -Name "MyVM" -DestinationHost "HV-Node02"
}

# Check Kerberos delegation in AD (run on DC or with RSAT)
Get-ADComputer "HV-Node01" -Properties msDS-AllowedToDelegateTo |
    Select-Object -ExpandProperty msDS-AllowedToDelegateTo

# Verify SPNs are registered
setspn -L HV-Node01$
# Look for: Microsoft Virtual System Migration Service/HV-Node01.domain.com
#           cifs/HV-Node01.domain.com
```

### Resolution

```powershell
# Option A: Switch to Kerberos (recommended for remote management)
Set-VMHost -VirtualMachineMigrationAuthenticationType Kerberos

# Then configure constrained delegation in AD (on Domain Controller):
# Allow HV-Node01 to delegate to HV-Node02 for migration + CIFS
Set-ADComputer "HV-Node01" -Add @{
    'msDS-AllowedToDelegateTo' = @(
        'Microsoft Virtual System Migration Service/HV-Node02.contoso.com',
        'Microsoft Virtual System Migration Service/HV-Node02',
        'cifs/HV-Node02.contoso.com',
        'cifs/HV-Node02'
    )
}
# Repeat in reverse direction for HV-Node02 → HV-Node01

# Option B: If using CredSSP, ensure you're on the source host
# Enable CredSSP client on management machine
Enable-WSManCredSSP -Role Client -DelegateComputer "HV-Node01.contoso.com"
# Enable CredSSP server on source host
Enable-WSManCredSSP -Role Server
```

## 📌 Scenario 3: Network Connectivity Issues

### Symptoms
- Migration hangs at a percentage and eventually times out
- Error: *"The virtual machine migration operation for 'VM' failed"* with network-related details
- Extremely slow transfer speeds

### Diagnosis

```powershell
# Verify migration networks are configured
Get-VMMigrationNetwork

# Test basic connectivity between hosts on migration network
Test-NetConnection -ComputerName "HV-Node02" -Port 6600
# Port 6600 = default Live Migration listener

# Check if firewall rules allow migration traffic
Get-NetFirewallRule -DisplayGroup "Hyper-V Replica*" |
    Select-Object DisplayName, Enabled, Direction, Action

# Also check these firewall groups:
Get-NetFirewallRule -DisplayGroup "Hyper-V" |
    Where-Object { $_.Enabled -eq 'True' } |
    Select-Object DisplayName, Direction

# Check network adapter speed and status
Get-NetAdapter | Where-Object { $_.Status -eq "Up" } |
    Select-Object Name, LinkSpeed, InterfaceDescription

# Monitor real-time transfer speed during migration
Get-Counter '\Hyper-V Virtual Machine Bus\Bytes Sent/sec' -Continuous
```

### Key Ports and Protocols

| Service | Port | Protocol | Notes |
|---|---|---|---|
| Live Migration (TCP) | **6600** | TCP | Default; configurable |
| Live Migration (SMB) | **445** | TCP | When using SMB transfer |
| Cluster communication | **3343** | UDP | Failover Cluster heartbeat |
| WinRM / PowerShell Remoting | **5985/5986** | TCP | HTTP / HTTPS |

## 📌 Scenario 4: Resource Contention

### Symptoms
- Error: *"Not enough memory on the destination host to start the virtual machine"*
- Migration fails immediately after compatibility check
- Slow migration due to bandwidth saturation

### Diagnosis

```powershell
# Check available memory on destination
Get-VMHost -ComputerName "HV-Node02" |
    Select-Object MemoryCapacity, @{
        Name='MemoryAvailableMB'
        Expression={ [math]::Round(
            (Get-Counter -ComputerName "HV-Node02" `
             '\Memory\Available MBytes').CounterSamples.CookedValue, 0)
        }
    }

# Check how much memory the VM needs
Get-VM -Name "MyVM" | Select-Object Name,
    @{N='AssignedMemoryMB'; E={$_.MemoryAssigned / 1MB}},
    @{N='DemandMB'; E={$_.MemoryDemand / 1MB}},
    MemoryStartup, DynamicMemoryEnabled

# Check current simultaneous migrations (is the queue full?)
Get-VMHost | Select-Object MaximumVirtualMachineMigrations

# Check how many migrations are in progress
Get-VM | Where-Object { $_.Status -match "Migration" } |
    Select-Object Name, Status, ComputerName
```

### Resolution

| Issue | Fix |
|---|---|
| Insufficient memory | Stop/migrate other VMs first, or add memory to destination |
| Too many simultaneous migrations | Increase limit: `Set-VMHost -MaximumVirtualMachineMigrations 4` |
| Network bandwidth saturated | Use dedicated migration NIC, enable compression, or use SMB Direct |
| Dynamic Memory VM showing high demand | Wait for workload to decrease, or migrate during maintenance window |

## 📌 Reading Hyper-V Event Logs

The **most important** event log for migration troubleshooting:

```powershell
# Primary migration event log
Get-WinEvent -LogName "Microsoft-Windows-Hyper-V-VMMS-Admin" -MaxEvents 50 |
    Where-Object { $_.Message -match "migrat" } |
    Format-Table TimeCreated, Id, LevelDisplayName, Message -Wrap

# Additional useful logs
$logs = @(
    "Microsoft-Windows-Hyper-V-VMMS-Admin",
    "Microsoft-Windows-Hyper-V-VMMS-Networking",
    "Microsoft-Windows-Hyper-V-VMMS-Storage",
    "Microsoft-Windows-Hyper-V-Worker-Admin",
    "Microsoft-Windows-Hyper-V-High-Availability-Admin"
)

foreach ($log in $logs) {
    Write-Host "`n=== $log ===" -ForegroundColor Cyan
    Get-WinEvent -LogName $log -MaxEvents 10 -ErrorAction SilentlyContinue |
        Where-Object { $_.LevelDisplayName -in @("Error","Warning") } |
        Format-Table TimeCreated, Id, LevelDisplayName, Message -Wrap
}
```

### Key Event IDs

| Event ID | Log | Meaning |
|---|---|---|
| **20400** | VMMS-Admin | Live Migration started |
| **20401** | VMMS-Admin | Live Migration completed successfully |
| **20402** | VMMS-Admin | Live Migration failed |
| **20500** | VMMS-Admin | Migration compatibility check failed |
| **20504** | VMMS-Admin | Authentication failure during migration |
| **20506** | VMMS-Admin | Destination rejected — insufficient resources |
| **20508** | VMMS-Admin | Migration network error |

## 📌 Performance Counters for Migration

```powershell
# Monitor migration progress in real time
$counters = @(
    '\Hyper-V VM Live Migration\Transfer Pass'
    '\Hyper-V VM Live Migration\Transferred Bytes'
    '\Hyper-V VM Live Migration\Compressed Bytes'
    '\Hyper-V VM Live Migration\Pages/sec Dirty'
    '\Hyper-V VM Live Migration\Pages/sec Transferred'
)

# Snapshot during an active migration
Get-Counter -Counter $counters -SampleInterval 2 -MaxSamples 30 |
    ForEach-Object {
        $_.CounterSamples | Format-Table Path, CookedValue
    }
```

| Counter | What It Tells You |
|---|---|
| **Transfer Pass** | Current iteration round — if this keeps climbing, the dirty rate is too high |
| **Pages/sec Dirty** | How fast the VM is dirtying memory — high = harder to converge |
| **Pages/sec Transferred** | Network transfer speed in pages — compare to dirty rate |
| **Compressed Bytes** | How much compression is saving — useful for tuning |

> 💡 **Rule of Thumb:** If **Pages/sec Dirty** consistently exceeds **Pages/sec Transferred**, the migration will struggle to converge and will eventually trigger CPU throttling or time out.

## 📌 Anti-Affinity and Placement Issues

In a Failover Cluster, anti-affinity rules prevent certain VMs from running on the same host. This can block migrations.

```powershell
# Check anti-affinity groups
Get-ClusterGroup | Get-ClusterParameter -Name AntiAffinityClassNames |
    Where-Object { $_.Value -ne "" } |
    Select-Object ClusterObject, Value

# If migration is blocked by anti-affinity:
# Temporarily clear it, migrate, then re-apply
(Get-ClusterGroup "MyVM").AntiAffinityClassNames = @()
# ... perform migration ...
(Get-ClusterGroup "MyVM").AntiAffinityClassNames = "WebServers"
```

## 📌 Pre-Migration Validation Checklist

Use `Test-VMMigration` before every planned migration:

```powershell
# Test if migration will succeed (checks all prerequisites)
$result = Test-VMMigration -Name "MyVM" -DestinationHost "HV-Node02"

# Comprehensive pre-flight check script
$vm = "MyVM"
$dest = "HV-Node02"

Write-Host "=== Pre-Migration Check for $vm → $dest ===" -ForegroundColor Green

# 1. Compatibility
$test = Test-VMMigration -Name $vm -DestinationHost $dest
Write-Host "Migration Test: $(if($test){'PASS'}else{'FAIL'})" -ForegroundColor $(if($test){'Green'}else{'Red'})

# 2. Destination memory
$freeMemMB = (Get-Counter -ComputerName $dest '\Memory\Available MBytes').CounterSamples.CookedValue
$vmMemMB = (Get-VM $vm).MemoryAssigned / 1MB
Write-Host "Dest Free Memory: ${freeMemMB}MB | VM Needs: ${vmMemMB}MB" -ForegroundColor $(if($freeMemMB -gt $vmMemMB){'Green'}else{'Red'})

# 3. Network connectivity
$net = Test-NetConnection -ComputerName $dest -Port 6600
Write-Host "Migration Port 6600: $(if($net.TcpTestSucceeded){'OPEN'}else{'BLOCKED'})" -ForegroundColor $(if($net.TcpTestSucceeded){'Green'}else{'Red'})

# 4. Virtual switch names
$srcSwitches = (Get-VMNetworkAdapter -VMName $vm).SwitchName | Sort-Object -Unique
$destSwitches = (Get-VMSwitch -ComputerName $dest).Name
foreach ($sw in $srcSwitches) {
    $exists = $sw -in $destSwitches
    Write-Host "Switch '$sw': $(if($exists){'EXISTS on dest'}else{'MISSING on dest!'})" -ForegroundColor $(if($exists){'Green'}else{'Red'})
}
```
