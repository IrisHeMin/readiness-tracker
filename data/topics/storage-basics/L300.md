## 1. Volume Corruption Scenarios

### chkdsk Phases

| Phase | Name | What It Does |
|---|---|---|
| 1 | File verification | Examines every File Record Segment (FRS) in the MFT |
| 2 | Index verification | Checks directory indexes — validates parent/child relationships |
| 3 | Security descriptors | Verifies security descriptor stream consistency |
| 4 | File data verification (`/R`) | Reads every cluster; remaps bad clusters to spare sectors |
| 5 | Free space verification (`/R`) | Scans unused clusters for bad sectors |

### chkdsk Flags — When to Use Each

| Flag | Behavior | When to Use |
|---|---|---|
| (none) | Read-only scan — reports but does not fix | Initial assessment; safe on production |
| `/F` | Fixes metadata errors (phases 1-3) | Known filesystem inconsistency; requires exclusive lock |
| `/R` | Implies /F + scans for bad clusters (phases 4-5) | Suspected media errors (Event ID 7, 153) |
| `/B` | Re-evaluates bad clusters on NTFS (clears bad cluster list) | After disk replacement or firmware fix |
| `/scan` | Online scan (Server 2012+), no dismount needed | Proactive check on production volumes |
| `/spotfix` | Applies fixes gathered from `/scan` during brief offline | Minimal-downtime repair |

> ⚠️ **Production risk:** Running `chkdsk /R` on a multi-TB volume can take **hours** and requires exclusive volume access. Prefer `/scan` + `/spotfix` on Server 2012+ to minimize downtime.

### The Dirty Bit

When NTFS detects an inconsistency or unclean shutdown, it sets the **dirty bit** on the volume.

```powershell
# Check dirty bit
fsutil dirty query C:
chkntfs C:

# Manually set dirty bit (force chkdsk on next boot)
fsutil dirty set C:
```

If the system volume is dirty, chkdsk runs automatically at boot (autochk.exe). To cancel a scheduled check:

```cmd
chkntfs /x C:
```

## 2. Disk Error Diagnosis

### Critical Event IDs

| Event ID | Source | Severity | Description |
|---|---|---|---|
| 7 | disk | Error | Bad block detected — device `\Device\Harddisk0\DR0` has a bad block |
| 11 | disk | Error | Controller error — driver detected a controller error on `\Device\Harddisk0\DR0` |
| 15 | disk | Error | Device not ready — the device is not ready for access |
| 51 | disk | Warning | Paging error during I/O operation |
| 52 | Ntfs | Warning | Delayed write failed — Windows was unable to save data to file |
| 55 | Ntfs | Error | NTFS file system structure corruption detected (self-healing triggered) |
| 98 | Ntfs | Warning | Volume requires offline chkdsk |
| 129 | storahci / stornvme | Warning | Reset to device `\Device\RaidPort0` was issued (I/O timeout) |
| 153 | disk | Warning | I/O operation retried — logical block address, device, status |
| 157 | disk | Information | Disk was surprise-removed |

### Event Correlation Approach

```
 Event 153 (IO retry) ──→ Multiple 153s in sequence?
      │                          │
      │                     YES ──→ Event 129 (device reset) likely follows
      │                              │
      │                         Is disk reporting Event 7 (bad block)?
      │                              │
      │                         YES ──→ Media failure — check SMART
      │                         NO  ──→ Controller/path issue — check HBA, cable, MPIO
      │
      └── Isolated single 153 ──→ Transient; monitor but not critical
```

```powershell
# Pull recent disk errors from System log
Get-WinEvent -FilterHashtable @{
    LogName = 'System'
    ProviderName = 'disk','Ntfs','storahci','stornvme','volmgr'
    Level = 1,2,3    # Critical, Error, Warning
    StartTime = (Get-Date).AddDays(-7)
} | Format-Table TimeCreated, Id, LevelDisplayName, Message -Wrap
```

## 3. S.M.A.R.T. Monitoring

**S.M.A.R.T.** (Self-Monitoring, Analysis and Reporting Technology) is built into most disk firmware and exposes predictive failure indicators.

```powershell
# Retrieve reliability counters (Windows Storage Management API)
Get-PhysicalDisk | Get-StorageReliabilityCounter |
    Select-Object DeviceId, Temperature, Wear,
                  ReadErrorsTotal, ReadErrorsCorrected, ReadErrorsUncorrected,
                  WriteErrorsTotal, WriteErrorsCorrected, WriteErrorsUncorrected
```

### Key S.M.A.R.T. Attributes

| Attribute | ID | Critical Threshold | What It Means |
|---|---|---|---|
| Reallocated Sectors Count | 0x05 | Any non-zero value warrants monitoring | Sectors remapped to spare area due to defects |
| Current Pending Sector Count | 0xC5 | > 0 = potential failure | Sectors waiting to be remapped on next write |
| Uncorrectable Errors | 0xC6 | > 0 = disk degrading | Sectors that could not be recovered |
| Wear Leveling Count (SSD) | 0xAD | Varies by vendor | % of SSD write endurance consumed |
| Power-On Hours | 0x09 | Informational | Total operational hours |
| Temperature | 0xC2 | > 60°C for HDD | Operating temperature |

> 🔑 **Action rule:** If `ReadErrorsUncorrected > 0` or `Reallocated Sectors` is climbing, escalate for proactive disk replacement.

## 4. Storage Stack Internals

### Driver Stack Hierarchy

```
 ┌─────────────────────────────────────────────────────┐
 │                   Filesystem (ntfs.sys)              │
 ├─────────────────────────────────────────────────────┤
 │              Volume Manager (volmgr.sys)             │
 ├─────────────────────────────────────────────────────┤
 │           Partition Manager (partmgr.sys)            │
 ├─────────────────────────────────────────────────────┤
 │ ┌────────────────┐  ┌──────────────────────────┐    │
 │ │  MPIO (mpio.sys)│  │  (bypassed if single path)│   │
 │ └───────┬────────┘  └──────────────────────────┘    │
 ├─────────┴───────────────────────────────────────────┤
 │            Disk Class Driver (disk.sys)              │
 │       Constructs SRBs (SCSI Request Blocks)          │
 ├─────────────────────────────────────────────────────┤
 │            Port Driver (storport.sys)                │
 │       Queue management, DMA, interrupt dispatch      │
 ├─────────────────────────────────────────────────────┤
 │            Miniport Driver (vendor HBA)              │
 │       e.g., lsi_sas3.sys, ql2300.sys, nvme.sys      │
 ├─────────────────────────────────────────────────────┤
 │            Physical Adapter (HBA / NVMe)             │
 └─────────────────────────────────────────────────────┘
```

### I/O Flow in Detail

1. Application calls `WriteFile()` → I/O Manager creates an IRP (I/O Request Packet)
2. Filesystem driver translates file offset → Logical Block Address (LBA)
3. Volume Manager maps volume LBA → physical disk LBA
4. Partition Manager adds partition offset
5. Disk class driver packages LBA + data into a SCSI CDB (Command Descriptor Block) within an SRB
6. Storport queues the SRB, performs DMA mapping, and delivers to miniport
7. Miniport programs the HBA hardware to execute the transfer

## 5. Performance Tuning

### Partition Alignment

Misaligned partitions cause a single logical I/O to span two physical sectors, doubling I/O operations.

```powershell
# Check partition offset (should be divisible by 4096 for 4K sectors)
Get-Partition | Select-Object DiskNumber, PartitionNumber, Offset,
    @{Name='AlignedTo4K'; Expression={($_.Offset % 4096) -eq 0}}

# WMI method (legacy)
wmic partition get Name, StartingOffset
```

### Sector Size Impact

| Sector Format | Physical | Logical | Notes |
|---|---|---|---|
| 512 native | 512 bytes | 512 bytes | Legacy — disappearing from market |
| 512e (emulation) | 4096 bytes | 512 bytes | Most common today; alignment critical |
| 4Kn (native 4K) | 4096 bytes | 4096 bytes | Best performance; requires OS + app support |

```powershell
# Identify sector sizes
Get-PhysicalDisk | Select FriendlyName, MediaType,
    @{N='PhysicalSector';E={$_.PhysicalSectorSize}},
    @{N='LogicalSector';E={$_.LogicalSectorSize}}
```

### Queue Depth Tuning

- **Too low:** Underutilizes disk/SAN IOPS capability
- **Too high:** Causes excessive queuing latency
- Default Storport queue depth per LUN: **254**
- For SAN LUNs shared across many hosts, vendor guidance may recommend lowering to 32–64

```powershell
# Check current queue depth (registry-based)
Get-ItemProperty "HKLM:\SYSTEM\CurrentControlSet\Services\disk" -Name MaximumQueueDepth -ErrorAction SilentlyContinue
```

## 6. Storport ETW Tracing

When PerfMon counters are insufficient, ETW tracing captures per-I/O detail.

### Capture Storport Trace

```powershell
# Start Storport trace (Microsoft-Windows-StorPort provider)
logman create trace StorportTrace -ow -o C:\Logs\storport.etl `
    -p "Microsoft-Windows-StorPort" 0xFFFFFFFF 0xFF -bs 1024 -ft 1 -max 512

logman start StorportTrace

# Reproduce the issue...

logman stop StorportTrace
logman delete StorportTrace
```

### Key Storport ETW Providers

| Provider | GUID | Use Case |
|---|---|---|
| Microsoft-Windows-StorPort | `c4636a1e-7986-4646-bf10-7bc3b4a76e8e` | Port driver I/O tracing |
| Microsoft-Windows-StorDiag | `F5D05B38-80A6-4653-825D-C414E4AB3C68` | Diagnostic events |
| Microsoft-Windows-Ntfs | `3B9C9951-3480-4220-9377-9C8E5184F5CD` | Filesystem operations |

### Analysis Approach

1. Open the `.etl` file in **Windows Performance Analyzer (WPA)** or **StorPort Perf Analysis Tool**
2. Look for I/O operations with high latency (> 50 ms)
3. Correlate with specific disk numbers and LBAs
4. Check if latency is in **miniport time** (hardware/SAN) or **storport queue time** (host congestion)
