## 1. iSCSI Fundamentals

iSCSI encapsulates SCSI commands over TCP/IP, allowing block-level access to remote storage.

```
 ┌──────────────┐         TCP/IP Network         ┌──────────────┐
 │   Initiator  │◄──────── iSCSI Session ────────►│    Target    │
 │  (Windows)   │   1+ connections per session    │  (SAN/NAS)   │
 │              │                                 │              │
 │  IQN:        │                                 │  IQN:        │
 │  iqn.1991-05 │                                 │  iqn.2024-01 │
 │  .com.ms:srv │                                 │  .com.vendor │
 └──────────────┘                                 └──────────────┘
```

**Key terms:**
- **IQN** — iSCSI Qualified Name, globally unique identifier (e.g., `iqn.1991-05.com.microsoft:server01`)
- **Session** — logical connection between initiator and target (can have multiple TCP connections)
- **Connection** — single TCP socket within a session
- **Portal** — IP:port pair on target that accepts connections (default port 3260)

### iSCSI Discovery & Connection Commands

```powershell
# PowerShell approach
New-IscsiTargetPortal -TargetPortalAddress 10.0.0.50
Get-IscsiTarget                                         # List discovered targets
Connect-IscsiTarget -NodeAddress "iqn.2024-01.com.vendor:lun01" -IsPersistent $true
Get-IscsiSession | Format-Table InitiatorNodeAddress, TargetNodeAddress, IsConnected

# Legacy iscsicli commands (still useful in WinPE / debug)
iscsicli ListTargets
iscsicli SessionList
iscsicli ReportTargetMappings
iscsicli PersistentLoginTarget <TargetName> T * * * * * * * * * * * * * * * 0
```

## 2. iSCSI Troubleshooting

### Common Failure Scenarios

| Symptom | Likely Cause | Diagnostic Step |
|---|---|---|
| Target not discovered | Firewall blocking 3260, DNS failure | `Test-NetConnection -Port 3260 -ComputerName <target>` |
| Session drops intermittently | Network instability, NIC teaming issues | Check `iSCSI-Event` in Event Viewer, `netstat -an \| findstr 3260` |
| CHAP authentication fails | Mismatched secret, wrong initiator name | Verify CHAP creds in iSCSI Initiator → Configuration tab |
| Persistent login not surviving reboot | Service not set to Automatic | `Get-Service MSiSCSI \| Set-Service -StartupType Automatic` |
| Slow I/O over iSCSI | Jumbo frames mismatch, TCP offload issues | Verify MTU end-to-end: `ping -f -l 8972 <target>` |

### Key Event IDs

| Event ID | Source | Meaning |
|---|---|---|
| 1 | iScsiPrt | Target login successful |
| 9 | iScsiPrt | Target login failed |
| 20 | iScsiPrt | Connection to target lost |
| 34 | iScsiPrt | Session recovery failed |
| 39 | iScsiPrt | Session timeout — initiator attempted reconnection |
| 70 | iScsiPrt | Target removed from persistent list |

## 3. MPIO (Multipath I/O)

MPIO provides path redundancy and load balancing for storage accessed over multiple physical paths (iSCSI, Fibre Channel, SAS).

**DSM (Device Specific Module)** — the component that decides which path to use. Windows includes a generic Microsoft DSM; vendors may supply their own.

### Path Policy Comparison

| Policy | Behavior | Best For |
|---|---|---|
| Round Robin | Distributes I/Os across all active paths equally | Balanced workloads, most common choice |
| Failover Only | Uses one active path; switches on failure | Legacy apps, simple HA |
| Weighted Paths | Assigns weights to each path; prefers lower weight | Asymmetric SANs with preferred controllers |
| Least Queue Depth | Sends next I/O to path with fewest outstanding I/Os | Mixed workloads with variable latency |
| Least Blocks | Sends next I/O to path with fewest outstanding bytes | Large sequential I/O workloads |

### MPIO Commands

```powershell
# Check installed MPIO features
Get-WindowsFeature Multipath-IO

# View supported hardware list
Get-MSDSMSupportedHW | Format-Table VendorId, ProductId

# Add hardware to supported list
New-MSDSMSupportedHW -VendorId "VENDOR" -ProductId "MODEL"

# View current available paths
Get-MPIOAvailableHW

# Set global default load balance policy
Set-MSDSMGlobalDefaultLoadBalancePolicy -Policy RR   # RR, FOO, WP, LQD, LB

# Legacy command — claim disks for MPIO
mpclaim -r -i -a ""
mpclaim -s -d 0        # Show paths for disk 0
```

## 4. Storport Driver Model

Storport (`storport.sys`) replaced SCSIport as the preferred port driver starting with Windows Server 2003.

| Feature | SCSIport | Storport |
|---|---|---|
| I/O model | Half-duplex (serialized) | Full-duplex (concurrent) |
| Queue support | Single global queue | Per-LU and per-adapter queues |
| DMA mapping | Limited | Scatter/gather, 64-bit DMA |
| MSI-X support | ❌ No | ✅ Yes |
| Hot-plug | Limited | ✅ Full support |
| Interrupt model | Line-based | MSI / MSI-X |

> 💡 **Key concept:** Storport manages I/O queues and delivers SRBs (SCSI Request Blocks) to the miniport. If the miniport signals the queue is full, Storport pauses I/O and retries — this surfaces as Event ID 129 (reset to device).

## 5. Disk Performance Analysis

### Essential PerfMon Counters

| Counter | Object | What It Measures | Threshold |
|---|---|---|---|
| Avg. Disk sec/Read | PhysicalDisk | Average read latency | < 10 ms good; > 25 ms degraded |
| Avg. Disk sec/Write | PhysicalDisk | Average write latency | < 10 ms good; > 25 ms degraded |
| Disk Reads/sec | PhysicalDisk | Read IOPS | Depends on disk capability |
| Disk Writes/sec | PhysicalDisk | Write IOPS | Depends on disk capability |
| Current Disk Queue Length | PhysicalDisk | Outstanding I/Os right now | > 2× spindle count = congestion |
| Avg. Disk Queue Length | PhysicalDisk | Average outstanding I/Os | > 2× spindle count = sustained congestion |

### Bottleneck Identification Flow

```
 Is Avg. Disk sec/Read or Write > 25ms?
      │
      ├── YES → Check Current Disk Queue Length
      │           │
      │           ├── High (> 2× spindles) → Storage subsystem saturated
      │           │     → Check MPIO policy, SAN latency, HBA queue depth
      │           │
      │           └── Low → Possible single-LUN hotspot
      │                 → Spread workload across volumes
      │
      └── NO → Latency OK
                → Check IOPS vs disk capability
                → If IOPS maxed → Add spindles, tiered storage, or cache
```

### Quick Data Collection

```powershell
# 60-second snapshot of key counters
Get-Counter -Counter @(
    "\PhysicalDisk(*)\Avg. Disk sec/Read",
    "\PhysicalDisk(*)\Avg. Disk sec/Write",
    "\PhysicalDisk(*)\Current Disk Queue Length",
    "\PhysicalDisk(*)\Disk Reads/sec",
    "\PhysicalDisk(*)\Disk Writes/sec"
) -SampleInterval 5 -MaxSamples 12 | Export-Counter -Path "C:\Logs\DiskPerf.blg"
```
