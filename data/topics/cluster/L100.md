## 📌 What is Failover Clustering?

Failover Clustering is a Windows Server feature that groups multiple servers (nodes) together so that if one node fails, another **automatically takes over** the workloads — providing **high availability (HA)**.

**Key Architecture:**
```
                    ┌──────────────────────────┐
                    │   Cluster Virtual Name   │
                    │   (CNO / Cluster Name)   │
                    └────────────┬─────────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              ▼                  ▼                   ▼
      ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
      │    Node 1     │  │    Node 2     │  │    Node 3     │
      │  (Active)     │  │  (Passive)    │  │  (Passive)    │
      │              │  │              │  │              │
      │ clussvc.exe  │  │ clussvc.exe  │  │ clussvc.exe  │
      │ RHS.exe      │  │ RHS.exe      │  │ RHS.exe      │
      └──────┬───────┘  └──────┬───────┘  └──────┬───────┘
             │                 │                  │
             └────────┬────────┴──────────────────┘
                      ▼
              ┌──────────────┐
              │   Shared     │
              │   Storage    │   (SAN/iSCSI/S2D/CSV)
              └──────────────┘
```

**Core Components:**
| Component | Role |
|-----------|------|
| **Cluster Service (clussvc.exe)** | Brain of the cluster — manages membership, heartbeat, failover |
| **RHS.exe (Resource Hosting Subsystem)** | Hosts and monitors clustered resources |
| **Quorum** | Voting mechanism to prevent split-brain |
| **Cluster Network** | Communication between nodes (heartbeat + client traffic) |
| **Cluster Shared Volume (CSV)** | Shared filesystem accessible from all nodes simultaneously |

## 📌 Quorum — The Voting System

Quorum answers: **"Do we have enough votes to keep running?"**

The cluster needs **majority of votes** (>50%) to stay online. If a node loses connectivity, quorum decides if it's the cluster that's down or just that node.

**Quorum Witness Types:**
| Witness | What It Is | When to Use |
|---------|-----------|-------------|
| **Disk Witness** | A small shared disk (<1GB) | On-premises with shared storage |
| **File Share Witness (FSW)** | An SMB file share | On-premises, no shared disk |
| **Cloud Witness** | Azure Blob Storage | Hybrid or multi-site clusters |
| **No Witness** | Nodes vote only | Only if odd number of nodes (not recommended) |

**⚠️ Key Rule**: For an N-node cluster, you can survive losing up to **(N+1)/2 - 1** nodes (with a witness). Example: 3 nodes + 1 witness = 4 votes → survives losing 1 node.

## 📌 Cluster Networks

A cluster typically has **at least 2 networks**:

```
┌─────────────────────────────────────────────┐
│  Cluster Network 1 (Cluster + Client)       │
│  "Production Network"                        │
│  → Client access + cluster heartbeat         │
│  → Typically: 10.0.0.0/24                   │
├─────────────────────────────────────────────┤
│  Cluster Network 2 (Cluster Only)           │
│  "Heartbeat / Internal"                      │
│  → Node-to-node communication only           │
│  → Typically: 192.168.0.0/24 (private)      │
└─────────────────────────────────────────────┘
```

**Network Roles:**
- **Cluster and Client**: Used for both cluster traffic and client access
- **Cluster Only**: Used only for internal cluster communication (heartbeat)
- **None**: Excluded from cluster use

## 📌 Clustered Roles (Resources & Groups)

A **Clustered Role** (also called Resource Group) is a collection of resources that fail over together.

Common clustered roles:
- **File Server** — SMB shares that fail over between nodes
- **Hyper-V VM** — Virtual machines that can live-migrate or failover
- **SQL Server FCI** — SQL Failover Cluster Instance
- **Generic Service/Script** — Any Windows service or script made HA
- **DHCP / DFS Namespace** — Network services

**Resource Dependencies:**
```
Clustered Role: "SQL Server"
├── Network Name (Virtual Computer Object)
│   └── IP Address
├── SQL Server Service
│   └── Disk Resource (Data)
│   └── Disk Resource (Log)
└── SQL Server Agent
    └── SQL Server Service
```
Resources fail over **as a group**. If a critical resource fails → the entire group moves to another node.

## 📌 Essential Commands (Your Daily Tools)

### Cluster Status
```powershell
# Overall cluster info
Get-Cluster | Format-List Name, Domain, SharedVolumesRoot

# Node status
Get-ClusterNode | Format-Table Name, State, DynamicWeight

# All clustered roles and their current owner
Get-ClusterGroup | Format-Table Name, OwnerNode, State

# Cluster networks
Get-ClusterNetwork | Format-Table Name, State, Role

# Quorum configuration
Get-ClusterQuorum | Format-List *
```

### Quick Health Check
```powershell
# Get cluster events (last 24 hours)
Get-ClusterLog -TimeSpan 1440 -Destination C:\temp\

# Cluster validation (non-disruptive subset)
Test-Cluster -Node Node1,Node2 -Include "Network","Inventory"
```

## 📌 Where to Find Logs

| Log Source | Location / Command |
|-----------|-------------------|
| **Cluster Log** | `Get-ClusterLog -Destination C:\temp\` |
| **System Event Log** | `Event Viewer > System` — filter Source: FailoverClustering |
| **Cluster Event Channel** | `Event Viewer > Applications and Services > Microsoft > Windows > FailoverClustering > Operational` |
| **Cluster Validation Report** | Generated by `Test-Cluster` → HTML report |
| **Cluster Debug Tracing** | `Get-ClusterLog -UseLocalTime` — includes debug-level detail |

**Key Event IDs:**
| Event ID | Meaning |
|----------|---------|
| **1135** | Node removed from cluster membership |
| **1177** | Quorum lost |
| **1069** | Resource failed |
| **1146** | Node joined cluster |
| **1034** | Cluster service started |
