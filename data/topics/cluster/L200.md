# L200: Windows Failover Clustering Troubleshooting
**⏱ Estimated: 10 minutes | 🎯 Goal: Diagnose and resolve common cluster issues**

---

## 📌 Troubleshooting Framework

When a customer reports a cluster issue, follow this flow:

```
Step 1: Is the cluster service running on all nodes?
         ↓
Step 2: What is the node membership state?
         ↓
Step 3: Is quorum healthy?
         ↓
Step 4: Is it a resource failure or a node failure?
         ↓
Step 5: Collect cluster log & analyze root cause
```

---

## 📌 Step 1: Check Cluster Service Health

```powershell
# Check cluster service on all nodes
Invoke-Command -ComputerName Node1,Node2,Node3 -ScriptBlock {
    Get-Service ClusSvc | Select MachineName, Status, StartType
}

# If cluster service is stopped, check why
Get-WinEvent -FilterHashtable @{LogName='System'; ProviderName='FailoverClustering'} -MaxEvents 20 |
    Format-Table TimeCreated, Id, Message -Wrap
```

**Common reasons cluster service won't start:**
- Quorum lost (can't form cluster)
- Network connectivity issue (can't reach other nodes)
- Corrupted cluster database (rare)
- Certificate issues (Kerberos/cluster authentication)

**Force start (last resort):**
```powershell
# Force quorum on a single node (DANGER — only if majority lost)
net start clussvc /forcequorum
# Or in PowerShell:
Start-ClusterNode -Name Node1 -FixQuorum
```

## 📌 Step 2: Node Membership Issues

### Node Shows "Down" or Was Evicted
```powershell
# Check node state
Get-ClusterNode | Format-Table Name, State, StatusInformation

# States:
#   Up        — Normal, participating
#   Down      — Not communicating (may have crashed or lost network)
#   Paused    — Manually paused, won't accept new workloads
#   Joining   — In process of joining
```

### Common Cause: Heartbeat Loss
The cluster uses heartbeats to detect node health. If a node misses heartbeats for **5 seconds** (default), other nodes start isolation detection. After the **SameSubnetThreshold** (default=10 missed heartbeats), the node is removed.

```powershell
# Check heartbeat settings
(Get-Cluster).SameSubnetDelay        # Delay between heartbeats (ms), default=1000
(Get-Cluster).SameSubnetThreshold    # Missed heartbeats before marking down, default=10
(Get-Cluster).CrossSubnetDelay       # For multi-site, default=1000
(Get-Cluster).CrossSubnetThreshold   # For multi-site, default=20
```

**Causes of heartbeat loss:**
- NIC failure / cable disconnected
- Network switch issue / port flapping
- High CPU causing delayed heartbeat response
- NIC driver bug (dropped packets)
- VMQ / RSS misconfiguration on physical NIC

## 📌 Step 3: Quorum Health

```powershell
# Check quorum state
Get-ClusterQuorum

# Check each node's vote
Get-ClusterNode | Format-Table Name, NodeWeight, DynamicWeight, State
```

**Quorum lost scenarios:**
| Scenario | Result |
|----------|--------|
| 2-node + witness: both nodes up, witness fails | ✅ Cluster runs (2/3 votes) |
| 2-node + witness: 1 node down, witness up | ✅ Cluster runs (2/3 votes) |
| 2-node + witness: 1 node down + witness down | ❌ Quorum lost (1/3 votes) |
| 3-node no witness: 1 node down | ✅ Cluster runs (2/3 votes) |
| 3-node no witness: 2 nodes down | ❌ Quorum lost (1/3 votes) |

**Fix quorum loss:**
```powershell
# Option 1: Bring failed node/witness back online
# Option 2: Force quorum on surviving node(s)
Start-ClusterNode -Name SurvivingNode -FixQuorum

# Option 3: Change witness (if witness is the problem)
Set-ClusterQuorum -CloudWitness -AccountName "storageaccount" -AccessKey "key"
```

## 📌 Step 4: Resource Failures

### Identify Failed Resources
```powershell
# Check all resource states
Get-ClusterResource | Format-Table Name, ResourceType, OwnerGroup, OwnerNode, State

# States: Online, Offline, Failed, Pending
# Look for "Failed" resources

# Get details on a failed resource
Get-ClusterResource "ResourceName" | Format-List *
```

### Understand Failover Behavior
When a resource fails:
1. Cluster tries to **restart** it on the same node (IsAlive check)
2. If restart fails → **failover** the entire group to another node
3. If failover also fails → resource enters **Failed** state

```powershell
# Check restart/failover settings for a resource
Get-ClusterResource "ResourceName" | 
    Format-List RestartAction, RestartPeriod, RestartThreshold, RetryPeriodOnFailure
```

### Common Resource Failures
| Resource Type | Common Cause | Quick Check |
|--------------|-------------|-------------|
| **Disk / CSV** | Storage connectivity, SCSI reservation | `Get-ClusterSharedVolume`, check disk health |
| **Network Name** | DNS registration failed, CNO disabled in AD | Check Computer Object in AD |
| **IP Address** | IP conflict, NIC issue | `Test-NetConnection`, check NIC binding |
| **Generic Service** | Service binary missing, dependency failed | Check service dependencies |
| **VM** | VM config path inaccessible, resource exhaustion | Check Hyper-V event log |

## 📌 Step 5: Collecting and Reading Cluster Logs

### Generate Cluster Log
```powershell
# Standard log (last 60 minutes)
Get-ClusterLog -Destination C:\temp\ -TimeSpan 60

# With more detail
Get-ClusterLog -Destination C:\temp\ -TimeSpan 60 -UseLocalTime

# ⚠️ Always collect from ALL nodes simultaneously for correlation
```

### Key Patterns to Search in Cluster Log

```
# Node removal — search for:
"lost communication"
"is no longer an active member"
"removing node"

# Resource failure — search for:
"resource has failed"
"IsAlive check"
"LooksAlive check"
"exceeded its restart threshold"

# Quorum — search for:
"quorum lost"
"no longer has a quorum"
"witness"
"vote"

# Network — search for:
"interface is unreachable"
"route is unreachable"
"missed heartbeat"
```

## 📌 Quick Reference: Cluster Validation

```powershell
# Full validation (some tests are disruptive!)
Test-Cluster -Node Node1,Node2

# Non-disruptive tests only
Test-Cluster -Node Node1,Node2 -Include "Inventory","Network","System Configuration"

# Storage tests (WARNING: takes storage offline briefly)
Test-Cluster -Node Node1,Node2 -Include "Storage"
```

**⚠️ Always ask the customer before running storage validation** — it temporarily takes disks offline!
