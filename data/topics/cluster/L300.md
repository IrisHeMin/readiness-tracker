## 📌 Cluster Service Internals

```
clussvc.exe (Cluster Service)
├── Global Update Manager (GUM)
│   └── Ensures consistent state across all nodes
│   └── All configuration changes go through GUM as "updates"
├── Membership Manager
│   └── Heartbeat detection
│   └── Node join/leave decisions
│   └── Regroup process (determines new membership)
├── Failover Manager (FM)
│   └── Decides WHERE resources run
│   └── Handles failover logic and preferred owners
├── Resource Control Manager (RCM)
│   └── Communicates with RHS.exe processes
│   └── Manages resource state machine
└── Database Manager
    └── Cluster hive (HKLM\Cluster)
    └── Replicated across all nodes via GUM
```

### RHS.exe (Resource Hosting Subsystem)
- Each resource type runs inside an **RHS.exe** process
- If RHS crashes → all resources it hosts fail
- **IsAlive** check: deep health check (e.g., can the service respond?)
- **LooksAlive** check: lightweight check (e.g., is the process running?)

```powershell
# Check which RHS processes are running and what they host
Get-WmiObject -Namespace root\MSCluster -Class MSCluster_Resource |
    Select Name, State, SeparateMonitor | Format-Table

# A resource with SeparateMonitor=True runs in its own RHS.exe
# This isolates it so if it crashes, other resources are unaffected
```

## 📌 The Regroup Process (Node Membership Changes)

When a communication failure is detected, nodes enter a **regroup** — a protocol to determine the new cluster membership.

```
Timeline of a node failure:
─────────────────────────────────────────────────────
T=0s    Last successful heartbeat from Node2
T=1-5s  Missed heartbeats accumulate
T=5s    Node1 starts "regroup" process
T=5-8s  Regroup stages:
         Stage 1: Cleanup — abort previous regroup if any
         Stage 2: Join arbitration
         Stage 3: Exchange membership view
         Stage 4: Verify quorum
         Stage 5: Stabilize — commit new membership
T=8-10s  Node2 declared DOWN
         Failover Manager starts moving resources
─────────────────────────────────────────────────────
```

### Analyzing Regroup in Cluster Log
```
Search for: "[MM] regroup"

Key entries:
"Initiating regroup, reason: missed heartbeat"
"Regroup stage 2: Join arbitration"
"Regroup succeeded with X nodes"
"Node Y is no longer a member"
```

## 📌 CSV (Cluster Shared Volumes) Deep Dive

### How CSV Works
```
Normal I/O Path (Direct I/O):
┌──────────┐    ┌──────────────┐    ┌─────────────┐
│   App    │ →  │  CSVFS.sys   │ →  │   NTFS.sys  │ → Storage
│ (any     │    │ (CSV filter) │    │ (local disk │
│  node)   │    │              │    │  stack)     │
└──────────┘    └──────────────┘    └─────────────┘

Redirected I/O Path (when direct I/O unavailable):
┌──────────┐    ┌──────────────┐    ┌──────────┐    ┌──────────────┐
│   App    │ →  │  CSVFS.sys   │ →  │  SMB over │ →  │ Coordinator  │ → Storage
│ (non-    │    │ (detects     │    │  network  │    │   Node       │
│  coord   │    │  redirect)   │    │           │    │ (NTFS owner) │
│  node)   │    └──────────────┘    └──────────┘    └──────────────┘
```

### CSV States
```powershell
Get-ClusterSharedVolume | Format-Table Name, State, OwnerNode
Get-ClusterSharedVolumeState | Format-Table Name, Node, StateInfo, FileSystemRedirectedIOReason
```

| State | Meaning |
|-------|---------|
| **Direct** | Normal, healthy — I/O goes directly to storage |
| **Redirected** | I/O routed through coordinator node via SMB — SLOW |
| **Maintenance** | Disk in maintenance mode (for chkdsk, etc.) |

**Common causes of CSV redirected mode:**
- Storage connectivity issue on non-coordinator node
- Incompatible filter driver
- NTFS metadata operation in progress
- Disk in maintenance mode on the coordinator

```powershell
# Force CSV back to direct mode (if safe)
Get-ClusterSharedVolume "CSV1" | Resume-ClusterResource
```

## 📌 Complex Quorum Scenarios

### Dynamic Quorum & Dynamic Witness
Starting in Server 2012 R2, cluster uses **dynamic quorum** — it automatically adjusts votes as nodes go down to maximize availability.

```powershell
# Check dynamic quorum settings
(Get-Cluster).DynamicQuorum          # 1 = enabled (default)
Get-ClusterNode | Format-Table Name, DynamicWeight, NodeWeight, State
# DynamicWeight=0 means this node's vote was dynamically removed
```

### Multi-Site Cluster with Cloud Witness
```
Site A (Primary)          Azure          Site B (DR)
┌──────┐ ┌──────┐    ┌──────────┐    ┌──────┐ ┌──────┐
│Node1 │ │Node2 │    │  Cloud   │    │Node3 │ │Node4 │
│ (1)  │ │ (1)  │    │ Witness  │    │ (1)  │ │ (1)  │
└──────┘ └──────┘    │   (1)    │    └──────┘ └──────┘
                     └──────────┘
Total votes: 5    Majority: 3
Site A failure: Node3 + Node4 + CloudWitness = 3 votes → ✅ Survives
```

### Forced Quorum Recovery
When majority is lost and you MUST bring the cluster up:
```powershell
# 1. Stop cluster service on ALL surviving nodes
Stop-Service ClusSvc -Force

# 2. Start with forced quorum on ONE node
Start-ClusterNode -Name SurvivingNode1 -FixQuorum

# 3. Add other surviving nodes normally
Start-ClusterNode -Name SurvivingNode2

# 4. Fix the root cause (bring back failed nodes/witness)
# 5. CRITICAL: Do NOT start the failed nodes until you understand why they failed
#    They may have stale data and could cause data corruption
```

## 📌 Cluster Network Driver Stack

```
Kernel Network Stack with Cluster:
┌─────────────────────────────┐
│  Applications               │
├─────────────────────────────┤
│  Winsock / TDI              │
├─────────────────────────────┤
│  NetFT.sys                  │  ← Cluster Network Fault Tolerant driver
│  (Virtual adapter)          │     Routes cluster traffic across NICs
├─────────────────────────────┤
│  TCPIP.sys                  │
├─────────────────────────────┤
│  NDIS / NIC Drivers         │
└─────────────────────────────┘
```

**NetFT.sys** provides network fault tolerance:
- If the primary cluster network fails, traffic fails over to another network
- Manages cluster heartbeat routing across available networks

### Network Troubleshooting
```powershell
# Check cluster network health
Get-ClusterNetwork | Format-Table Name, State, Role, Address

# Check cluster network interfaces per node
Get-ClusterNetworkInterface | Format-Table Node, Network, State, Address

# Test cluster network connectivity
Test-Cluster -Include "Network"
```

## 📌 Cluster-Aware Updating (CAU)

CAU orchestrates Windows Updates across cluster nodes with zero downtime:
```
1. Pause Node1 → Drain workloads → Install updates → Reboot → Resume
2. Pause Node2 → Drain workloads → Install updates → Reboot → Resume
3. Repeat for all nodes...
```

```powershell
# Check CAU status
Get-CauRun | Format-List *

# Common issue: CAU fails because a node can't drain workloads
# Check: Get-ClusterGroup — are there groups that can't move?
# Usually: a resource has "preferred owner" set to only one node
```

## 📌 Live Migration Troubleshooting (Hyper-V Clusters)

```powershell
# Check live migration settings
Get-VMHost | Select *Migration*

# Common failures:
# 1. "Not enough resources" — check memory availability on destination
# 2. "Incompatible processor" — check CPU compatibility mode
# 3. "Authentication error" — check CredSSP or Kerberos constrained delegation
# 4. Timeout — check network bandwidth (live migration uses its own network)
```

### Configure dedicated Live Migration network:
```powershell
# Set live migration to use specific network
Set-VMHost -VirtualMachineMigrationPerformanceOption SMB
# Or: Compression, TCPIP
```

## 📌 Edge Cases You'll See in Cases

### 1. "Phantom" Node — Node Stuck in "Down" State
```powershell
# Node is gone but cluster still shows it
# Must evict the node
Remove-ClusterNode -Name "GhostNode" -Force
```

### 2. Cluster Name Object (CNO) Disabled in AD
- If AD admin disables the CNO computer object → cluster name resolution fails
- Clients can't connect to the cluster virtual name
- Fix: Re-enable the computer object in AD, reset its password
```powershell
# From a cluster node, repair the CNO
Get-ClusterResource "Cluster Name" | Repair-ClusterResource
```

### 3. Split-Brain (Both Sides Think They're the Cluster)
- Should NOT happen with proper quorum, but can occur with misuse of `/forcequorum`
- **DANGER**: Both sides may write to shared storage → data corruption
- Fix: Immediately stop cluster service on one side, identify which side has latest data

### 4. Cluster Database Corruption
```powershell
# Symptoms: Cluster service crashes repeatedly, GUM errors in log
# Nuclear option: Restore cluster config from another healthy node
# Stop cluster service on affected node, delete:
#   C:\Windows\Cluster\CLUSDB
#   C:\Windows\Cluster\CLUSDB.LOG*
# Start cluster service — it will replicate DB from other nodes
```
