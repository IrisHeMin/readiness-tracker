## 1. What is CSV and Why It Exists

In a traditional Windows failover cluster, each physical disk is **owned by one node at a time**. If a virtual machine needs to move to another node, the disk resource must fail over first вҖ” causing a brief interruption while the new owner mounts the file system.

**Cluster Shared Volumes (CSV)** eliminate this limitation by allowing **all cluster nodes to read and write the same NTFS/ReFS volume simultaneously**.

| Feature | Traditional Cluster Disk | Cluster Shared Volume |
|---|---|---|
| Concurrent access | Single owner node | All nodes simultaneously |
| Live migration | Requires disk failover | Seamless вҖ” storage stays online |
| Mount point | Drive letter on owner only | `C:\ClusterStorage\VolumeN` on all nodes |
| Use case | SQL Server FCI, file server | Hyper-V, Scale-Out File Server (SOFS) |
| I/O path | Direct through owner | Direct or redirected through coordinator |

**Primary use cases:**
- **Hyper-V Live Migration** вҖ” VMs move between nodes without storage failover
- **Scale-Out File Server (SOFS)** вҖ” All nodes serve the same SMB shares simultaneously
- **Storage Spaces Direct (S2D)** вҖ” CSV is the default volume layer on top of virtual disks

### Key Components

| Component | Role |
|---|---|
| **Coordinator Node** | Owns the CSV volume resource. Handles all NTFS/ReFS metadata operations (file create, rename, delete). One coordinator per CSV volume. |
| **CSVFS Filter Driver** | Installed on every cluster node. Intercepts I/O requests and routes them вҖ” either directly to storage or through the coordinator via MDS. |
| **MDS (MetaData Server) Protocol** | SMB-based protocol used by non-coordinator nodes to send metadata requests (open, close, rename) to the coordinator node. |
| **NTFS/ReFS** | The underlying file system on the physical disk. Only the coordinator mounts it natively; other nodes see it through CSVFS. |

**How it works:**
1. The coordinator node mounts the volume with full NTFS/ReFS access
2. Non-coordinator nodes mount the volume through the CSVFS filter driver
3. Metadata operations (file create, delete, rename) are forwarded to the coordinator via MDS
4. Data I/O (read/write blocks) goes directly to storage in normal operation

| I/O Mode | Data Path | Metadata Path | When It Happens | Performance |
|---|---|---|---|---|
| **Direct I/O** | All nodes вҶ’ Storage | Non-coord вҶ’ Coordinator via MDS | Normal operation | вң… Best |
| **Redirected I/O** (Block Redirected) | All I/O вҶ’ Coordinator вҶ’ Storage | Through coordinator | Storage connectivity lost on non-coord node, disk in maintenance | вқҢ Degraded вҖ” all I/O funneled through one node + network |
| **File System Redirected I/O** | All nodes вҶ’ Storage | Non-coord вҶ’ Coordinator via MDS (heavier) | NTFS dirty bit set, chkdsk needed, volume not cleanly mounted | вҡ пёҸ Moderate impact вҖ” data still direct |

> **Key takeaway:** Direct I/O is the healthy state. Any form of redirection adds latency and reduces throughput. Always investigate why a volume left Direct I/O mode.

## 4. CSV Namespace

CSV volumes are mounted under a **unified namespace** accessible from all cluster nodes:

```
C:\ClusterStorage\
в”ңв”Җв”Җ Volume1\       вҶҗ CSV Volume 1 (Coordinator: Node A)
в”Ӯ   в”ңв”Җв”Җ VM1\
в”Ӯ   в””в”Җв”Җ VM2\
в”ңв”Җв”Җ Volume2\       вҶҗ CSV Volume 2 (Coordinator: Node B)
в”Ӯ   в””в”Җв”Җ VM3\
в””в”Җв”Җ Volume3\       вҶҗ CSV Volume 3 (Coordinator: Node C)
    в””в”Җв”Җ VM4\
```

**Key points:**
- The path `C:\ClusterStorage\VolumeN` is **identical on every node** вҖ” applications use the same path regardless of which node they run on
- Each CSV volume has **one coordinator node** at a time, but the coordinator can be moved
- Volume numbering (`Volume1`, `Volume2`, etc.) is assigned when the volume is added to CSV and persists across reboots
- The `C:\ClusterStorage` folder is a **reparse point** managed by CSVFS вҖ” do not manually create or delete folders here

## 5. Essential Commands

```powershell
# List all CSV volumes with their state and coordinator (owner)
Get-ClusterSharedVolume | Format-Table Name, State, OwnerNode

# Check detailed CSV state including I/O mode
Get-ClusterSharedVolume | Get-ClusterSharedVolumeState

# Move a CSV volume coordinator to a different node
Move-ClusterSharedVolume -Name "Cluster Disk 1" -Node "NodeB"

# Add a cluster disk to CSV
Add-ClusterSharedVolume -Name "Cluster Disk 5"

# Remove a disk from CSV (returns to regular cluster disk)
Remove-ClusterSharedVolume -Name "Cluster Disk 5"

# Check CSV volume info from all nodes
Get-ClusterSharedVolumeState | Format-Table VolumeName, Node, StateInfo
```

### Quick Status Check

```powershell
# One-liner: Check if any CSV volumes are NOT in Direct I/O
Get-ClusterSharedVolume | Get-ClusterSharedVolumeState |
    Where-Object { $_.StateInfo -ne "Direct" } |
    Format-Table VolumeName, Node, StateInfo, FileSystemRedirectedIOReason
```
