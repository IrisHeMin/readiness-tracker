## 1. CSV Internals

### CSVFS Filter Driver Architecture

The CSVFS filter driver (`csvflt.sys`) is the core component that enables simultaneous multi-node access to a single volume. It operates differently depending on whether the local node is the coordinator:

```
  ┌─────────────────────────────────────────────────┐
  │                  Application I/O                 │
  └──────────────────────┬──────────────────────────┘
                         ▼
  ┌─────────────────────────────────────────────────┐
  │              CSVFS Filter Driver (csvflt.sys)    │
  │                                                  │
  │  ┌────────────────┐    ┌──────────────────────┐  │
  │  │  Metadata Ops   │    │     Data I/O Ops     │  │
  │  │  (Create, Del,  │    │  (Read, Write blocks)│  │
  │  │   Rename, ACL)  │    │                      │  │
  │  └───────┬────────┘    └──────────┬───────────┘  │
  └──────────┼────────────────────────┼──────────────┘
             │                        │
     ┌───────▼───────┐       ┌───────▼───────┐
     │ Am I the      │       │ Direct I/O    │
     │ coordinator?  │       │ mode?         │
     │               │       │               │
     │ YES → Local   │       │ YES → Direct  │
     │        NTFS   │       │   to storage  │
     │               │       │               │
     │ NO  → MDS     │       │ NO  → Route   │
     │   to coord.   │       │   via coord.  │
     └───────────────┘       └───────────────┘
```

### MDS Protocol Details

The **MetaData Server (MDS)** protocol handles file system metadata operations between non-coordinator nodes and the coordinator:

| Operation Type | MDS Handling | Notes |
|---|---|---|
| File Create / Open | Forwarded to coordinator | Coordinator performs NTFS create, returns handle metadata |
| File Delete | Forwarded to coordinator | Coordinator deletes from NTFS, notifies all nodes |
| File Rename / Move | Forwarded to coordinator | Serialized through coordinator to prevent conflicts |
| ACL / Security Changes | Forwarded to coordinator | Only coordinator writes security descriptors |
| File Read / Write (Direct) | **NOT** sent via MDS | Goes directly to storage from the requesting node |
| File Read / Write (Redirected) | Sent via SMB 3.0 to coordinator | Coordinator performs actual disk I/O on behalf of requesting node |

**MDS transport:** MDS runs over SMB 3.0, using the cluster network. Each non-coordinator node maintains an MDS session to the coordinator for each CSV volume.

### How Direct I/O Bypass Works

In Direct I/O mode, the CSVFS filter driver on a non-coordinator node:

1. **Intercepts** the I/O request from the application
2. **Checks** if the operation is metadata or data
3. For **data I/O**: Translates the CSV path to the underlying volume offset and issues the I/O request directly to the storage stack (through the local SCSI/storage miniport)
4. For **metadata**: Sends an MDS request to the coordinator over SMB 3.0
5. The coordinator's NTFS stack processes the metadata operation and returns the result

This means in Direct I/O mode, data never traverses the cluster network — only metadata does. This is why Direct I/O has significantly better performance than redirected modes.

## 2. CSV Cache (Block Cache)

### How It Works

CSV Block Cache is a **read-through cache** that uses unused physical memory on each cluster node to cache frequently accessed disk blocks from CSV volumes. It operates at the block level below the file system.

```
  Application Read Request
          │
          ▼
  ┌──────────────────┐
  │  CSVFS Filter     │
  │                    │
  │  ┌──────────────┐ │
  │  │ Block Cache   │ │    Cache HIT → Return from RAM
  │  │ (RAM-based)   │─┼──────────────────────────────►
  │  └──────┬───────┘ │
  │         │         │    Cache MISS → Read from disk,
  │         ▼         │    store in cache, return
  │  ┌──────────────┐ │
  │  │ Storage Stack │─┼──────────────────────────────►
  │  └──────────────┘ │
  └──────────────────┘
```

### Configuration

```powershell
# Check current block cache size (in MB, 0 = disabled)
(Get-Cluster).BlockCacheSize

# Enable block cache — set to 2048 MB (2 GB) per node
# This takes effect after a restart of the cluster service on each node
(Get-Cluster).BlockCacheSize = 2048

# Common sizes:
#   512  MB — Light caching, minimal memory impact
#   1024 MB — Moderate caching, good for mixed workloads
#   2048 MB — Aggressive caching, recommended for read-heavy VDI
#   4096 MB — Maximum common setting for large memory hosts

# Verify the setting
(Get-Cluster).BlockCacheSize
```

> **Note:** Changes to `BlockCacheSize` require restarting the Cluster Service on each node to take effect. Plan for a rolling restart in production.

### When Block Cache Helps vs. Doesn't Help

| Workload | Cache Benefit | Reason |
|---|---|---|
| **VDI (Virtual Desktop)** — boot storm | ✅ High | Many VMs read the same OS blocks (parent VHDX), cache serves repeated reads |
| **Random read workloads** | ✅ High | Frequently accessed blocks stay in RAM |
| **Read-heavy databases** (read replicas) | ✅ Moderate | Hot data pages cached in RAM |
| **Sequential large reads** (backup, copy) | ❌ Low | Each block read once, cache turnover too fast |
| **Write-heavy workloads** | ❌ None | Block cache is read-only; writes bypass cache entirely |
| **Workloads with own caching** (SQL Buffer Pool) | ❌ Wasted RAM | Application already caches in RAM; double-caching wastes memory |

### Monitoring Block Cache Performance

```powershell
# Check block cache hit ratio using PerfMon counters
Get-Counter "\Cluster CSV Volume Cache\Cache IO Read - Bytes" -SampleInterval 5 -MaxSamples 3
Get-Counter "\Cluster CSV Volume Cache\IO Read - Bytes" -SampleInterval 5 -MaxSamples 3

# A high ratio of Cache IO Read / Total IO Read indicates effective caching
```

## 3. CSV and BitLocker

CSV volumes can be encrypted with BitLocker, but the configuration has specific requirements:

### Requirements

| Requirement | Detail |
|---|---|
| **Key Protector** | Must use **Active Directory Account or Group protector** (ADAccountOrGroup) — TPM-only protection does not work because multiple nodes need the key |
| **Cluster Name Object (CNO)** | The CNO computer account is typically used as the protector identity |
| **Enable before adding to CSV** | Best practice: encrypt the volume while it's a regular cluster disk, then add to CSV |
| **All nodes must have BitLocker feature** | Install the BitLocker feature on every cluster node |

### Setup Procedure

```powershell
# Step 1: Install BitLocker on all nodes (if not present)
Install-WindowsFeature -Name BitLocker -IncludeManagementTools -Restart

# Step 2: Suspend CSV (make it a regular disk temporarily)
Remove-ClusterSharedVolume -Name "Cluster Disk 1"

# Step 3: Enable BitLocker with AD protector (run on owner node)
$CNO = "CLUSTERNAME$"  # Cluster Name Object
Enable-BitLocker -MountPoint "E:" -ADAccountOrGroupProtector -ADAccountOrGroup "DOMAIN\$CNO"

# Step 4: Wait for encryption to complete
Get-BitLockerVolume -MountPoint "E:" | Select-Object EncryptionPercentage

# Step 5: Add back to CSV
Add-ClusterSharedVolume -Name "Cluster Disk 1"

# Step 6: Verify BitLocker status across nodes
Invoke-Command -ComputerName NodeA, NodeB, NodeC -ScriptBlock {
    Get-BitLockerVolume | Where-Object VolumeType -eq "Data" |
    Select-Object MountPoint, ProtectionStatus, EncryptionPercentage
}
```

### Limitations

- BitLocker on CSV adds **5-15% CPU overhead** for encryption/decryption
- **Key rotation** requires suspending encryption on all nodes, rotating, then resuming
- Not supported with **CSV Block Cache** — cache is disabled on encrypted volumes

## 4. CSV with Storage Spaces Direct

### How CSV Sits on Top of S2D

In a Storage Spaces Direct (S2D) cluster, CSV operates on top of virtual disks created by Storage Spaces:

```
  ┌────────────────────────────────────────────────┐
  │         Applications (Hyper-V VMs, SOFS)        │
  ├────────────────────────────────────────────────┤
  │              CSV Layer (CSVFS)                  │
  │         C:\ClusterStorage\Volume1..N            │
  ├────────────────────────────────────────────────┤
  │           ReFS / NTFS File System               │
  ├────────────────────────────────────────────────┤
  │         Storage Spaces Virtual Disks            │
  │     (Mirror, Parity, or Mirror-Accelerated)     │
  ├────────────────────────────────────────────────┤
  │            Storage Pool (S2D)                   │
  │    Aggregated local NVMe / SSD / HDD drives     │
  ├────────────────────────────────────────────────┤
  │          Software Storage Bus (SSB)             │
  │   Connects local storage across cluster nodes   │
  └────────────────────────────────────────────────┘
```

### S2D-Specific CSV Behaviors

| Aspect | Behavior with S2D |
|---|---|
| **Direct I/O** | Still applies — data I/O from non-coordinator goes to the local S2D stack, which routes to the correct node(s) holding the data |
| **Redirection** | Can happen for CSV reasons (dirty bit, maintenance) OR S2D reasons (storage rebuild, disk failure) |
| **ReFS preferred** | Microsoft recommends ReFS on S2D for CSV volumes (integrity streams, auto-repair) |
| **Storage tiers** | CSV volumes on S2D can span SSD and HDD tiers; tiering operations may cause transient File System Redirected I/O |
| **Resiliency** | S2D handles disk failures transparently below CSV; CSV may briefly redirect during rebuild |

### Performance Considerations

```powershell
# Check S2D virtual disk health under CSV
Get-VirtualDisk | Format-Table FriendlyName, OperationalStatus, HealthStatus, ResiliencySettingName

# Check storage job progress (rebuild after disk failure)
Get-StorageJob | Format-Table Name, JobState, PercentComplete

# Verify CSV volumes on top of S2D are healthy
Get-ClusterSharedVolume | ForEach-Object {
    $csv = $_
    $state = $_ | Get-ClusterSharedVolumeState
    [PSCustomObject]@{
        Name       = $csv.Name
        OwnerNode  = $csv.OwnerNode
        State      = $state.StateInfo | Select-Object -First 1
        FreeSpace  = [math]::Round(($csv.SharedVolumeInfo.Partition.FreeSpace / 1GB), 2)
    }
} | Format-Table -AutoSize
```

## 5. Performance Analysis

### CSV-Specific PerfMon Counters

| Counter Set | Key Counters | What They Indicate |
|---|---|---|
| **Cluster CSV Volume Cache** | `Cache IO Read - Bytes`, `IO Read - Bytes`, `Direct IO Read - Bytes` | Block cache effectiveness: compare cache reads vs. total reads to calculate hit ratio |
| **Cluster CSVFS** | `Reads/sec`, `Writes/sec`, `Read Bytes/sec`, `Write Bytes/sec` | Overall CSV volume I/O throughput and IOPS |
| **Cluster CSVFS** | `Redirected Read Bytes/sec`, `Redirected Write Bytes/sec` | Volume of I/O being redirected — should be 0 in healthy Direct I/O state |
| **Cluster CSVFS** | `Read Latency`, `Write Latency` | CSV-level latency; compare with physical disk latency to isolate CSV overhead |
| **Cluster CSV Volume Manager** | `IO Reads/sec`, `IO Writes/sec` | Volume manager layer performance |

### Collecting CSV Performance Data

```powershell
# Capture CSV-specific counters for 60 seconds
$counters = @(
    "\Cluster CSV Volume Cache(*)\*"
    "\Cluster CSVFS(*)\*"
    "\Cluster CSV Volume Manager(*)\*"
    "\PhysicalDisk(*)\Avg. Disk sec/Read"
    "\PhysicalDisk(*)\Avg. Disk sec/Write"
)

Get-Counter -Counter $counters -SampleInterval 5 -MaxSamples 12 |
    Export-Counter -Path "C:\Logs\CSV_PerfData.blg" -FileFormat BLG

# Quick check: Is any CSV I/O being redirected?
Get-Counter "\Cluster CSVFS(*)\Redirected Read Bytes/sec" -SampleInterval 2 -MaxSamples 5
Get-Counter "\Cluster CSVFS(*)\Redirected Write Bytes/sec" -SampleInterval 2 -MaxSamples 5
```

## 6. Edge Cases

### CSV Maintenance Mode

When you need to perform maintenance on a CSV volume (chkdsk, extend, shrink), use `Suspend-ClusterResource`:

```powershell
# Put CSV volume in maintenance mode — I/O is paused or redirected
Suspend-ClusterResource -Name "Cluster Disk 1" -VolumeName "C:\ClusterStorage\Volume1" -RedirectedAccess

# Perform maintenance (e.g., chkdsk)
chkdsk C:\ClusterStorage\Volume1 /F

# Resume normal operation
Resume-ClusterResource -Name "Cluster Disk 1" -VolumeName "C:\ClusterStorage\Volume1"

# Alternative: Full drain (pauses all VMs on the volume first)
Suspend-ClusterResource -Name "Cluster Disk 1" -Force
```

### CSV Volume Corruption Recovery

| Scenario | Recovery Steps |
|---|---|
| **NTFS dirty bit set** | 1. Enter maintenance mode → 2. `chkdsk /F` → 3. Resume |
| **Metadata inconsistency** | 1. Move coordinator → 2. If persists, `Repair-ClusterSharedVolume` → 3. If still failing, remove from CSV, run full `chkdsk /R`, re-add |
| **Volume won't come online** | 1. Check event logs for disk errors → 2. Verify storage connectivity from all nodes → 3. Try `Start-ClusterResource` → 4. If failed, check underlying physical disk health |
| **Data corruption (S2D)** | 1. Check `Get-VirtualDisk` health → 2. Check `Get-PhysicalDisk` for failed drives → 3. Let S2D auto-repair → 4. If needed, `Repair-VirtualDisk` |

### Handling Stuck CSV Resources

```powershell
# If a CSV resource is stuck in a failed or pending state:

# Step 1: Check the resource state
Get-ClusterResource "Cluster Disk 1" | Format-List State, OwnerNode

# Step 2: Try taking it offline and back online
Stop-ClusterResource "Cluster Disk 1"
Start-ClusterResource "Cluster Disk 1"

# Step 3: If still stuck, force offline (may interrupt VMs!)
Stop-ClusterResource "Cluster Disk 1" -Force

# Step 4: Move to a different node and try again
Move-ClusterResource "Cluster Disk 1" -Node "NodeB"
Start-ClusterResource "Cluster Disk 1"

# Step 5: Check for open handles preventing state change
Get-SmbOpenFile | Where-Object { $_.Path -match "ClusterStorage" }
```
