# L200: CSV Troubleshooting
**⏱ Estimated: 10 minutes | 🎯 Goal: Diagnose CSV I/O redirection and access issues**

---

## 1. CSV in Redirected I/O Mode

When a CSV volume leaves **Direct I/O** mode, all data I/O or metadata I/O (or both) gets routed through the coordinator node's network stack instead of going directly to storage. This causes:

- **Increased latency** — every I/O traverses the cluster network before reaching storage
- **Reduced throughput** — the coordinator node becomes a bottleneck
- **Single point of failure** — if the coordinator goes down, all I/O stops until failover completes

### Common Causes of Redirection

| Cause | Redirection Type | What Happened |
|---|---|---|
| Non-coordinator node lost SAN connectivity | **Block Redirected** | Node can't reach storage; routes I/O through coordinator |
| CSV volume placed in maintenance mode | **Block Redirected** | Admin ran `Suspend-ClusterResource` with drain option |
| NTFS dirty bit set on volume | **File System Redirected** | Unclean shutdown, sudden power loss, or NTFS detected corruption |
| Volume needs chkdsk | **File System Redirected** | File system marked for repair |
| Incompatible filter driver | **File System Redirected** | A filter driver (AV, backup agent) isn't CSV-aware |
| Disk in maintenance mode (`/blockredirected`) | **Block Redirected** | Intentional maintenance via cluster API |
| ReFS volume with specific conditions | **File System Redirected** | ReFS integrity streams or specific repair operations |

### How to Identify Current I/O Mode

```powershell
# Check the I/O state for all CSV volumes across all nodes
Get-ClusterSharedVolume | Get-ClusterSharedVolumeState |
    Format-Table VolumeName, Node, StateInfo -AutoSize

# Example output:
# VolumeName                Node    StateInfo
# ----------                ----    ---------
# Cluster Virtual Disk (CSV1) NodeA   Direct
# Cluster Virtual Disk (CSV1) NodeB   Direct
# Cluster Virtual Disk (CSV2) NodeA   FileSystemRedirected
# Cluster Virtual Disk (CSV2) NodeB   FileSystemRedirected
```

> **Important:** Check the state from **every node**. A volume might be in Direct I/O from the coordinator's perspective but redirected from another node.

## 2. Diagnosing Redirection Reasons

The `Get-ClusterSharedVolumeState` cmdlet provides detailed properties that tell you exactly **why** a volume entered redirection:

```powershell
# Get detailed redirection reasons
Get-ClusterSharedVolume | Get-ClusterSharedVolumeState |
    Select-Object VolumeName, Node, StateInfo,
                  FileSystemRedirectedIOReason,
                  BlockRedirectedIOReason |
    Format-List

# Example output:
# VolumeName                 : Cluster Virtual Disk (CSV2)
# Node                       : NodeB
# StateInfo                  : FileSystemRedirected
# FileSystemRedirectedIOReason : FileSystemOnDiskNotCleanlyShutdown
# BlockRedirectedIOReason      : NotBlockRedirected
```

### Common Redirection Reasons and Fixes

| StateInfo | Reason Property Value | Root Cause | Fix |
|---|---|---|---|
| `FileSystemRedirected` | `FileSystemOnDiskNotCleanlyShutdown` | NTFS dirty bit set after unclean shutdown | Run `chkdsk /F` on the volume, then resume |
| `FileSystemRedirected` | `IncompatibleFileSystemFilter` | Non-CSV-aware filter driver loaded | Identify and update/remove the offending driver |
| `FileSystemRedirected` | `FileSystemTiering` | Storage tiering operation in progress | Wait for tiering to complete |
| `BlockRedirected` | `UserRequest` | Admin explicitly put volume in maintenance | Resume with `Resume-ClusterResource` |
| `BlockRedirected` | `NoDiskConnectivity` | Node lost access to the underlying disk | Check SAN connectivity, HBA, multipath |
| `BlockRedirected` | `VolumeMaintenance` | Volume undergoing maintenance operation | Wait for maintenance or manually resume |
| `BlockRedirected` | `IncompatibleDiskFilter` | Disk-level filter driver not CSV-aware | Update or remove the driver |

### Fix Workflow

```powershell
# Step 1: Identify the problem
$state = Get-ClusterSharedVolume "Cluster Disk 1" | Get-ClusterSharedVolumeState
$state | Format-List StateInfo, *Reason*

# Step 2: For dirty bit / chkdsk needed
# Put volume in maintenance mode first
Suspend-ClusterResource "Cluster Disk 1"
# Run chkdsk (from coordinator node)
chkdsk C:\ClusterStorage\Volume1 /F
# Resume the resource
Resume-ClusterResource "Cluster Disk 1"

# Step 3: Verify it returns to Direct I/O
Get-ClusterSharedVolume "Cluster Disk 1" | Get-ClusterSharedVolumeState
```

## 3. CSV Ownership Issues

### Moving the Coordinator

Each CSV volume has one coordinator node. Moving the coordinator can help with load balancing or troubleshooting:

```powershell
# Move coordinator to a specific node
Move-ClusterSharedVolume -Name "Cluster Disk 1" -Node "NodeB"

# Check current ownership
Get-ClusterSharedVolume | Format-Table Name, OwnerNode
```

### When Ownership Gets Stuck

| Symptom | Likely Cause | Resolution |
|---|---|---|
| `Move-ClusterSharedVolume` hangs | Open file handles on the volume | Identify processes with handles: `Get-SmbOpenFile` on coordinator |
| Coordinator fails, volume goes offline | No other node can take ownership | Check cluster quorum, verify storage connectivity on other nodes |
| Ownership constantly bounces | Preferred owner settings or cluster balancer | Check `Get-ClusterResource "Cluster Disk 1" \| Get-ClusterOwnerNode` |
| Volume stuck in "Offline" after coordinator crash | Cluster detects potential data corruption | Check event logs, may need `Repair-ClusterSharedVolume` |

### Impact of Coordinator Failure

When the coordinator node for a CSV volume fails:
1. The cluster detects the failure (via heartbeat timeout — typically 5-10 seconds)
2. A surviving node is elected as the new coordinator
3. The new coordinator mounts the NTFS/ReFS volume
4. I/O from all nodes resumes (may enter FileSystemRedirected briefly during recovery)

```powershell
# Monitor coordinator changes in real-time
Get-WinEvent -FilterHashtable @{LogName="Microsoft-Windows-FailoverClustering/Operational"; Id=5120,5121} -MaxEvents 20 |
    Format-Table TimeCreated, Id, Message -Wrap
```

## 4. CSV Access Problems

### Access from Non-Coordinator Nodes

| Problem | Cause | Fix |
|---|---|---|
| "Access Denied" opening files on CSV | NTFS permissions not propagated | Set permissions from the coordinator node; ensure all nodes have matching local accounts |
| Backup application fails on CSV | Backup agent not CSV-aware | Use VSS Hardware Provider or CSV-aware backup (e.g., System Center DPM, Veeam with CSV support) |
| Application can't follow reparse points | Legacy application doesn't understand CSVFS | Run the application on the coordinator node, or use UNC path instead |
| Slow file enumeration on CSV | Large number of files + metadata overhead | Consider restructuring directory layout; reduce file count per directory |

### Antivirus Exclusions for CSV

Antivirus software must be configured to exclude CSV paths and processes. Without exclusions, AV scanning can cause **File System Redirected I/O** or severe performance degradation.

**Required exclusions:**
```
# Paths
C:\ClusterStorage\                    (and all subfolders)
%systemroot%\Cluster\                 (cluster binaries)

# Processes
clussvc.exe        (Cluster Service)
rhs.exe            (Resource Hosting Subsystem)

# File types on CSV volumes
.vhd, .vhdx, .avhd, .avhdx          (virtual hard disks)
.vsv, .iso, .xml                     (VM configuration files)
```

## 5. Network Requirements

### CSV Network Architecture

CSV uses **SMB 3.0** for all redirected I/O and MDS (metadata) traffic between nodes. Proper network configuration is essential.

```
  Node A (Coordinator)              Node B
  ┌─────────────────┐              ┌─────────────────┐
  │                  │◄── CSV ────►│                  │
  │  CSV Traffic     │   Traffic    │  CSV Traffic     │
  │  (SMB 3.0)      │  (Redirect + │  (SMB 3.0)      │
  │                  │   Heartbeat) │                  │
  │  Cluster Network │              │  Cluster Network │
  │  (Heartbeat)     │◄── HB ─────►│  (Heartbeat)     │
  └─────────────────┘              └─────────────────┘
```

### Network Configuration Checklist

```powershell
# Check which networks are enabled for CSV
Get-ClusterNetwork | Format-Table Name, State, Role

# Role values:
# 0 = Not used by cluster
# 1 = Cluster only (heartbeat)
# 3 = Cluster and client (CSV can use this)

# Check CSV-specific network settings
(Get-Cluster).EnableSharedVolumes     # Should be "Enabled"

# View current CSV network metric (lower = preferred)
Get-ClusterNetwork | Format-Table Name, Metric, AutoMetric, Role
```

**Best practices:**
- Use **dedicated high-bandwidth networks** (10 GbE or higher) for CSV traffic
- Enable **SMB Multichannel** for CSV by having multiple NIC paths between nodes
- Ensure at least **two cluster networks** — one for CSV + client traffic, one for heartbeat
- If using RDMA (SMB Direct), ensure all nodes have matching RDMA-capable NICs

## 6. Event Logs

### Key Event Sources

CSV-related events are logged in:
- `Microsoft-Windows-FailoverClustering/Operational`
- `Microsoft-Windows-FailoverClustering/Diagnostic` (verbose, enable manually)

### Critical Event IDs

| Event ID | Meaning | Severity |
|---|---|---|
| **5120** | CSV volume transitioning to redirected I/O | ⚠️ Warning — investigate cause |
| **5121** | CSV volume returning to direct I/O | ℹ️ Info — good sign |
| **5125** | CSV volume entering maintenance mode | ℹ️ Info — expected during maintenance |
| **5126** | CSV volume exiting maintenance mode | ℹ️ Info — normal |
| **5140** | CSV volume snapshot created | ℹ️ Info |
| **5142** | CSV volume access denied from a node | 🔴 Error — check permissions |
| **5143** | CSV volume I/O timeout | 🔴 Error — check storage connectivity |
| **5144** | CSV volume I/O error | 🔴 Error — check storage health |

### Querying CSV Events

```powershell
# Get recent CSV redirection events
Get-WinEvent -FilterHashtable @{
    LogName = "Microsoft-Windows-FailoverClustering/Operational"
    Id = 5120, 5121
} -MaxEvents 50 | Format-Table TimeCreated, Id, Message -Wrap

# Get all CSV-related warnings and errors from the last 24 hours
Get-WinEvent -FilterHashtable @{
    LogName = "Microsoft-Windows-FailoverClustering/Operational"
    Level = 2,3   # Error and Warning
    StartTime = (Get-Date).AddHours(-24)
} | Where-Object { $_.Message -match "CSV|ClusterSharedVolume|SharedVolume" } |
    Format-Table TimeCreated, Id, LevelDisplayName, Message -Wrap
```
