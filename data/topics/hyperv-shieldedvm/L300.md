## 📌 HGS Architecture Deep Dive

HGS is a Windows Server role that runs two core services on a dedicated failover cluster:

```
┌──────────────────────────────────────────────────────────────────┐
│                    HGS Cluster (3+ nodes recommended)            │
│                                                                  │
│  ┌─────────────────────────┐   ┌──────────────────────────────┐  │
│  │   Attestation Service   │   │   Key Protection Service     │  │
│  │                         │   │                              │  │
│  │  - Validates host       │   │  - Wraps/unwraps transport   │  │
│  │    identity & health    │   │    keys (TKs)                │  │
│  │  - TPM quote validation │   │  - Uses signing cert to      │  │
│  │  - CI policy matching   │   │    prove KPS identity        │  │
│  │  - Issues health certs  │   │  - Uses encryption cert to   │  │
│  │                         │   │    decrypt key protectors    │  │
│  │  Endpoint:              │   │                              │  │
│  │  /Attestation           │   │  Endpoint:                   │  │
│  └─────────────────────────┘   │  /KeyProtection              │  │
│                                └──────────────────────────────┘  │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │                    Active Directory                          │ │
│  │  - HGS runs its own AD forest (isolated)                    │ │
│  │  - Stores attestation policies, registered hosts            │ │
│  │  - Replicated across all HGS nodes                          │ │
│  └─────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────┘
```

### Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| **Separate AD forest** | HGS must not be compromised if the fabric AD is compromised. Separate forest = separate trust boundary. |
| **3-node cluster** | Tolerates 1 node failure while maintaining quorum. HGS downtime = no new shielded VMs can start. |
| **HTTP + HTTPS endpoints** | Attestation metadata served over HTTP; actual attestation over HTTPS. Hosts must trust HGS SSL certificate. |
| **Stateless attestation** | Health certificates are short-lived (~8 hours). Hosts must re-attest periodically. |

### Attestation Service Internals

The attestation service validates different artifacts depending on the mode:

**TPM Attestation Artifacts:**

```
┌─────────────────────────────────────────────────────────┐
│              Artifacts Sent by Guarded Host              │
│                                                         │
│  1. TPM Endorsement Key (EK) certificate                │
│     └─► Proves "I have a genuine TPM"                   │
│                                                         │
│  2. TPM Attestation Identity Key (AIK) quote            │
│     └─► Signed by AIK, which chains to EK               │
│     └─► Contains PCR measurements                       │
│                                                         │
│  3. TCG Boot Log                                        │
│     └─► Full measured boot log from UEFI → OS           │
│     └─► HGS replays log to verify PCR values            │
│                                                         │
│  4. Code Integrity Policy hash                          │
│     └─► Hash of the applied CI policy on the host       │
│                                                         │
│  5. Secure Boot state                                   │
│     └─► Whether Secure Boot is enabled in firmware      │
│                                                         │
│  6. Boot configuration                                  │
│     └─► DEP, HVCI, VBS status                           │
└─────────────────────────────────────────────────────────┘
```

### Key Protection Service Internals

```
  Guarded Host                           KPS (on HGS)
  ────────────                       ─────────────────
       │                                    │
       │  1. Presents:                      │
       │     - Health certificate           │
       │     - VM's key protector           │
       │  ──────────────────────────────►   │
       │                                    │
       │                                    │  2. Validates health cert
       │                                    │     (signed by attestation svc)
       │                                    │
       │                                    │  3. Decrypts key protector using
       │                                    │     HGS encryption private key
       │                                    │
       │                                    │  4. Extracts transport key (TK)
       │                                    │
       │  5. Returns wrapped TK        ◄──  │
       │     (encrypted to host's           │
       │      transport session key)        │
       │                                    │
       │  6. Host decrypts TK              │
       │     and uses it to unlock          │
       │     vTPM state                     │
```

## 📌 TPM Attestation Flow in Detail

### PCR Measurements and What They Cover

Platform Configuration Registers (PCRs) in the TPM record measurements taken during the boot process:

| PCR | Measures | Why HGS Cares |
|-----|----------|---------------|
| PCR 0 | UEFI firmware code | Detects firmware tampering |
| PCR 1 | UEFI firmware configuration | Detects BIOS setting changes |
| PCR 2 | Option ROM code | Detects rogue expansion cards |
| PCR 4 | MBR / boot manager | Detects bootkit attacks |
| PCR 7 | Secure Boot policy | Verifies Secure Boot is enforced |
| PCR 11 | BitLocker access control | Verifies BitLocker integrity |
| PCR 14 | Boot authority | Identifies boot chain trust |

### The Full Attestation Handshake

```
  Guarded Host                                HGS Attestation Service
  ────────────                            ─────────────────────────
       │                                          │
       │  ── PHASE 1: Identity ─────────────────  │
       │                                          │
       │  1. Create AIK using TPM                 │
       │     (Attestation Identity Key)           │
       │                                          │
       │  2. Send EK cert + AIK public key        │
       │  ────────────────────────────────────►   │
       │                                          │  3. Verify EK cert against
       │                                          │     registered EK list
       │                                          │
       │                                          │  4. Issue AIK challenge
       │  ◄────────────────────────────────────   │     (encrypted to EK)
       │                                          │
       │  5. Decrypt challenge using TPM          │
       │     (proves TPM has matching EK)         │
       │  ────────────────────────────────────►   │
       │                                          │  6. AIK identity confirmed
       │                                          │
       │  ── PHASE 2: Health ───────────────────  │
       │                                          │
       │  7. TPM signs PCR values with AIK        │
       │     (called a "TPM quote")               │
       │  ────────────────────────────────────►   │
       │                                          │  8. Verify quote signature
       │  8. Send TCG boot log                    │     (trust AIK from Phase 1)
       │  ────────────────────────────────────►   │
       │                                          │  9. Replay boot log to verify
       │                                          │     PCR values match log
       │                                          │
       │                                          │  10. Compare CI policy hash
       │                                          │      against registered policies
       │                                          │
       │                                          │  11. Check Secure Boot, VBS,
       │                                          │      HVCI status
       │                                          │
       │  ── PHASE 3: Certificate ──────────────  │
       │                                          │
       │  ◄────────────────────────────────────   │  12. Issue health certificate
       │     (valid ~8 hours, signed by HGS)      │      (if ALL checks pass)
       │                                          │
```

## 📌 Code Integrity Policies and Their Role

### What CI Policies Do in the Shielded VM Context

Code Integrity (CI) policies (also called Windows Defender Application Control / WDAC policies) ensure that the guarded host only runs **approved code**. This prevents:

- A rogue admin from loading a malicious kernel driver that could read VM memory
- Malware from injecting code into the hypervisor
- Unauthorized software from weakening the security boundary

### CI Policy Lifecycle

```
  1. CAPTURE           2. SIGN              3. DEPLOY            4. REGISTER
  ─────────            ──────               ────────             ──────────
  Scan a "gold"   →   Sign policy with  →  Deploy signed    →   Register hash
  host to create       enterprise cert      policy to all        on HGS for
  baseline policy                           guarded hosts        attestation

  New-CIPolicy        Set-CIPolicy          Copy .p7b to         Add-HgsAttestation
  -Level              -Sign                 CodeIntegrity\       CIPolicy
  FilePublisher                             SIPolicy.p7b
```

### Audit Mode vs Enforce Mode

```powershell
# Create CI policy in audit mode first (won't block anything)
New-CIPolicy -FilePath "C:\Temp\CIPolicy.xml" `
    -Level FilePublisher -Fallback Hash `
    -Audit

# After validating no legitimate apps are blocked, switch to enforce
Set-RuleOption -FilePath "C:\Temp\CIPolicy.xml" -Option 3 -Delete
# Option 3 = "Enabled:Audit Mode" — deleting it enables enforcement

# Convert and deploy
ConvertFrom-CIPolicy -XmlFilePath "C:\Temp\CIPolicy.xml" `
    -BinaryFilePath "C:\Windows\System32\CodeIntegrity\SIPolicy.p7b"

# Restart required for enforcement
Restart-Computer
```

> **⚠️ Critical**: A CI policy that is too restrictive will prevent the host from booting properly. Always test in audit mode first. Always maintain a recovery plan.

## 📌 Disaster Recovery for HGS

### HGS Failure Impact

| HGS State | Impact on Running VMs | Impact on New VM Starts |
|-----------|----------------------|------------------------|
| All nodes down | ✅ Running VMs continue | ❌ New VMs cannot start |
| Health certs expired (~8hrs) | ❌ VMs cannot restart after reboot | ❌ New VMs cannot start |
| HGS data lost (no backup) | ❌ Catastrophic — all shielded VMs are permanently inaccessible | ❌ No attestation possible |

### Backup Strategy

```powershell
# ===== CRITICAL BACKUP ITEMS =====

# 1. Export HGS configuration (includes attestation policies)
Export-HgsServerState -Path "C:\HGSBackup\HgsServerState.xml"

# 2. Backup signing and encryption certificates WITH private keys
$signingCert = (Get-HgsServer).SigningCertificate
$encryptionCert = (Get-HgsServer).EncryptionCertificate

Export-PfxCertificate -Cert $signingCert `
    -FilePath "C:\HGSBackup\signing.pfx" `
    -Password (ConvertTo-SecureString -String "BackupP@ss!" -AsPlainText -Force)

Export-PfxCertificate -Cert $encryptionCert `
    -FilePath "C:\HGSBackup\encryption.pfx" `
    -Password (ConvertTo-SecureString -String "BackupP@ss!" -AsPlainText -Force)

# 3. Backup HGS Active Directory (System State backup)
wbadmin start systemstatebackup -backupTarget:E:

# 4. Document the attestation mode and registered hosts
Get-HgsAttestationTpmHost | Export-Csv "C:\HGSBackup\registered_hosts.csv"
Get-HgsAttestationCIPolicy | Export-Csv "C:\HGSBackup\ci_policies.csv"
```

### Recovery Procedure

```powershell
# On new HGS server — initialize with existing AD forest
Install-HgsServer -HgsDomainName "hgs.contoso.com" `
    -SafeModeAdministratorPassword (ConvertTo-SecureString "P@ssw0rd" -AsPlainText -Force)

# Import saved state
Import-HgsServerState -Path "C:\HGSBackup\HgsServerState.xml"

# Import certificates
Import-PfxCertificate -FilePath "C:\HGSBackup\signing.pfx" `
    -CertStoreLocation Cert:\LocalMachine\My `
    -Password (ConvertTo-SecureString "BackupP@ss!" -AsPlainText -Force)

# Initialize HGS with imported certs
Initialize-HgsServer -HgsServiceName "HgsService" `
    -SigningCertificateThumbprint $signingThumbprint `
    -EncryptionCertificateThumbprint $encryptionThumbprint `
    -TrustTpm
```

> **🔴 Key Point**: If you lose both the HGS data AND the certificates with no backup, all shielded VMs protected by that HGS are **permanently unrecoverable**. The VM disks are encrypted with keys that only HGS could release. Treat HGS backups with the same criticality as CA root key backups.

## 📌 Networking Considerations for HGS

### Network Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                    RECOMMENDED NETWORK DESIGN                │
│                                                             │
│  ┌──────────────┐         ┌──────────────┐                  │
│  │ Management   │         │ HGS Network  │                  │
│  │ Network      │         │ (Dedicated)  │                  │
│  │ 10.0.1.0/24  │         │ 10.0.2.0/24  │                  │
│  └──────┬───────┘         └──────┬───────┘                  │
│         │                        │                          │
│  ┌──────▼───────┐         ┌──────▼───────┐                  │
│  │ Guarded Host │─────────│  HGS Cluster │                  │
│  │  NIC 1       │  NIC 2  │  (3 nodes)   │                  │
│  └──────────────┘         └──────────────┘                  │
│                                                             │
│  KEY POINTS:                                                │
│  • HGS should be on a separate network/VLAN                 │
│  • Firewall: Allow TCP 80, 443 from guarded hosts to HGS   │
│  • HGS nodes communicate on TCP 445 (SMB), 135 (RPC),      │
│    and dynamic RPC ports for AD replication                  │
│  • DNS: Guarded hosts must resolve the HGS cluster name     │
└─────────────────────────────────────────────────────────────┘
```

### Firewall Rules

| Direction | Source | Destination | Port | Purpose |
|-----------|--------|-------------|------|---------|
| Inbound to HGS | Guarded Hosts | HGS VIP | TCP 80, 443 | Attestation & Key Protection |
| Inbound to HGS | HGS Nodes | HGS Nodes | TCP 445 | SMB (cluster) |
| Inbound to HGS | HGS Nodes | HGS Nodes | TCP 135, 49152-65535 | RPC / AD Replication |
| Outbound from HGS | HGS | Domain Controllers (HGS forest) | TCP 389, 636 | LDAP/LDAPS |

### DNS Configuration

```powershell
# HGS uses a distributed network name (cluster DNS)
# All guarded hosts must be able to resolve this name

# Verify DNS from a guarded host
Resolve-DnsName -Name "hgs.contoso.com" -Type A
# Should return IPs of ALL HGS cluster nodes

# If using conditional forwarders (separate AD forests):
Add-DnsServerConditionalForwarderZone -Name "hgs.contoso.com" `
    -MasterServers 10.0.2.10, 10.0.2.11, 10.0.2.12
```

## 📌 Linux Shielded VMs

Starting with Windows Server 2019, Hyper-V supports shielded Linux VMs.

### Key Differences from Windows Shielded VMs

| Feature | Windows Shielded VM | Linux Shielded VM |
|---------|-------------------|-------------------|
| Secure Boot Template | MicrosoftWindows | MicrosoftUEFICertificateAuthority |
| Disk Encryption | BitLocker | dm-crypt (LUKS) |
| vTPM | Yes | Yes |
| Boot process | UEFI → Bootmgr → Windows | UEFI → GRUB → Linux kernel |
| Provisioning | Shielding data (PDK) | Specialized Linux template disk |
| Console access (shielded mode) | Blocked | Blocked |

### Provisioning a Linux Shielded VM

```powershell
# 1. Prepare a Linux template VHDX using lsvmtools
# (Inside the Linux VM before shielding)
# Install lsvmtools package — handles dm-crypt integration with vTPM

# 2. On the guarded host — set Secure Boot template
Set-VMFirmware -VMName "LinuxShielded01" `
    -SecureBootTemplate MicrosoftUEFICertificateAuthority

# 3. Enable vTPM
Enable-VMTPM -VMName "LinuxShielded01"

# 4. Set VM security policy
$owner = New-HgsGuardian -Name "Owner" -GenerateCertificates
$guardian = Import-HgsGuardian -Path "C:\HGS\Guardian.xml" -Name "HGS"
$kp = New-HgsKeyProtector -Owner $owner -Guardian $guardian -AllowUntrustedRoot
Set-VMKeyProtector -VMName "LinuxShielded01" -KeyProtector $kp.RawData
Set-VMSecurityPolicy -VMName "LinuxShielded01" -Shielded $true
```

### Supported Linux Distributions

- Ubuntu 16.04 LTS and later
- Red Hat Enterprise Linux 7.x and later
- SUSE Linux Enterprise Server 12 SP2 and later

> **Note**: Linux shielded VMs require the `lsvmtools` package, which handles the dm-crypt key exchange with the vTPM during boot.

## 📌 Security Boundaries: What Shielded VMs Do NOT Protect Against

Understanding limitations is as important as understanding capabilities:

### Protected Against ✅

| Threat | How Protected |
|--------|---------------|
| Fabric admin reading VM disk | BitLocker encryption with vTPM-stored keys |
| Fabric admin inspecting VM memory | Memory encryption via VBS |
| VM running on unauthorized host | HGS attestation — keys only released to healthy hosts |
| Fabric admin attaching debugger | Shielded VM policy blocks debug access |
| Offline VHDX theft | BitLocker — disk unreadable without vTPM |
| Boot-level attacks on host | Measured Boot + Secure Boot + CI policy |

### NOT Protected Against ❌

| Threat | Why Not | Mitigation |
|--------|---------|------------|
| **Compromised guest OS** | Shielded VMs protect the VM from the host, not from itself | Patch guest OS, use endpoint protection |
| **Side-channel attacks** (Spectre, Meltdown) | Hardware-level attacks that cross VM boundaries | Apply microcode/OS patches, consider physical isolation for highest-sensitivity workloads |
| **Network traffic inspection** | Network packets leaving the VM are visible to the fabric | Use IPsec or TLS for all VM network traffic |
| **Denial of Service** | Fabric admin can still stop a VM, delete it, or disconnect networking | Operational controls, monitoring, SLAs |
| **HGS compromise** | If an attacker gains HGS admin access, they can approve rogue hosts | Harden HGS: separate forest, physical security, limited admin access, PAM |
| **Supply chain attacks** | Compromised firmware that reports fake TPM measurements | Use hardware from trusted vendors, verify firmware integrity |
| **Guest-to-guest attacks** | Shielded VMs don't add isolation between VMs on the same host | Use separate hosts for different trust levels |

### Security Architecture Summary

```
┌─────────────────────────────────────────────────────────────────┐
│                   SECURITY BOUNDARY MAP                          │
│                                                                 │
│  ┌────────────────────────────────────────┐                     │

│  │                                        │                     │
```

## ✅ L300 Checkpoint — You Should Now Know:

1. **HGS architecture** — separate AD forest, attestation service + key protection service on a failover cluster
2. **TPM attestation internals** — EK validation → AIK challenge → PCR quote verification → TCG log replay → health cert issuance
3. **PCR registers** — which PCRs measure what, and why HGS checks them
4. **CI policy role** — prevents unauthorized kernel code on guarded hosts; must be captured, signed, deployed, and registered
5. **HGS disaster recovery** — backup certs with private keys, export server state, backup AD; loss of both = permanent VM loss
6. **Network design** — dedicated VLAN, specific firewall rules, DNS resolution requirements
7. **Linux shielded VMs** — supported since Server 2019; use dm-crypt instead of BitLocker, require lsvmtools
8. **Security boundaries** — Shielded VMs protect against untrusted fabric but NOT against guest OS compromise, network sniffing, side-channel attacks, or DoS
