## 📌 What Are Shielded VMs and Why Do They Exist?

Shielded VMs solve a fundamental trust problem in virtualized environments: **what happens when you don't trust the fabric administrators?**

In traditional Hyper-V deployments, a fabric admin (anyone with access to the host) can:
- Inspect VM memory
- Attach a debugger to a running VM
- Copy and open VHDX files
- Connect to the VM console without guest credentials
- Take snapshots and export VMs

**Shielded VMs** encrypt the VM's state and virtual disks so that only authorized owners and designated trusted hosts can run them. Even a compromised or malicious Hyper-V administrator cannot access the VM's data.

### Threat Model: Untrusted Fabric

```
┌──────────────────────────────────────────────────────────────┐
│                    TRADITIONAL HYPER-V                        │
│                                                              │
│   Fabric Admin ──► Full access to VM disks, memory, console  │
│   Compromised Host ──► All VMs on host are exposed           │
│   Rogue Admin ──► Can copy/export any VM                     │
│                                                              │
│   TRUST BOUNDARY: You trust the fabric completely            │
├──────────────────────────────────────────────────────────────┤
│                    SHIELDED VMs                               │
│                                                              │
│   Fabric Admin ──► Cannot read VM disks (BitLocker)          │
│   Fabric Admin ──► Cannot inspect VM memory (encrypted)      │
│   Fabric Admin ──► Cannot connect to console (disabled)      │
│   Compromised Host ──► VM refuses to start (attestation)     │
│                                                              │
│   TRUST BOUNDARY: You trust the HOST HARDWARE, not the admin │
└──────────────────────────────────────────────────────────────┘
```

### Architecture Overview

```
                           │          │

              identity     │          │     keys
                           │          │

                    │     Guarded Host          │
                    │     (Hyper-V Server)      │
                    │                           │
                    │  ┌───────────────────┐    │
                    │  │   Shielded VM     │    │
                    │  │  ┌─────────────┐  │    │
                    │  │  │   vTPM      │  │    │
                    │  │  │  BitLocker  │  │    │

                    │  └───────────────────┘    │
                    └───────────────────────────┘
```

### Component Details

| Component | Role | Key Details |
|-----------|------|-------------|
| **Host Guardian Service (HGS)** | Central trust authority | Runs attestation and key protection services; typically 3-node cluster for HA |
| **Guarded Host** | Hyper-V host approved to run shielded VMs | Must pass attestation before receiving VM encryption keys |
| **Shielding Data (PDK)** | Provisioning data file | Contains owner certificate, VM trust policies, and secrets for VM creation |
| **Key Protector** | Wraps the VM's encryption keys | Stored in VM configuration; keys released only to attested hosts |
| **vTPM** | Virtual Trusted Platform Module | Provides BitLocker key storage inside the VM |

## 📌 How Attestation Works

Attestation is the process by which a guarded host **proves its identity and health** to HGS before being allowed to run shielded VMs.

### Attestation Flow

```
  Guarded Host                        HGS
  ────────────                    ─────────
       │                              │
       │  1. "I want to run a         │
       │     shielded VM"             │
       │  ─────────────────────────►  │
       │                              │
       │  2. "Prove your identity     │
       │     and health"              │
       │  ◄─────────────────────────  │
       │                              │
       │  3. Sends TPM quotes,        │
       │     CI policy, Secure Boot   │
       │     state, etc.              │
       │  ─────────────────────────►  │
       │                              │
       │  4. HGS validates evidence   │
       │     against registered       │
       │     policies                 │
       │                              │
       │  5. Issues health cert       │
       │     (valid ~8 hours)         │
       │  ◄─────────────────────────  │
       │                              │
       │  6. Host uses cert to        │
       │     request VM keys from     │
       │     Key Protection Service   │
       │  ─────────────────────────►  │
       │                              │
       │  7. VM encryption keys       │
       │     released                 │
       │  ◄─────────────────────────  │
       │                              │
       │  8. VM starts with           │
       │     decrypted vTPM           │
       │                              │
```

### Attestation Modes

| Mode | Trust Basis | Security Level | Use Case |
|------|-------------|----------------|----------|
| **TPM Attestation** | Hardware TPM 2.0 + Measured Boot | **Highest** | Production environments requiring maximum security |
| **Host Key Attestation** | Asymmetric key pair per host | **Medium** | Environments without TPM 2.0 or where TPM attestation is impractical |
| **AD Attestation** | Active Directory group membership | **⚠️ Deprecated** | Do NOT use — removed in Windows Server 2019+ |

> **⚠️ Important**: AD-based attestation was deprecated starting with Windows Server 2019. Microsoft recommends TPM attestation for production and Host Key attestation as a simpler alternative.

### Checking Attestation Mode

```powershell
# On the HGS server — check current attestation mode
Get-HgsServer

# On a guarded host — test attestation
Get-HgsClientConfiguration

# Expected output for a healthy host:
# IsHostGuarded             : True
# Mode                      : HostGuardianService
# KeyProtectionServerUrl    : https://hgs.contoso.com/KeyProtection
# AttestationServerUrl      : https://hgs.contoso.com/Attestation
# AttestationOperationMode  : Tpm
# AttestationStatus         : Passed
```

## 📌 VM Trust Levels: Shielded vs Encryption Supported

When creating a VM with a key protector, you choose one of two trust levels:

| Feature | Shielded | Encryption Supported |
|---------|----------|---------------------|
| vTPM | ✅ Yes | ✅ Yes |
| BitLocker (OS disk) | ✅ Enforced | ✅ Available (manual) |
| Secure Boot | ✅ Enforced | ⚙️ Configurable |
| VM Console (VMConnect) | ❌ Blocked | ✅ Allowed |
| PowerShell Direct | ❌ Blocked | ✅ Allowed |
| Integration Services | 🔒 Limited set | ✅ Full set |
| Live Migration | ✅ Only to guarded hosts | ✅ Only to guarded hosts |
| Fabric admin can inspect memory | ❌ No | ❌ No |
| Fabric admin can attach debugger | ❌ No | ❌ No |

> **When to use which?**
> - **Shielded**: Maximum protection. The VM owner doesn't trust fabric admins at all.
> - **Encryption Supported**: The VM needs encryption at rest, but the admin needs console access for troubleshooting.

## 📌 vTPM, BitLocker, and Secure Boot

### Virtual TPM (vTPM)

The vTPM is a **software-emulated TPM 2.0** provided by Hyper-V to the guest VM. It enables:
- BitLocker without a physical TPM in the guest
- Key storage for the guest OS
- Platform integrity measurements inside the VM

The vTPM state is encrypted and stored alongside the VM configuration. **Only an attested guarded host can decrypt the vTPM state.**

### BitLocker Integration

```
┌─────────────────────────────────────────┐
│            Shielded VM                   │
│                                          │
│   OS Volume ◄── BitLocker Encrypted     │
│       │                                  │
│       ▼                                  │
│   BitLocker Key ◄── Stored in vTPM      │
│       │                                  │
│       ▼                                  │
│   vTPM State ◄── Encrypted by Host Key  │
│       │                                  │
│       ▼                                  │
│   Host Key ◄── Released by HGS only     │
│                  after attestation       │
└─────────────────────────────────────────┘
```

### Secure Boot

Shielded VMs enforce Secure Boot with one of two templates:
- **MicrosoftWindows** — for Windows guest VMs
- **MicrosoftUEFICertificateAuthority** — for Linux guest VMs

```powershell
# Check Secure Boot template on a VM
Get-VMFirmware -VMName "ShieldedVM01" | Select-Object SecureBootTemplate

# Set Secure Boot template for Linux
Set-VMFirmware -VMName "LinuxShielded01" -SecureBootTemplate MicrosoftUEFICertificateAuthority
```

## 📌 Quick Reference: PowerShell Commands

```powershell
# === On HGS Server ===
# Check HGS configuration
Get-HgsServer

# View registered TPM certificates
Get-HgsAttestationTpmHost

# View code integrity policies
Get-HgsAttestationCIPolicy

# === On Guarded Host ===
# Test if host passes attestation
Get-HgsClientConfiguration

# View VM shielding type
Get-VM -Name "MyVM" | Get-VMSecurity

# === VM Provisioning ===
# Create shielding data file (PDK)
New-ShieldingDataFile -ShieldingDataFilePath "C:\Temp\ShieldedVM.pdk" `
    -Owner $OwnerGuardian -Guardian $HGSGuardian `
    -VolumeIDQualifier (New-VolumeIDQualifier -VolumeSignatureCatalogFilePath "C:\Temp\vsc.cat" `
    -VersionRule Equals)
```

## ✅ L100 Checkpoint — You Should Now Know:

1. **Why Shielded VMs exist** — they protect VMs from untrusted fabric administrators and compromised hosts
2. **The three key components** — HGS (trust authority), Guarded Hosts (approved Hyper-V servers), and Shielding Data (provisioning secrets)
3. **How attestation works** — hosts prove their identity/health to HGS before receiving VM encryption keys
4. **Two attestation modes** — TPM (hardware-based, strongest) and Host Key (key-based, simpler); AD attestation is deprecated
5. **Shielded vs Encryption Supported** — Shielded blocks all admin access; Encryption Supported allows console access
6. **vTPM + BitLocker chain** — BitLocker keys stored in vTPM, vTPM state encrypted by host key, host key released only after attestation
