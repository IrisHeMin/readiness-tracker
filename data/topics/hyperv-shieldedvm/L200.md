# L200: Shielded VMs Troubleshooting
**⏱ Estimated: 10 minutes | 🎯 Goal: Diagnose and resolve Shielded VM and HGS issues**

---

## 📌 Troubleshooting Framework

When a shielded VM issue occurs, follow this diagnostic order:

```
┌──────────────────────────────────────────────────────────────┐
│               Shielded VM Troubleshooting Flow               │
│                                                              │
│  1. Is the host attested?                                    │
│     └─► Get-HgsClientConfiguration                          │
│         └─► AttestationStatus = Passed?                      │
│                                                              │
│  2. Can the host reach HGS?                                  │
│     └─► Test-NetConnection -ComputerName hgs -Port 443       │
│     └─► Resolve-DnsName hgs.contoso.com                      │
│                                                              │
│  3. Is HGS healthy?                                          │
│     └─► Get-HgsServer (on HGS node)                         │
│     └─► Get-HgsTrace -RunDiagnostics                        │
│                                                              │
│  4. Is the VM configuration correct?                         │
│     └─► Get-VMSecurity / Get-VMKeyProtector                  │
│                                                              │
│  5. Check event logs on both host and HGS                    │
│     └─► See Event Log Reference below                        │
└──────────────────────────────────────────────────────────────┘
```

## 📌 Common Issue #1: HGS Attestation Failures

### Symptom
`Get-HgsClientConfiguration` shows `AttestationStatus: InsecureHostGroup` or `AttestationStatus: Failed`.

### Root Causes and Fixes

| Cause | Diagnostic | Fix |
|-------|-----------|-----|
| **TPM not recognized** | `Get-HgsAttestationTpmHost` on HGS — host EK cert not listed | Register the host's TPM EK: `Add-HgsAttestationTpmHost -Path C:\Temp\host_ek.xml` |
| **CI policy mismatch** | Event 7000 in HGS Admin log — CI policy hash doesn't match | Re-capture CI policy from host, re-register on HGS |
| **TCG boot log issue** | Event 7001 — TCG log validation failed | Reboot host to regenerate TCG log; ensure firmware is up to date |
| **Secure Boot disabled** | Event 7002 — Secure Boot not enabled | Enable Secure Boot in host UEFI/BIOS |
| **Host clock skew** | Certificate validation failures | Sync time: `w32tm /resync /force` |

### Diagnostic Commands

```powershell
# Step 1: On the guarded host — check attestation status
Get-HgsClientConfiguration
# Look at:
#   AttestationStatus         — should be "Passed"
#   AttestationSubStatus      — gives detailed failure reason

# Step 2: On the guarded host — get the TPM EK certificate for registration
(Get-PlatformIdentifier).InnerXml | Out-File "C:\Temp\host_ek.xml" -Encoding utf8

# Step 3: On HGS — register the host's TPM
Add-HgsAttestationTpmHost -Name "HV-HOST-01" -Path "C:\Temp\host_ek.xml" -Force

# Step 4: On HGS — check what's registered
Get-HgsAttestationTpmHost | Format-Table Name, Identifier

# Step 5: On the guarded host — force re-attestation
Set-HgsClientConfiguration -AttestationServerUrl "https://hgs.contoso.com/Attestation" `
    -KeyProtectionServerUrl "https://hgs.contoso.com/KeyProtection"
```

## 📌 Common Issue #2: Code Integrity Policy Mismatch

### Symptom
TPM attestation fails even though TPM EK is registered. HGS event log shows policy hash mismatch.

### Understanding CI Policies

```
  Guarded Host                           HGS
  ────────────                       ─────────
                                     Stores HASH of expected
                                     CI policy for each host
       │
       │  Sends current CI policy
       │  hash during attestation
       │  ─────────────────────────►
       │                              Compares hash:
       │                              MATCH → Pass
       │                              MISMATCH → Fail
       │  ◄─────────────────────────
       │
```

### Fix: Recapture and Re-register CI Policy

```powershell
# Step 1: On guarded host — capture current CI policy
$CIPolicy = Get-CIPolicy -FilePath "C:\Windows\System32\CodeIntegrity\SIPolicy.p7b"
# Or capture a new one:
New-CIPolicy -FilePath "C:\Temp\HostCIPolicy.xml" -Level FilePublisher -Fallback Hash

# Step 2: Convert to binary
ConvertFrom-CIPolicy -XmlFilePath "C:\Temp\HostCIPolicy.xml" `
    -BinaryFilePath "C:\Temp\HostCIPolicy.p7b"

# Step 3: On HGS — remove old policy and register new one
Remove-HgsAttestationCIPolicy -Name "OldPolicy"
Add-HgsAttestationCIPolicy -Name "HostCIPolicy-v2" `
    -Path "C:\Temp\HostCIPolicy.p7b"

# Step 4: On guarded host — re-attest
Get-HgsClientConfiguration
```

> **⚠️ Common Gotcha**: Installing Windows updates or new drivers changes the CI policy hash. After patching, you MUST recapture the CI policy and re-register it on HGS.

## 📌 Common Issue #3: Shielded VM Won't Start

### Symptom
VM fails to start with errors like:
- `"The virtual machine 'VM01' could not be started because the hypervisor is not running"`
- `"The key protector for this virtual machine could not be unwrapped"`
- `"VM01 cannot start on this host because it is not a guarded host"`

### Diagnostic Flowchart

```
VM won't start
│
├── Is the host attested?
│   ├── YES → Check key protector (see below)
│   └── NO  → Fix attestation first (see Issue #1)
│
├── Key protector error?
│   ├── Check if HGS guardian is in the key protector
│   │   └─► Get-VMKeyProtector -VMName "VM01"
│   ├── Was HGS rebuilt or certificates rotated?
│   │   └─► Re-shield the VM with new guardian
│   └── Was VM imported from another fabric?
│       └─► Need to add new fabric's guardian to key protector
│
├── "Hypervisor not running"?
│   ├── Verify VBS is enabled
│   │   └─► bcdedit /enum | findstr hypervisor
│   └── Check UEFI Secure Boot is ON
│
└── Event logs
    └─► See Event Log Reference section
```

### Key Protector Troubleshooting

```powershell
# Check the VM's key protector
$kp = Get-VMKeyProtector -VMName "ShieldedVM01"

# Inspect who can run this VM
# The key protector contains the HGS guardian certificate
# If HGS was rebuilt, the old certificate won't match

# Export VM key protector for inspection
$kp | Out-File "C:\Temp\keyprotector.txt"

# If the VM was moved from another fabric, you need to grant
# the new HGS guardian access:
$newGuardian = New-HgsGuardian -Name "NewFabricHGS" `
    -SigningCertificate $signingCert -EncryptionCertificate $encCert
Grant-VMKeyProtectorAccess -VMName "ShieldedVM01" -Guardian $newGuardian
```

## 📌 Common Issue #4: Certificate Problems

### Symptom
- HGS services fail to start
- Attestation fails with SSL/TLS errors
- Key Protection Service returns certificate errors

### Certificate Checklist

| Check | Command | Expected |
|-------|---------|----------|
| HGS signing cert valid | `Get-HgsServer` — check `SigningCertificate` | Not expired, has private key |
| HGS encryption cert valid | `Get-HgsServer` — check `EncryptionCertificate` | Not expired, has private key |
| SSL cert trusted by hosts | `Test-HgsServer -HgsUrl https://hgs.contoso.com` | No TLS errors |
| Host trusts HGS root CA | `certutil -verify` on host | Chain builds successfully |

```powershell
# On HGS — check certificate health
Get-HgsServer | Select-Object -ExpandProperty SigningCertificate
Get-HgsServer | Select-Object -ExpandProperty EncryptionCertificate

# Check if certs are expiring
Get-ChildItem Cert:\LocalMachine\My | Where-Object {
    $_.Subject -like "*HGS*" -and $_.NotAfter -lt (Get-Date).AddDays(30)
} | Format-Table Subject, NotAfter, Thumbprint

# Rotate certificates if needed
Set-HgsServer -SigningCertificate $newSigningThumbprint `
    -EncryptionCertificate $newEncryptionThumbprint

# On guarded host — verify TLS connectivity
[System.Net.ServicePointManager]::SecurityProtocol = [System.Net.SecurityProtocolType]::Tls12
Invoke-WebRequest -Uri "https://hgs.contoso.com/Attestation/metadata" -UseBasicParsing
```

## 📌 Diagnostic Tools

### Get-HgsTrace — The Primary Diagnostic Tool

```powershell
# Run full HGS diagnostics (run on HGS server)
Get-HgsTrace -RunDiagnostics -Detailed

# Run diagnostics targeting a specific guarded host
Get-HgsTrace -RunDiagnostics -Target "HV-HOST-01" -Credential (Get-Credential)

# Common output to look for:
# [PASS] HGS Server is reachable
# [PASS] Attestation service is operational
# [FAIL] TPM EK certificate not registered  ← Action needed
# [FAIL] CI policy hash mismatch           ← Action needed
# [WARN] Certificate expiring in 25 days   ← Plan rotation
```

### Additional Diagnostic Commands

```powershell
# Test HGS health from a guarded host
Test-HgsClientConfiguration

# Check if HGS cluster is healthy
Get-ClusterNode | Format-Table Name, State
Get-ClusterGroup | Format-Table Name, State, OwnerNode

# Verify HGS DNS registration
Resolve-DnsName hgs.contoso.com
# Should return all HGS cluster node IPs

# Test network connectivity
Test-NetConnection -ComputerName hgs.contoso.com -Port 80
Test-NetConnection -ComputerName hgs.contoso.com -Port 443
```

## 📌 Event Log Reference

### On the Guarded Host

| Log Path | Event ID | Meaning |
|----------|----------|---------|
| `Microsoft-Windows-HostGuardianService-Client/Admin` | 1000 | Attestation succeeded |
| `Microsoft-Windows-HostGuardianService-Client/Admin` | 1001 | Attestation failed — check sub-status |
| `Microsoft-Windows-HostGuardianService-Client/Admin` | 1002 | Key protection request failed |
| `Microsoft-Windows-Hyper-V-Worker-Admin` | 3040 | VM failed to start — key protector issue |
| `Microsoft-Windows-Hyper-V-VMMS-Admin` | 12520 | VM configuration error |

### On the HGS Server

| Log Path | Event ID | Meaning |
|----------|----------|---------|
| `Microsoft-Windows-HostGuardianService-Attestation/Admin` | 7000 | CI policy mismatch |
| `Microsoft-Windows-HostGuardianService-Attestation/Admin` | 7001 | TCG log validation failed |
| `Microsoft-Windows-HostGuardianService-Attestation/Admin` | 7002 | Secure Boot validation failed |
| `Microsoft-Windows-HostGuardianService-Attestation/Admin` | 7003 | TPM EK not registered |
| `Microsoft-Windows-HostGuardianService-KeyProtection/Admin` | 8000 | Key release succeeded |
| `Microsoft-Windows-HostGuardianService-KeyProtection/Admin` | 8001 | Key release failed |

```powershell
# Query HGS attestation events (last 24 hours)
Get-WinEvent -LogName "Microsoft-Windows-HostGuardianService-Attestation/Admin" `
    -MaxEvents 50 | Where-Object {
    $_.TimeCreated -gt (Get-Date).AddHours(-24)
} | Format-Table TimeCreated, Id, Message -Wrap

# Query host-side attestation events
Get-WinEvent -LogName "Microsoft-Windows-HostGuardianService-Client/Admin" `
    -MaxEvents 20 | Format-Table TimeCreated, Id, Message -Wrap

# Look for VM start failures
Get-WinEvent -LogName "Microsoft-Windows-Hyper-V-Worker-Admin" `
    -MaxEvents 20 | Where-Object { $_.Id -eq 3040 } |
    Format-Table TimeCreated, Message -Wrap
```

## 📌 Migrating Shielded VMs Between Guarded Fabrics

### Scenario
You need to move a shielded VM from Fabric A (HGS-A) to Fabric B (HGS-B).

### Process

```
  Fabric A (source)              Fabric B (destination)
  ──────────────────            ──────────────────────
  HGS-A                         HGS-B
  │                              │
  │ 1. Export HGS-B guardian     │
  │    metadata                  │
  │ ◄────────────────────────── │
  │                              │
  │ 2. Add HGS-B as guardian    │
  │    to VM's key protector    │
  │                              │
  │ 3. Export/migrate VM ───────►│
  │                              │
  │                              │ 4. VM starts using
  │                              │    HGS-B attestation
```

```powershell
# Step 1: On Fabric B HGS — export guardian metadata
Export-HgsGuardian -Path "C:\Temp\FabricB_Guardian.xml" -InputObject (Get-HgsGuardian)

# Step 2: On Fabric A host — import Fabric B guardian and grant access
$fabricBGuardian = Import-HgsGuardian -Path "C:\Temp\FabricB_Guardian.xml" -Name "FabricB-HGS"
Grant-VMKeyProtectorAccess -VMName "ShieldedVM01" -Guardian $fabricBGuardian

# Step 3: Export the VM
Export-VM -Name "ShieldedVM01" -Path "D:\VMExport"

# Step 4: On Fabric B host — import and start the VM
Import-VM -Path "D:\VMExport\ShieldedVM01\Virtual Machines\*.vmcx"
Start-VM -Name "ShieldedVM01"
```

> **⚠️ Important**: After migration, you can optionally remove Fabric A's guardian from the key protector to prevent the VM from being started back on Fabric A:
> ```powershell
> Revoke-VMKeyProtectorAccess -VMName "ShieldedVM01" -Guardian $fabricAGuardian
> ```
