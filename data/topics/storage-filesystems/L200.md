## 1. NTFS Corruption & chkdsk

### Recognizing Corruption

Common symptoms of NTFS corruption:
- Error: **"The file or directory is corrupted and unreadable"**
- Error: **"The volume is dirty"** at boot
- Files or folders suddenly disappearing or inaccessible
- Event Log: `Ntfs` source warnings or errors (Event IDs 55, 98, 137)

### Checking the Dirty Bit

The dirty bit indicates the volume was not cleanly dismounted:

```powershell
# Check if a volume is marked dirty
fsutil dirty query C:

# If dirty, Windows schedules chkdsk at next reboot
```

### chkdsk Phases

| Phase | Name                  | What It Does                                           |
|-------|-----------------------|--------------------------------------------------------|
| 1     | File verification     | Checks each MFT record for internal consistency        |
| 2     | Index verification    | Verifies directory indexes match MFT entries           |
| 3     | Security descriptors  | Validates $Secure file and SD references               |
| 4     | USN Journal verify    | Verifies $UsnJrnl integrity (only with /R or /B)       |
| 5     | Bad cluster scan      | Surface scan for bad sectors (only with /R)             |

### Modern chkdsk Workflow (Windows 8+)

```
┌─────────────────────────────────────────────────┐
│           chkdsk Decision Workflow               │
├─────────────────────────────────────────────────┤
│                                                  │
│  1. Online scan (no downtime):                   │
│     chkdsk C: /scan                              │
│     → Detects issues while volume is mounted     │
│     → Logs fixes needed to $corrupt system file  │
│                                                  │
│  2. Apply fixes (brief offline):                 │
│     chkdsk C: /spotfix                           │
│     → Takes volume offline momentarily           │
│     → Applies queued fixes from /scan            │
│     → Much faster than legacy full chkdsk        │
│                                                  │
│  3. Legacy full check (longest downtime):        │
│     chkdsk C: /R                                 │
│     → Full offline scan + surface test           │
│     → Use only when /scan + /spotfix not enough  │
│                                                  │
└─────────────────────────────────────────────────┘
```

```powershell
# Recommended: online scan first
chkdsk C: /scan

# Apply queued fixes (requires reboot for boot volume)
chkdsk C: /spotfix

# Full repair including bad sector recovery (last resort)
chkdsk D: /R

# For boot volume, schedule at next reboot
chkdsk C: /R /X
```

## 2. NTFS Permission Troubleshooting

### Effective Access Calculation

Effective permissions are determined by combining:
1. **Explicit permissions** on the object
2. **Inherited permissions** from parent folders
3. **Group membership** – all group SIDs in the user's token
4. **Deny overrides Allow** – an explicit Deny at any level blocks access

```
User Token SIDs ──┐
                   ├──► Evaluate DACL ──► Effective Access
Object DACL ───────┘
   Rule: Deny ACEs are evaluated FIRST
   Rule: Inherited Deny < Explicit Allow (but explicit Deny always wins)
```

### Common "Access Denied" Scenarios

| Scenario                        | Diagnostic Command                                 | Fix                                |
|---------------------------------|----------------------------------------------------|------------------------------------|
| Ownership mismatch              | `icacls "path" /Q`                                 | `takeown /F "path" /R`            |
| Broken inheritance              | `icacls "path"` – look for `(I)` markers           | `icacls "path" /reset /T`         |
| Deny overrides Allow            | `icacls "path"` – look for `(DENY)` entries        | Remove explicit Deny ACE          |
| Missing NTFS permissions        | `Get-Acl "path" \| Format-List`                    | `icacls "path" /grant User:F`     |
| Share + NTFS combined           | Check both share perms and NTFS; most restrictive wins | Align both permission sets     |

### Key Commands

```powershell
# View permissions
icacls "C:\Data\Reports"

# View detailed ACL with inheritance info
Get-Acl "C:\Data\Reports" | Format-List

# Grant full control
icacls "C:\Data\Reports" /grant "Domain\User:(OI)(CI)F"

# Take ownership recursively
takeown /F "C:\Data\Reports" /R /D Y

# Reset to inherited permissions
icacls "C:\Data\Reports" /reset /T /C

# Check effective access for a specific user
Get-Acl "C:\Data\Reports" | Select-Object -ExpandProperty Access |
    Where-Object { $_.IdentityReference -like "*UserName*" }
```

## 3. ReFS Integrity and Repair

### Integrity Streams

ReFS can compute and store checksums alongside data blocks:

```powershell
# Check if integrity streams are enabled on a volume
Get-Item "R:\Data" | Get-FileIntegrity

# Enable integrity streams on a directory
Set-FileIntegrity -FileName "R:\Data" -Enable $true
```

When a read detects a checksum mismatch:
- **Mirrored volume** → ReFS automatically repairs from the good copy
- **Non-mirrored volume** → Read fails with error; data must be restored from backup

### ReFS Repair Commands

```powershell
# Online scan for integrity issues
Repair-Volume -DriveLetter R -Scan

# Apply spot fixes (brief offline)
Repair-Volume -DriveLetter R -SpotFix

# ReFS does NOT support chkdsk /R equivalent
# There is no surface scan or bad-sector relocation for ReFS
```

> **⚠ Key limitation:** ReFS relies on Storage Spaces mirroring for data redundancy. On a simple (non-mirrored) volume, a detected corruption cannot be auto-repaired.

## 4. USB Block Policy

### Device Installation Restrictions

**GPO Path:** `Computer Configuration > Administrative Templates > System > Device Installation > Device Installation Restrictions`

| Policy Setting                                      | Effect                                         |
|-----------------------------------------------------|-------------------------------------------------|
| Prevent installation of devices not described by other policy rules | Block all unknown devices          |
| Prevent installation of devices matching these device IDs           | Block specific device hardware IDs |
| Prevent installation of devices using these device setup classes     | Block by class GUID               |
| Allow installation of devices matching these device IDs             | Whitelist specific devices         |

### Removable Storage Access

**GPO Path:** `Computer Configuration > Administrative Templates > System > Removable Storage Access`

| Policy Setting                           | Effect                                  |
|------------------------------------------|-----------------------------------------|
| Removable Disks: Deny read access        | Block reading from USB drives           |
| Removable Disks: Deny write access       | Block writing to USB drives             |
| All Removable Storage: Deny all access   | Complete block on all removable media    |
| CD and DVD: Deny read/write access       | Block optical media                     |

### Common Device Class GUIDs

```
{36fc9e60-c465-11cf-8056-444553540000}  → USB controllers
{4d36e967-e325-11ce-bfc1-08002be10318}  → Disk drives
{eec5ad98-8080-425f-922a-dabf3de3f69a}  → WPD (portable devices)
```

### Verifying USB Policy

```powershell
# Check applied device installation policies via registry
Get-ItemProperty "HKLM:\SOFTWARE\Policies\Microsoft\Windows\DeviceInstall\Restrictions" -ErrorAction SilentlyContinue

# List currently connected USB devices
Get-PnpDevice -Class USB | Where-Object Status -eq 'OK'

# Check removable storage policy
Get-ItemProperty "HKLM:\SOFTWARE\Policies\Microsoft\Windows\RemovableStorageDevices" -ErrorAction SilentlyContinue

# Use gpresult to confirm policy application
gpresult /H gpreport.html
```

## 5. Common Error Scenarios

### "The parameter is incorrect"

This error can mean many things—narrow it down:

| Possible Cause                 | Diagnostic                                        |
|--------------------------------|---------------------------------------------------|
| Filesystem corruption          | `chkdsk X: /scan`                                 |
| Incompatible cluster size      | `fsutil fsinfo ntfsinfo X:` – check cluster size  |
| Path too long (> 260 chars)    | Check path length; enable LongPathsEnabled GPO    |
| Mismatched NTFS version        | Compare NTFS version between source and target     |

### "The file or directory is corrupted and unreadable"

```powershell
# Step 1: Identify the affected volume
# Step 2: Run online scan
chkdsk X: /scan

# Step 3: If issues found, apply spotfix
chkdsk X: /spotfix

# Step 4: Check Event Log for Ntfs errors
Get-WinEvent -FilterHashtable @{LogName='System'; ProviderName='Ntfs'} -MaxEvents 20
```

### "Access is denied" – Systematic Approach

```
1. Verify identity    → whoami /all
2. Check ownership    → icacls "path" /Q
3. Check NTFS ACL     → icacls "path"
4. Check share perms  → Get-SmbShareAccess -Name "ShareName"
5. Check GPO denies   → gpresult /R
6. Check file attrs   → attrib "path" (read-only?)
7. Check encryption   → cipher /C "path"
```
