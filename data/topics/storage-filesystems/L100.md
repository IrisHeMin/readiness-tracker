## 1. NTFS Architecture

NTFS organizes all data—including its own metadata—as files stored in the Master File Table (MFT).

```
┌─────────────────────────────────────────────────────────┐
│                    NTFS Volume Layout                    │
├──────────┬──────────────────────────────────────────────┤
│ Boot     │  VBR (Volume Boot Record) – bootstrap code   │
│ Sector   │  Points to $MFT location on disk             │
├──────────┼──────────────────────────────────────────────┤
│          │  $MFT ─────────── Master File Table (rec 0)  │
│          │  $MFTMirr ─────── First 4 MFT records (rec 1)│
│  MFT     │  $LogFile ─────── Transaction journal (rec 2)│
│  Zone    │  $Volume ──────── Volume metadata    (rec 3) │
│          │  $AttrDef ─────── Attribute definitions(rec 4)│
│          │  . (root dir) ─── Root directory      (rec 5)│
│          │  $Bitmap ──────── Cluster allocation  (rec 6)│
│          │  $Boot ────────── Boot sector copy    (rec 7)│
│          │  $BadClus ─────── Bad cluster list    (rec 8)│
│          │  $Secure ──────── Security descriptors(rec 9)│
│          │  $UsnJrnl ─────── Change journal     (rec 11)│
├──────────┼──────────────────────────────────────────────┤
│ Data     │  User files and directories                  │
│ Area     │  (each file = one or more MFT records)       │
└──────────┴──────────────────────────────────────────────┘
```

### Journaling

| Journal     | Purpose                                | Scope                        |
|-------------|----------------------------------------|------------------------------|
| **$LogFile**  | Transaction log for crash recovery   | Metadata changes only        |
| **$UsnJrnl**  | Update Sequence Number Journal       | Tracks all file/dir changes  |

- **$LogFile** – Records metadata transactions (create, rename, delete, permission changes). On mount after unclean shutdown, NTFS replays or undoes incomplete transactions to restore consistency. This is why NTFS is called a *journaling* filesystem.
- **$UsnJrnl** – Maintains a persistent change log with a USN (64-bit counter) for every change. Used by Search Indexer, File Server Resource Manager, and backup applications to detect what changed since last scan.

## 2. NTFS Features

| Feature              | Details                                                                 |
|----------------------|-------------------------------------------------------------------------|
| **ACL Permissions**  | DACL (Discretionary) controls access; SACL (System) controls auditing   |
| **Compression**      | Per-file or per-folder; only effective on cluster size ≤ 4 KB           |
| **Encryption (EFS)** | Encrypting File System – per-file encryption tied to user certificate    |
| **Hard Links**       | Multiple directory entries pointing to same MFT record (same volume)    |
| **Junction Points**  | Directory-level reparse points; resolved by the kernel (local paths)    |
| **Symbolic Links**   | File or directory; can target network paths; resolved in user mode       |

### Junction Points vs Symbolic Links

| Aspect             | Junction Point                  | Symbolic Link                     |
|--------------------|---------------------------------|-----------------------------------|
| Target type        | Directories only                | Files or directories              |
| Target scope       | Local absolute paths only       | Local or remote (UNC) paths       |
| Resolution         | Kernel mode (transparent)       | User mode                         |
| Privilege required | No                              | SeCreateSymbolicLinkPrivilege     |
| Create command     | `mklink /J Link Target`         | `mklink /D Link Target` (dir)     |

## 3. ReFS Architecture

ReFS (Resilient File System) was introduced in Windows Server 2012 with a fundamentally different design:

- **B+ Trees** – All metadata is organized in B+ tree structures (no flat MFT table).
- **Integrity Streams** – Optional checksums on data and metadata to detect silent corruption (bit rot).
- **Copy-on-Write (Allocate-on-Write)** – Metadata updates are written to new locations, never modifying existing data in place. The pointer is updated atomically, ensuring the on-disk state is always consistent.
- **No MFT** – Eliminates the single-point-of-failure that MFT represents.
- **No $LogFile journal** – Consistency is guaranteed by the allocate-on-write model; no replay needed.

```
┌─────────────────────────────────────────┐
│           ReFS Volume Layout            │
├─────────────────────────────────────────┤
│  Superblock (checkpoint area)           │
│    ├── Object Table (B+ tree root)      │
│    ├── Directory B+ trees               │
│    ├── File extents B+ trees            │
│    └── Allocator tables                 │
├─────────────────────────────────────────┤
│  Data area (copy-on-write updates)      │
│  Integrity-stream-protected regions     │
└─────────────────────────────────────────┘
```

## 4. NTFS vs ReFS Comparison

| Feature                  | NTFS                        | ReFS                              |
|--------------------------|-----------------------------|-----------------------------------|
| Max volume size          | 256 TB (practical)          | 35 PB (practical)                 |
| Max file size            | 256 TB (practical)          | 35 PB (practical)                 |
| Journaling ($LogFile)    | ✅ Yes                      | ❌ No (uses allocate-on-write)    |
| Self-healing             | ❌ No                       | ✅ Yes (online repair)            |
| Integrity streams        | ❌ No                       | ✅ Yes (checksums for bit rot)    |
| Compression              | ✅ Yes (≤ 4KB clusters)     | ❌ No                             |
| EFS encryption           | ✅ Yes                      | ❌ No                             |
| Short names (8.3)        | ✅ Yes                      | ❌ No                             |
| Disk quotas              | ✅ Yes                      | ❌ No                             |
| Data deduplication       | ✅ Yes                      | ✅ Yes (Server 2019+)             |
| Block cloning            | ❌ No                       | ✅ Yes (instant file copies)      |
| Boot volume support      | ✅ Yes                      | ❌ No                             |
| Supported OS             | All Windows versions        | Server 2012+ / Win 10 (limited)   |

## 5. When to Use Which

**Choose ReFS when:**
- Running **Hyper-V** workloads – block cloning enables instant VHD/VHDX creation and merge
- Deploying **Storage Spaces Direct (S2D)** – designed and optimized for ReFS
- Building **backup targets** – integrity streams protect against silent corruption
- Managing **very large volumes** (> 64 TB)

**Choose NTFS when:**
- Volume is a **boot/system volume** (ReFS cannot be a boot volume)
- **General-purpose** file server or workstation use
- Need **compression, EFS encryption, or disk quotas**
- Using **removable media** (USB drives, external disks)
- Application compatibility requires NTFS-specific features

## 6. Essential Commands

```powershell
# NTFS volume information
fsutil fsinfo ntfsinfo C:

# Free space on a volume
fsutil volume diskfree C:

# List volumes with filesystem type
Get-Volume | Format-Table DriveLetter, FileSystemType, SizeRemaining, Size

# Format a volume
Format-Volume -DriveLetter E -FileSystem NTFS -AllocationUnitSize 4096
Format-Volume -DriveLetter F -FileSystem ReFS

# Check filesystem type
Get-Volume -DriveLetter C | Select-Object FileSystemType
```
