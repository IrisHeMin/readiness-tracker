## 1. NTFS Internals

### MFT Record Structure

Each MFT record is **1 KB** by default (configurable to 2 KB or 4 KB at format time). Every file and directory on an NTFS volume has at least one MFT record.

```
┌──────────────────────────────────────────────────────────┐
│                  MFT Record (1024 bytes)                  │
├──────────┬───────────────────────────────────────────────┤
│ Header   │ Signature "FILE", sequence number, link count │
│ (42 B)   │ Offset to first attribute, flags (in-use/dir) │
├──────────┼───────────────────────────────────────────────┤
│ Attr 1   │ $STANDARD_INFORMATION (timestamps, flags)     │
├──────────┼───────────────────────────────────────────────┤
│ Attr 2   │ $FILE_NAME (name, parent ref, timestamps)     │
├──────────┼───────────────────────────────────────────────┤
│ Attr 3   │ $DATA (file content or data runs)             │
├──────────┼───────────────────────────────────────────────┤
│ Attr N   │ Additional attributes...                      │
├──────────┼───────────────────────────────────────────────┤
│ End      │ 0xFFFFFFFF marker                             │
└──────────┴───────────────────────────────────────────────┘
```

### NTFS Attribute Types

| Attribute Type          | Type ID | Purpose                                              |
|-------------------------|---------|------------------------------------------------------|
| $STANDARD_INFORMATION   | 0x10    | Timestamps (MACE), file attributes, owner ID, security ID |
| $ATTRIBUTE_LIST         | 0x20    | Lists attributes when they span multiple MFT records  |
| $FILE_NAME              | 0x30    | Filename (Unicode), parent directory MFT ref, timestamps |
| $OBJECT_ID              | 0x40    | Unique GUID for distributed link tracking             |
| $SECURITY_DESCRIPTOR    | 0x50    | ACLs (rarely used directly; usually indexed in $Secure) |
| $VOLUME_NAME            | 0x60    | Volume label (only on $Volume metadata file)          |
| $VOLUME_INFORMATION     | 0x70    | NTFS version, flags (only on $Volume)                 |
| $DATA                   | 0x80    | File content (default unnamed stream)                 |
| $INDEX_ROOT             | 0x90    | Root node of B-tree index (always resident)           |
| $INDEX_ALLOCATION       | 0xA0    | Non-resident index nodes (directory B-tree entries)   |
| $BITMAP                 | 0xB0    | Allocation bitmap for index or MFT                    |
| $REPARSE_POINT          | 0xC0    | Symlinks, junction points, dedup, HSM tags            |
| $EA_INFORMATION         | 0xD0    | Extended attribute metadata (OS/2 compatibility)      |
| $EA                     | 0xE0    | Extended attribute data                               |

### Resident vs Non-Resident Attributes

- **Resident** – Attribute data fits inside the MFT record itself (< ~700 bytes for a 1KB record). Small files are stored entirely in the MFT.
- **Non-Resident** – Attribute data is stored in clusters outside the MFT. The attribute header contains **data runs** (also called "cluster runs") that map logical offsets to physical cluster locations.

```
Data Run encoding example:
  Header byte: 0x31 → 1 byte for length, 3 bytes for offset
  Run: starting at cluster 0x002A00, length 0x40 clusters

  Multiple runs form an "extent list" for fragmented files.
```

## 2. USN Journal Analysis

The Update Sequence Number Journal (`$Extend\$UsnJrnl`) records a log entry for every change made to files and directories.

### Reading the USN Journal

```powershell
# Query USN journal status
fsutil usn queryjournal C:

# Read recent entries (raw output)
fsutil usn readjournal C: csv

# Enumerate specific file changes by reason
fsutil usn readjournal C: csv | findstr /i "close"
```

### USN Reason Codes

| Reason Flag              | Hex Value    | Meaning                              |
|--------------------------|-------------|--------------------------------------|
| DATA_OVERWRITE           | 0x00000001  | File data was overwritten            |
| DATA_EXTEND              | 0x00000002  | File data was appended               |
| DATA_TRUNCATION          | 0x00000004  | File data was truncated              |
| NAMED_DATA_OVERWRITE     | 0x00000010  | Named stream was overwritten         |
| FILE_CREATE              | 0x00000100  | File or directory was created        |
| FILE_DELETE              | 0x00000200  | File or directory was deleted        |
| RENAME_OLD_NAME          | 0x00001000  | Old name before rename               |
| RENAME_NEW_NAME          | 0x00002000  | New name after rename                |
| SECURITY_CHANGE          | 0x00000800  | ACL was modified                     |
| CLOSE                    | 0x80000000  | File handle was closed               |

### Forensic Use Cases

- **Change detection**: Identify all files modified in a time window without full-volume scan
- **Deleted file discovery**: `FILE_DELETE` + `CLOSE` entries reveal deleted file names and parent directories
- **Ransomware detection**: Burst of `RENAME_NEW_NAME` + `DATA_OVERWRITE` in short interval
- **Audit trail**: Track permission changes via `SECURITY_CHANGE` reason code

```powershell
# PowerShell: Query USN journal entries for a specific file
$volume = Get-Volume -DriveLetter C
fsutil usn readdata "C:\important-file.txt"
```

## 3. ReFS Integrity Streams & Scrubbing

### How Integrity Streams Work

When integrity streams are enabled, ReFS computes a **checksum** for each data block and stores it in the metadata B+ tree (separate from the data itself).

```
Write path:
  App data ──► Compute checksum ──► Write data to new location
                                  ──► Update metadata pointer + checksum

Read path:
  Read data ──► Recompute checksum ──► Compare with stored checksum
                                         ├── Match → Return data
                                         └── Mismatch → Corruption detected
                                              ├── Mirror available → Auto-repair
                                              └── No mirror → Return error
```

### Background Scrubbing

The **integrity scrubber** is a background process that proactively reads all data blocks and validates checksums:

```powershell
# Check scrubber status
Get-StoragePool | Get-StorageHealthReport

# Manually trigger a scrub
Start-StorageDiagnosticLog  # or use Repair-Volume -Scan

# View integrity stream status for a file
Get-FileIntegrity -FileName "R:\Data\backup.vhdx"
```

## 4. ReFS Block Cloning

### How It Works

Block cloning allows ReFS to copy a range of file bytes by **duplicating metadata pointers** instead of physically copying data. The result is an instant, space-efficient copy.

```
Before clone:
  File A ──► [Block 1][Block 2][Block 3]    (physical data)

After clone (File B = copy of File A):
  File A ──► [Block 1][Block 2][Block 3]    (shared physical data)
  File B ──►  ↑         ↑        ↑          (metadata points to same blocks)

On write to File B Block 2:
  File A ──► [Block 1][Block 2][Block 3]
  File B ──► [Block 1][Block 2'][Block 3]   (copy-on-write for modified block)
```

### Impact on Hyper-V

| Operation                    | NTFS              | ReFS with Block Cloning  |
|------------------------------|--------------------|--------------------------|
| Create fixed VHDX (50 GB)   | Minutes (write zeros) | Seconds (metadata only) |
| Merge checkpoint             | Minutes (copy data)   | Seconds (pointer update)|
| Clone VM                     | Minutes               | Seconds                 |

### API Details

```
Win32 API: FSCTL_DUPLICATE_EXTENTS_TO_FILE
  Input: Source file handle, destination file handle, byte offset, byte count
  Requirements:
    - Both files must be on the same ReFS volume
    - Ranges must be cluster-aligned
    - Destination file must be preallocated
```

## 5. Filesystem Filter Drivers & Minifilters

### Filter Manager Architecture

The Windows Filter Manager (FltMgr.sys) provides a framework for filesystem minifilter drivers. Each minifilter is assigned an **altitude** (a numeric value) that determines its position in the I/O stack.

```
Application I/O Request
        │
        ▼
┌──────────────────────┐
│    I/O Manager        │
├──────────────────────┤
│  Filter Manager       │
│   ├─ AV Minifilter   │  altitude 320000-329999
│   ├─ Encryption       │  altitude 140000-149999
│   ├─ Dedup            │  altitude 130000-139999
│   ├─ HSM/Tiering      │  altitude 400000-409999
│   └─ Monitor/Audit    │  altitude 360000-369999
├──────────────────────┤
│  Filesystem (NTFS)    │
├──────────────────────┤
│  Volume / Disk Stack  │
└──────────────────────┘
```

### fltmc Commands

```powershell
# List all loaded minifilter drivers with altitudes
fltmc filters

# List minifilter instances attached to volumes
fltmc instances

# List volumes known to Filter Manager
fltmc volumes

# Detach a problematic minifilter from a volume (use with caution)
fltmc detach FilterName C:

# Load / unload a minifilter
fltmc load FilterName
fltmc unload FilterName
```

### Common Minifilter Issues

| Symptom                             | Likely Cause                              | Investigation                         |
|-------------------------------------|-------------------------------------------|---------------------------------------|
| Slow file I/O across all apps       | AV minifilter scanning every I/O          | `fltmc instances` → identify AV filter |
| File copy failures or corruption    | Buggy minifilter modifying data in-flight | Boot to Safe Mode to bypass            |
| BSOD with FLTMGR stack frames      | Minifilter driver crash                   | Check dump with `!fltkd.filters`       |
| Application-specific hangs          | Minifilter deadlock on specific file type  | Process Monitor filter on file ops    |

## 6. Advanced Recovery

### MFT Repair Techniques

When the MFT itself is damaged, recovery options include:

```powershell
# Check MFT zone and fragmentation
defrag C: /A /V | Select-String "MFT"

# If $MFT is damaged, use $MFTMirr (first 4 records are mirrored)
# chkdsk automatically attempts to restore from $MFTMirr

# Analyze MFT record for a specific file
fsutil file queryextents C:\important-file.txt

# Get detailed file layout information
fsutil volume filelayout C: 0x0000000000000000 0xFFFFFFFFFFFFFFFF
```

### Recovering Orphaned Files

When directory entries are lost but MFT records remain intact, chkdsk moves files to `found.000`, `found.001`, etc. in the volume root:

```powershell
# After chkdsk, check for recovered files
Get-ChildItem "C:\found.*" -Directory

# Files are renamed to fileNNNN.chk
# Identify file types by examining headers
# Use file-type-specific tools to rename/recover
```

### Low-Level fsutil Operations

```powershell
# Query file extents (physical cluster locations)
fsutil file queryextents "C:\file.dat"

# Query file ID (MFT record number)
fsutil file queryfileid "C:\file.dat"

# Find file by ID (reverse lookup)
fsutil file queryfilenamebyid C: 0x0000000000001234

# Query allocated ranges (sparse file info)
fsutil file queryallocranges offset=0 length=1073741824 "C:\sparse.dat"
```
