## 📌 Hyper-V Architecture Internals

### Hypercall Interface

The hypervisor exposes a **hypercall interface** — a set of API calls that partitions use to request privileged operations (memory mapping, interrupt delivery, inter-partition communication).

```
┌────────────────────────────────────────────────────────────┐
│  Guest OS (Child Partition)                                 │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  Application Layer                                    │  │
│  ├──────────────────────────────────────────────────────┤  │
│  │  Kernel + Enlightened Drivers                         │  │
│  │    ├── Hypercall (VMCALL/VMMCALL instruction)         │  │
│  │    ├── Synthetic Interrupt Controller (SynIC)         │  │
│  │    └── Reference TSC (fast time reads)                │  │
│  └──────────┬───────────────────────────────────────────┘  │
│             │ Hypercall / VM Exit                           │
├─────────────┼──────────────────────────────────────────────┤
│  Hypervisor │                                               │
│  ┌──────────▼───────────────────────────────────────────┐  │
│  │  Hypercall Dispatch Table                             │  │
│  │  ├── Memory Management (GPA → SPA translation)        │  │
│  │  ├── Virtual Processor Management                     │  │
│  │  ├── Inter-Partition Messaging                        │  │
│  │  └── Timer / Interrupt Delivery                       │  │
│  └──────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘

GPA = Guest Physical Address
SPA = System Physical Address (actual hardware)
```

### Enlightenments

**Enlightenments** are hypervisor-aware optimizations built into the guest OS kernel. An enlightened guest avoids expensive VM exits by using hypercalls directly.

| Enlightenment | Purpose | Impact |
|--------------|---------|--------|
| **Synthetic Interrupt Controller (SynIC)** | Delivers interrupts without VM exits | Major reduction in interrupt latency |
| **Reference TSC** | Fast timestamp reads via shared memory page | Avoids VM exit on every `RDTSC` |
| **Hypercall page** | Mapped page for issuing hypercalls | Eliminates privileged instruction traps |
| **VP Assist page** | Per-VP shared page for lazy TLB flush | Reduces TLB invalidation overhead |
| **Relaxed timing** | Tolerates timer inaccuracies | Helps non-real-time guests run faster |
| **Spin lock awareness** | Hypervisor detects guest spinlock contention | Prevents wasted CPU cycles on lock spinning |
| **APIC virtualization** | Hardware-accelerated APIC emulation | Near-native interrupt performance |

```powershell
# Check which enlightenments a VM has enabled (from inside guest)
Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Virtual Machine\Auto"

# From host — check CPUID features exposed to VM
Get-VMProcessor -VMName "DC01" | Select-Object *
```

## 📌 VMBus & Synthetic vs Emulated Devices

### Device Model

```
┌─────────────────────────────────────────────────┐
│              Child Partition (VM)                 │
│  ┌───────────────┐    ┌──────────────────────┐  │
│  │  Emulated      │    │  Synthetic (VMBus)    │  │
│  │  Devices       │    │  Devices              │  │
│  │  ─────────     │    │  ─────────            │  │
│  │  Legacy NIC    │    │  Network Adapter (VSC)│  │
│  │  IDE Controller│    │  SCSI Controller (VSC)│  │
│  │  S3 Video      │    │  Video (RemoteFX/VSC) │  │
│  │  i8042 KBD     │    │  Keyboard/Mouse       │  │
│  └───────┬───────┘    └──────────┬───────────┘  │
│          │ Port I/O traps         │ VMBus ring    │
│          │ (SLOW — VM exits)      │ (FAST — shared│
│          │                        │  memory)      │
├──────────┼────────────────────────┼──────────────┤
│          ▼                        ▼               │
│  ┌───────────────┐    ┌──────────────────────┐   │
│  │  vmwp.exe      │    │  VSP (in parent)     │   │
│  │  (emulation)   │    │  (direct HW access)  │   │
│  └───────────────┘    └──────────────────────┘   │
└──────────────────────────────────────────────────┘
```

| Aspect | Emulated Devices | Synthetic Devices |
|--------|-----------------|-------------------|
| **Mechanism** | Traps I/O port access → VM exit → `vmwp.exe` emulates | VMBus ring buffers in shared memory |
| **Performance** | Slow (every I/O = VM exit) | Near-native (bulk data via DMA-like transfer) |
| **Guest driver** | Standard in-box driver (e.g., Intel E1000) | Hyper-V Integration Services driver (VSC) |
| **When used** | Boot phase (Gen 1), PXE, legacy OS | Normal operation with IC installed |
| **Gen 2 VMs** | Mostly removed | All devices are synthetic |

> **Key insight:** Gen 2 VMs eliminated most emulated devices. If you see poor I/O in a Gen 1 VM, verify that Integration Services are installed so synthetic drivers are active.

## 📌 Nested Virtualization

Nested virtualization allows running a hypervisor inside a Hyper-V VM — essential for training labs, CI/CD with containers, and testing.

```
┌─────────────────────────────────────────┐
│  Physical Host (L0 Hypervisor)           │
│  ├── Parent Partition                    │
│  └── Child VM (L1 Hypervisor)            │
│      ├── L1 Parent Partition             │
│      └── L2 Nested VM                   │
│          └── Guest OS                    │
└─────────────────────────────────────────┘
```

```powershell
# Enable nested virtualization for a VM (VM must be OFF)
Set-VMProcessor -VMName "LabHost" -ExposeVirtualizationExtensions $true

# Verify it's enabled
Get-VMProcessor -VMName "LabHost" | Select-Object ExposeVirtualizationExtensions

# Also enable MAC address spoofing (needed for nested VM networking)
Set-VMNetworkAdapter -VMName "LabHost" -MacAddressSpoofing On
```

**Requirements:**
- Host must run Windows Server 2016+ or Windows 10 1607+
- VM must be Gen 2 with at least 4 GB RAM
- Intel VT-x CPUs (AMD support added in Windows Server 2022 / Win 11)
- VM Configuration Version 8.0+

**Limitations:**
- No SR-IOV, Dynamic Memory, or live migration of L1 VMs with nested VMs running
- L2 VM performance is reduced (~80-90% of L1)

## 📌 Discrete Device Assignment (DDA) & GPU-P

### DDA (Discrete Device Assignment)

DDA passes an entire PCIe device (GPU, NVMe, NIC) directly to a VM with near-native performance.

```
Without DDA:                    With DDA:
┌──────────┐                   ┌──────────┐
│    VM    │                   │    VM    │
│ Emulated │                   │  Direct  │
│  Device  │                   │  Access  │
└────┬─────┘                   └────┬─────┘
     │ VMBus                        │ PCIe Passthrough
┌────┴─────┐                   ┌────┴─────┐
│   Host   │                   │   IOMMU  │
│  Driver  │                   │  (VT-d)  │
└────┬─────┘                   └────┴─────┘
     │                              │
┌────┴─────┐                   ┌────┴─────┐
│  Device  │                   │  Device  │
└──────────┘                   └──────────┘
```

```powershell
# List devices eligible for DDA
Get-PnpDevice -PresentOnly | Where-Object {
    (Get-PnpDeviceProperty -KeyName DEVPKEY_Device_LocationPaths -InstanceId $_.InstanceId).Data -match "PCIROOT"
}

# Dismount device from host
Dismount-VMHostAssignableDevice -LocationPath "PCIROOT(0)#PCI(0200)#PCI(0000)" -Force

# Assign device to VM
Add-VMAssignableDevice -VMName "GPU-VM" -LocationPath "PCIROOT(0)#PCI(0200)#PCI(0000)"

# Requires: IOMMU (Intel VT-d / AMD-Vi), SLAT, Gen 2 VM
```

### GPU-P (GPU Partitioning)

GPU-P shares a single GPU across multiple VMs (used in Windows Sandbox, Azure NVv4).

```powershell
# Check GPU partitioning capability
Get-VMHostPartitionableGpu

# Assign GPU partition to VM
Add-VMGpuPartitionAdapter -VMName "DesktopVM"
Set-VMGpuPartitionAdapter -VMName "DesktopVM" `
    -MinPartitionVRAM 80000000 `
    -MaxPartitionVRAM 100000000 `
    -OptimalPartitionVRAM 100000000

# Note: Requires copying host GPU driver files into the VM
```

## 📌 Hyper-V Network Virtualization (HNV)

HNV decouples VM networks from physical networks using NVGRE or VXLAN encapsulation.

```
┌────────────────────────────────────────────────────┐
│  Customer/Tenant Network (Virtual)                  │
│  VM-A: 10.0.1.10      VM-B: 10.0.1.20             │
│  (Host 1)              (Host 2)                     │
└───────┬────────────────────────┬───────────────────┘
        │   NVGRE/VXLAN Tunnel   │
┌───────┴────────────────────────┴───────────────────┐
│  Physical Provider Network                          │
│  Host1: 192.168.1.100    Host2: 192.168.1.200      │
│  (Underlay / Fabric)                                │
└────────────────────────────────────────────────────┘
```

| Concept | Description |
|---------|-------------|
| **CA (Customer Address)** | The IP address visible inside the VM (e.g., 10.0.1.10) |
| **PA (Provider Address)** | The physical host IP used for encapsulation (e.g., 192.168.1.100) |
| **VSID** | Virtual Subnet ID — isolates tenants (like VLAN but scalable to 16M) |
| **HNV Gateway** | Routes traffic between virtual networks and physical/external networks |

```powershell
# View network virtualization policies
Get-NetVirtualizationProviderAddress
Get-NetVirtualizationCustomerRoute
Get-NetVirtualizationLookupRecord

# Check VXLAN/NVGRE encap on the vSwitch
Get-VMSwitch | Select-Object Name, *Virtualization*
```

> **Note:** In modern SDN deployments with Azure Stack HCI, HNV is managed via Network Controller rather than direct PowerShell.

## 📌 Performance Tuning: NUMA & Processor Compatibility

### NUMA Topology

Non-Uniform Memory Access (NUMA) means each CPU socket has "local" memory that's faster to access than "remote" memory on other sockets.

```
┌─────────────────────────────────────────────┐
│  NUMA Node 0               NUMA Node 1       │
│  ┌─────────────┐          ┌─────────────┐   │
│  │ CPU Socket 0│          │ CPU Socket 1│   │
│  │ 16 cores    │          │ 16 cores    │   │
│  └──────┬──────┘          └──────┬──────┘   │
│         │ Local                   │ Local     │
│  ┌──────┴──────┐          ┌──────┴──────┐   │
│  │  64 GB RAM  │◄─ QPI ──►│  64 GB RAM  │   │
│  │  (fast)     │ (slow)   │  (fast)     │   │
│  └─────────────┘          └─────────────┘   │
└─────────────────────────────────────────────┘
```

```powershell
# Check NUMA topology on the host
Get-VMHostNumaNode | Format-Table NodeId, ProcessorsAvailability, MemoryAvailable, MemoryTotal

# Check VM NUMA spanning setting
Get-VMProcessor -VMName "DC01" | Select-Object VMName, NumaNodesCount, NumaSpanningEnabled

# Disable NUMA spanning (forces VM onto single NUMA node)
Set-VMHost -NumaSpanningEnabled $false

# Pin a VM to a specific NUMA node for best performance
Set-VMProcessor -VMName "SQL-VM" -MaximumCountPerNumaNode 8 -MaximumCountPerNumaSocket 8
```

**Best practice:** For performance-critical VMs (SQL, Exchange), ensure the VM's vCPU and memory fit within a single NUMA node. Check `Get-VMHostNumaNode` to plan.

### Processor Compatibility Mode

```powershell
# Enable processor compatibility (strips advanced CPU features for live migration)
Set-VMProcessor -VMName "DC01" -CompatibilityForMigrationEnabled $true

# Check CPU feature set exposed to VM
Get-VMProcessor -VMName "DC01" | Select-Object CompatibilityForMigrationEnabled, EnableHostResourceProtection
```

## 📌 Advanced: WMI/CIM for Hyper-V Management

The Hyper-V WMI v2 namespace (`root\virtualization\v2`) provides programmatic access to everything.

```powershell
# Get all VM objects via CIM
Get-CimInstance -Namespace "root\virtualization\v2" -ClassName Msvm_ComputerSystem |
    Where-Object { $_.Caption -eq "Virtual Machine" } |
    Select-Object ElementName, EnabledState, HealthState

# Enabled states: 2=Running, 3=Off, 6=Saved, 9=Paused, 32768=Paused-Critical, 32769=Saved-Critical

# Get virtual switch info via CIM
Get-CimInstance -Namespace "root\virtualization\v2" -ClassName Msvm_VirtualEthernetSwitch |
    Select-Object ElementName, Description

# Get VM memory settings
Get-CimInstance -Namespace "root\virtualization\v2" -ClassName Msvm_MemorySettingData |
    Where-Object { $_.InstanceID -like "*DC01*" }

# WMI event subscription for VM state changes (advanced monitoring)
$query = "SELECT * FROM __InstanceModificationEvent WITHIN 5 WHERE TargetInstance ISA 'Msvm_ComputerSystem'"
Register-CimIndicationEvent -Namespace "root\virtualization\v2" -Query $query -Action {
    $vm = $Event.SourceEventArgs.NewEvent.TargetInstance
    Write-Host "VM '$($vm.ElementName)' changed state to $($vm.EnabledState)"
}
```

## 📌 VM State Files Reference

| File | Extension | Purpose |
|------|-----------|---------|
| VM Configuration | `.vmcx` (2016+) / `.xml` (2012R2) | VM settings, hardware config |
| VM Runtime State | `.vmrs` | Saved state (memory contents) |
| Planned VM | `.vmgs` | Guest state (Secure Boot, TPM) |
| Checkpoint Config | `.vmcx` in Snapshots folder | Checkpoint metadata |
| Checkpoint Runtime | `.vmrs` in Snapshots folder | Checkpoint saved state |
| Virtual Hard Disk | `.vhdx` / `.vhd` | VM disk data |
| AVHD/AVHDX | `.avhdx` | Checkpoint differencing disk |
| Smart Paging | `.vsv` | Smart Paging temporary file |
| VM Worker Memory | `.bin` | Worker process memory (legacy) |

```powershell
# Find all files associated with a VM
$vm = Get-VM -Name "DC01"
$vm | Select-Object ConfigurationLocation, SnapshotFileLocation, SmartPagingFilePath, CheckpointFileLocation

# Get all VHD paths for a VM
Get-VMHardDiskDrive -VMName "DC01" | Select-Object ControllerType, ControllerNumber, ControllerLocation, Path

# Default storage paths
Get-VMHost | Select-Object VirtualHardDiskPath, VirtualMachinePath
```
