## 1. S2D Architecture Internals

### Software Storage Bus

The Software Storage Bus is the foundational layer of S2D. It creates a unified
namespace across all physical disks in all cluster nodes, making local disks on
every node accessible to every other node over the network.

```
┌──────────────────────────────────────────────────────────────────┐
│                    SOFTWARE STORAGE BUS                          │
│                                                                  │
│   Node 1                  Node 2                 Node 3          │
│  ┌────────────┐         ┌────────────┐         ┌────────────┐   │
│  │ App / VM   │         │ App / VM   │         │ App / VM   │   │
│  ├────────────┤         ├────────────┤         ├────────────┤   │
│  │ CSV / ReFS │         │ CSV / ReFS │         │ CSV / ReFS │   │
│  ├────────────┤         ├────────────┤         ├────────────┤   │
│  │ Virtual    │         │ Virtual    │         │ Virtual    │   │
│  │ Disk Layer │         │ Disk Layer │         │ Disk Layer │   │
│  ├────────────┤         ├────────────┤         ├────────────┤   │
│  │ Storage    │◄═══════►│ Storage    │◄═══════►│ Storage    │   │
│  │ Bus (local)│  SMB    │ Bus (local)│  SMB    │ Bus (local)│   │
│  ├────────────┤ Direct  ├────────────┤ Direct  ├────────────┤   │
│  │ NVMe  SSD │  (RDMA) │ NVMe  SSD │  (RDMA) │ NVMe  SSD │   │
│  │ HDD   HDD │         │ HDD   HDD │         │ HDD   HDD │   │
│  └────────────┘         └────────────┘         └────────────┘   │
└──────────────────────────────────────────────────────────────────┘
```

### Cache Behavior — Deep Dive

The cache tier absorbs I/O to dramatically improve latency and throughput:

| Scenario | Read Cache | Write Cache | Write Policy |
|----------|-----------|-------------|-------------|
| NVMe (cache) + SSD (capacity) | ❌ No | ✅ Yes | **Write-back** — writes ack after hitting NVMe cache |
| SSD (cache) + HDD (capacity) | ✅ Yes | ✅ Yes | **Write-back** — writes ack after hitting SSD cache |
| NVMe (cache) + HDD (capacity) | ✅ Yes | ✅ Yes | **Write-back** — writes ack after hitting NVMe cache |
| All NVMe or All SSD (no cache) | N/A | N/A | **Write-through** — writes go directly to capacity |

**Why NVMe+SSD skips read cache:** SSDs are already fast enough for reads. Caching SSD
reads on NVMe yields minimal benefit, so NVMe cache is reserved entirely for writes.

**Write-back safety:** Cache writes are mirrored to cache drives on **two different nodes**
before acknowledging to the application. A single cache drive failure does not lose data.

## 2. Fault Domains

Fault domains define the failure boundary hierarchy. S2D uses them to ensure data copies
are placed on separate failure boundaries for maximum resiliency.

### Fault Domain Hierarchy

```
Site (Stretch Cluster)
 └── Rack
      └── Chassis (Enclosure)
           └── Node
                └── Disk
```

### Data Placement Rules

| Resiliency | Minimum Separation | Example |
|------------|-------------------|---------|
| Two-way Mirror | Different **Nodes** | Copy A on Node1, Copy B on Node2 |
| Three-way Mirror | Different **Nodes** | Copies on Node1, Node2, Node3 |
| Dual Parity | Different **Nodes** | Data and parity stripes across ≥7 nodes |
| Mirror (Stretch) | Different **Sites** | Copy A at Site1, Copy B at Site2 |

### Managing Fault Domains

```powershell
# View current fault domain topology
Get-StorageFaultDomain | Sort-Object Type |
    Format-Table FriendlyName, Type, ParentFriendlyName, OperationalStatus

# Create rack-level fault domains (physical rack awareness)
New-StorageFaultDomain -FriendlyName "Rack1" -FaultDomainType Rack
New-StorageFaultDomain -FriendlyName "Rack2" -FaultDomainType Rack

# Assign nodes to racks
Set-StorageFaultDomain -FriendlyName "Node1" -ParentFriendlyName "Rack1"
Set-StorageFaultDomain -FriendlyName "Node2" -ParentFriendlyName "Rack1"
Set-StorageFaultDomain -FriendlyName "Node3" -ParentFriendlyName "Rack2"
Set-StorageFaultDomain -FriendlyName "Node4" -ParentFriendlyName "Rack2"
```

> **Why it matters:** If you define rack fault domains, S2D ensures mirror copies land on
> different racks — surviving an entire rack power failure. Without rack definitions,
> S2D only separates copies across nodes.

## 3. Nested Resiliency

Nested resiliency is designed for **2-node S2D clusters** and provides the ability to
survive simultaneous node failure + disk failure on the surviving node.

### Available Nested Types

| Type | How It Works | Capacity Efficiency | Fault Tolerance |
|------|-------------|--------------------|-----------------| 
| **Nested Two-way Mirror** | Two-way mirror between nodes, then two-way mirror within each node | 25% (4 copies total) | Node + disk failure simultaneously |
| **Nested Mirror-Accelerated Parity** | Two-way mirror between nodes; mirror for hot data, parity for cold data within node | ~35-40% | Node + disk failure; better capacity than pure nested mirror |

### How Nested Resiliency Works

```
Normal Operation (both nodes healthy):
  Node1: [Copy A1] [Copy A2]     ← local mirror within Node1
  Node2: [Copy B1] [Copy B2]     ← local mirror within Node2
                                  ← A and B are mirrors of each other

Node2 fails + 1 disk fails on Node1:
  Node1: [Copy A1] [XXXX]        ← one local copy lost, but A1 still has data
  Node2: [XXXXX]  [XXXXX]        ← entire node offline
  
  ✅ Data survives — Copy A1 on Node1 is still readable
```

```powershell
# Create a nested mirror volume (2-node S2D)
New-Volume -FriendlyName "NestedMirrorVol" `
    -StoragePoolFriendlyName "S2D*" `
    -Size 1TB -FileSystem CSVFS_ReFS `
    -ResiliencySettingName "NestedMirror"

# Create a nested mirror-accelerated parity volume
New-Volume -FriendlyName "NestedMAPVol" `
    -StoragePoolFriendlyName "S2D*" `
    -Size 2TB -FileSystem CSVFS_ReFS `
    -ResiliencySettingName "NestedMirrorAcceleratedParity" `
    -StorageTierFriendlyNames "NestedPerformance", "NestedCapacity" `
    -StorageTierSizes 500GB, 1500GB
```

## 4. Performance Optimization

### Journal Sizing

Each S2D virtual disk has a **write journal** on the cache tier. The journal absorbs
random writes and destages sequentially to capacity.

```powershell
# Check current journal size (per virtual disk)
Get-VirtualDisk | Get-StorageTier |
    Select FriendlyName, @{N='JournalSizeGB';E={$_.WriteCacheSize / 1GB}}

# Increase journal size for write-heavy workloads
Set-StorageTier -FriendlyName "Capacity" -WriteCacheSize 8GB
```

**Guidelines:**
- Default journal: 1 GB per capacity tier per virtual disk
- Write-heavy workloads (SQL TempDB, OLTP): increase to 4-8 GB
- Larger journals improve burst write performance but consume cache drive space

### Delimiter (Separating Cache from Capacity)

When a node has mixed NVMe + SSD + HDD, S2D auto-selects the fastest media as cache.
Use the delimiter model to explicitly control which drives are cache vs capacity.

```powershell
# Enable S2D with explicit cache device model
Enable-ClusterS2D -CacheDeviceModel "INTEL SSDPE2MX012T7"

# Or disable cache entirely (all-flash scenarios)
Enable-ClusterS2D -CacheState Disabled
```

### Benchmark Tools

```powershell
# VMFleet — Microsoft's S2D benchmark tool
# (Run from a management VM with VMFleet installed)
.\Start-Sweep.ps1 -b 4096 -t 8 -o 32 -w 100 -d 300
# -b block size, -t threads, -o outstanding IOs, -w write%, -d duration(s)

# DiskSpd — granular disk performance testing
diskspd.exe -b8K -d60 -o16 -t4 -Sh -w30 -L -c50G C:\ClusterStorage\Volume1\test.dat
```

## 5. S2D Networking

S2D relies heavily on high-speed, low-latency networking between nodes. RDMA is
strongly recommended for production deployments.

### RDMA Technologies

| Technology | Full Name | Key Characteristics |
|-----------|-----------|-------------------|
| **iWARP** | Internet Wide Area RDMA Protocol | Works over standard TCP/IP; tolerates lossy networks; easier to deploy |
| **RoCE v2** | RDMA over Converged Ethernet v2 | Higher performance than iWARP; requires lossless (DCB/PFC) network config |
| **InfiniBand** | InfiniBand | Highest performance; requires dedicated InfiniBand switches (less common) |

### SMB Direct & SMB Multichannel

| Feature | Description |
|---------|-------------|
| **SMB Direct** | SMB protocol using RDMA for zero-copy, kernel-bypass data transfers |
| **SMB Multichannel** | Uses multiple NICs simultaneously for bandwidth aggregation and failover |

### Network Configuration Best Practices

```powershell
# Verify RDMA is enabled on adapters
Get-NetAdapterRdma | Format-Table Name, InterfaceDescription, Enabled

# Enable RDMA on an adapter
Enable-NetAdapterRdma -Name "Storage1"
Enable-NetAdapterRdma -Name "Storage2"

# Configure Switch Embedded Teaming (SET) for Hyper-V + S2D
New-VMSwitch -Name "SET-Switch" -NetAdapterName "NIC1","NIC2" `
    -EnableEmbeddedTeaming $true -AllowManagementOS $true

# Verify SMB Direct (RDMA) is active
Get-SmbConnection | Where-Object RdmaCapable -eq $true
Get-SmbMultichannelConnection | Format-Table ClientName, ServerName, RdmaCapable

# Test RDMA connectivity between nodes
Test-Rdma -IfIndex (Get-NetAdapter -Name "Storage1").InterfaceIndex `
    -IsRoCE $true -RemoteIpAddress 10.0.0.2
```

### Recommended Network Topology

```
┌──────────────────────────────────────────────────────┐
│           S2D Network Configuration                  │
│                                                      │
│  Each Node:                                          │
│   ├── Management NIC (1 GbE) ─── Management VLAN    │
│   ├── Storage NIC 1 (25 GbE, RDMA) ─── Storage VLAN │
│   ├── Storage NIC 2 (25 GbE, RDMA) ─── Storage VLAN │
│   └── VM Traffic (via SET vSwitch)  ─── VM VLANs    │
│                                                      │
│  Minimum: 10 GbE RDMA for storage traffic            │
│  Recommended: 25 GbE RDMA with dedicated storage net │
└──────────────────────────────────────────────────────┘
```

## 6. Storage Replica

Storage Replica provides block-level, volume-based replication between servers or
clusters for disaster recovery and stretch cluster scenarios.

### Replication Modes

| Mode | Description | RPO | Use Case |
|------|-------------|-----|----------|
| **Synchronous** | Writes committed on both ends before ack | RPO = 0 (zero data loss) | Same metro / low latency (<5ms RTT) |
| **Asynchronous** | Writes committed locally, replicated in background | RPO > 0 (some data loss possible) | Cross-region / high latency links |

### Scenarios

| Scenario | Description |
|----------|-------------|
| **Stretch Cluster** | Single failover cluster spanning two sites; automatic failover |
| **Cluster-to-Cluster** | Two independent clusters; manual or orchestrated failover |
| **Server-to-Server** | Two standalone servers; simplest DR setup |

### Key Commands

```powershell
# Create a replication partnership (server-to-server, synchronous)
New-SRPartnership -SourceComputerName "Server1" `
    -SourceRGName "RG1" -SourceVolumeName "E:" -SourceLogVolumeName "F:" `
    -DestinationComputerName "Server2" `
    -DestinationRGName "RG2" -DestinationVolumeName "E:" -DestinationLogVolumeName "F:" `
    -ReplicationMode Synchronous -LogSizeInBytes 8GB

# Check replication status
Get-SRGroup | Format-Table Name, Replicas, ReplicationMode, ReplicationStatus
Get-SRPartnership

# Monitor replication health
(Get-SRGroup).Replicas | Select-Object NumOfBytesRemaining, ReplicationStatus

# Reverse replication direction (planned failover)
Set-SRPartnership -NewSourceComputerName "Server2" `
    -SourceRGName "RG2" -DestinationRGName "RG1"

# Requirements check
Test-SRTopology -SourceComputerName "Server1" -SourceVolumeName "E:" `
    -SourceLogVolumeName "F:" -DestinationComputerName "Server2" `
    -DestinationVolumeName "E:" -DestinationLogVolumeName "F:" `
    -DurationInMinutes 5 -ResultPath "C:\SRResults"
```

**Requirements:**
- Windows Server Datacenter edition
- Minimum 1 GB log volume (recommended ≥ 8 GB)
- NTFS or ReFS formatted volumes
- Synchronous mode: ≤ 5ms round-trip network latency

## 7. Virtual Fibre Channel

Virtual Fibre Channel allows Hyper-V VMs to directly access Fibre Channel SANs using
**NPIV (N_Port ID Virtualization)**.

### How It Works

```
┌───────────────────────────────────────────────┐
│                Hyper-V Host                   │
│                                               │
│  ┌───────────┐   ┌───────────┐               │
│  │   VM 1    │   │   VM 2    │               │
│  │ ┌───────┐ │   │ ┌───────┐ │               │
│  │ │Virtual│ │   │ │Virtual│ │               │
│  │ │FC HBA │ │   │ │FC HBA │ │               │
│  │ │(vHBA) │ │   │ │(vHBA) │ │               │
│  │ └───┬───┘ │   │ └───┬───┘ │               │
│  └─────┼─────┘   └─────┼─────┘               │
│        │               │                     │
│  ┌─────┴───────────────┴─────────────┐       │
│  │   Virtual SAN (Hyper-V vSwitch)   │       │
│  └──────────────┬────────────────────┘       │
│                 │                             │
│  ┌──────────────┴──────────────┐             │
│  │  Physical FC HBA (NPIV)     │             │
│  └──────────────┬──────────────┘             │
└─────────────────┼─────────────────────────────┘
                  │
          ┌───────┴───────┐
          │  FC Switch     │  ◄── SAN Zoning: map each
          │  (SAN Fabric)  │      vHBA WWPN to storage ports
          └───────┬───────┘
                  │
          ┌───────┴───────┐
          │  SAN Storage   │
          └───────────────┘
```

### Key Concepts

| Concept | Description |
|---------|-------------|
| **NPIV** | Allows a single physical FC HBA port to present multiple virtual World Wide Port Names (WWPNs) |
| **Virtual HBA (vHBA)** | A virtual Fibre Channel adapter assigned to a VM; gets its own WWPN pair |
| **Virtual SAN** | A Hyper-V construct mapping virtual FC ports to physical FC HBAs |
| **SAN Zoning** | Must include each VM's virtual WWPN in the zone configuration |

### Configuration

```powershell
# Create a Virtual SAN on the Hyper-V host
$vSan = New-VMSan -Name "VirtualSAN1" `
    -HostBusAdapter (Get-InitiatorPort | Where-Object ConnectionType -eq "Fibre Channel")

# Add a virtual FC adapter to a VM
Add-VMFibreChannelHba -VMName "YOURVM" -SanName "VirtualSAN1"

# View the VM's virtual WWPNs (needed for SAN zoning)
Get-VMFibreChannelHba -VMName "YOURVM" |
    Select VMName, WorldWideNodeNameSetA, WorldWidePortNameSetA,
           WorldWideNodeNameSetB, WorldWidePortNameSetB
```

> **Live Migration:** VMs with virtual FC HBAs can live-migrate. Each vHBA has two WWPN
> sets (Set A and Set B) — one active on the source host, one pre-registered on the
> destination. SAN zones must include both sets for seamless migration.
