# L200: Storage Spaces Troubleshooting
**⏱ Estimated: 10 minutes | 🎯 Goal: Diagnose pool health, disk failures, and rebuild issues**

## 1. Pool Health States

Understanding pool health requires examining both **HealthStatus** and **OperationalStatus**:

| HealthStatus | OperationalStatus | Meaning | Action Required |
|-------------|-------------------|---------|-----------------|
| **Healthy** | OK | All components functioning normally | None |
| **Healthy** | OK | Recently repaired, fully redundant again | None — monitor |
| **Warning** | Degraded | One or more disks failed; redundancy reduced but data still accessible | Replace failed disk, trigger repair |
| **Warning** | Incomplete | Not enough disks online (e.g., node offline in S2D) | Bring node/disks back online |
| **Unhealthy** | Unknown | Pool cannot determine state — possible driver or connectivity issue | Check disk connectivity, storage controller |
| **Unhealthy** | Read-only | Pool lost quorum or too many disks failed; data protected but no writes | Critical — restore disks or nodes ASAP |
| **Unhealthy** | Other | Pool metadata corruption or unrecognized state | Engage Microsoft Support |

```powershell
# Quick health overview
Get-StoragePool | Format-Table FriendlyName, HealthStatus, OperationalStatus, IsReadOnly
Get-VirtualDisk  | Format-Table FriendlyName, HealthStatus, OperationalStatus
Get-PhysicalDisk | Format-Table FriendlyName, HealthStatus, OperationalStatus, Usage
```

## 2. Virtual Disk Degraded / Detached

### Degraded Virtual Disk

A virtual disk becomes **Degraded** when it has lost some redundancy but is still online.

**Common Causes:**
- Physical disk failure (one mirror copy lost)
- Physical disk removed or lost communication
- S2D node offline (its disk copies unavailable)

**Diagnosis:**

```powershell
# Check virtual disk state
Get-VirtualDisk | Where-Object HealthStatus -ne "Healthy"

# Find which physical disks back this virtual disk
Get-VirtualDisk -FriendlyName "MirrorVol" | Get-PhysicalDisk

# Check for unhealthy physical disks
Get-PhysicalDisk | Where-Object HealthStatus -ne "Healthy"
```

**Repair:**

```powershell
# Trigger manual repair (rebuilds data onto available healthy disks)
Repair-VirtualDisk -FriendlyName "MirrorVol"

# Monitor repair progress
Get-StorageJob | Where-Object Name -like "*Repair*"
```

### Detached Virtual Disk

A **Detached** virtual disk is completely offline — the volume is inaccessible.

**Common Causes:**
- Too many simultaneous disk failures exceeding the resiliency tolerance
- Pool quorum lost (majority of disks not available)
- Disk metadata corruption

**Recovery Steps:**

```powershell
# Check operational status
Get-VirtualDisk | Where-Object OperationalStatus -eq "Detached"

# Try to bring it online (if enough disks are now available)
Get-VirtualDisk -FriendlyName "DetachedVol" | Set-VirtualDisk -IsManualAttach $false
Connect-VirtualDisk -FriendlyName "DetachedVol"

# If pool itself is offline, try reconnecting
Get-StoragePool -IsPrimordial $false |
    Where-Object OperationalStatus -ne "OK" |
    Set-StoragePool -IsReadOnly $false
```

## 3. Physical Disk Issues

### Physical Disk States

| HealthStatus | OperationalStatus | Meaning |
|-------------|-------------------|---------|
| **Healthy** | OK | Normal operation |
| **Warning** | Predictive Failure | SMART warnings; disk may fail soon |
| **Warning** | Degraded | Disk responding but errors detected |
| **Unhealthy** | Lost Communication | Disk not responding; may be disconnected |
| **Unhealthy** | Split | Disk metadata split (seen in S2D cluster partition scenarios) |

### Disk Replacement Workflow

```
Step 1: Identify Failed Disk
         │
         ▼
Step 2: Retire the Disk
         │  Set-PhysicalDisk -Usage Retired
         ▼
Step 3: Wait for Data Evacuation
         │  Get-StorageJob (monitor evacuation)
         ▼
Step 4: Physically Remove the Disk
         │
         ▼
Step 5: Insert Replacement Disk
         │
         ▼
Step 6: Add to Pool (auto in S2D, manual in standalone)
         │  Add-PhysicalDisk -StoragePoolFriendlyName "Pool1" -PhysicalDisks (Get-PhysicalDisk -CanPool $true)
         ▼
Step 7: Repair Triggers Automatically
         │  Repair-VirtualDisk or automatic rebuild
         ▼
Step 8: Verify Health
           Get-VirtualDisk | FT FriendlyName, HealthStatus
```

```powershell
# Step 1: Find the bad disk
Get-PhysicalDisk | Where-Object HealthStatus -ne "Healthy" |
    Select FriendlyName, SerialNumber, SlotNumber, HealthStatus, OperationalStatus

# Step 2: Retire the disk (triggers data evacuation)
Set-PhysicalDisk -SerialNumber "WD-ABC123" -Usage Retired

# Step 3: Monitor evacuation
Get-StorageJob | Format-Table Name, JobState, PercentComplete

# Step 6: After physical replacement, add new disk
$newDisk = Get-PhysicalDisk -CanPool $true
Add-PhysicalDisk -StoragePoolFriendlyName "Pool1" -PhysicalDisks $newDisk
```

> **Important:** In S2D, replacement disks are automatically added to the pool if they
> meet the requirements (blank, no partitions). No manual Add-PhysicalDisk needed.

## 4. S2D Rebuild Process

When a disk or node fails, S2D must **rebuild** (re-create) the lost data copies on
remaining healthy disks across the cluster.

### How Rebuild Works

1. S2D detects missing data copies for affected extents
2. Reads surviving copies from other nodes across the Software Storage Bus
3. Writes new copies to available capacity on healthy nodes
4. Continues until full resiliency is restored

### Performance Impact

- Rebuild consumes **network bandwidth** (SMB Direct / RDMA between nodes)
- Rebuild consumes **disk I/O** on both source and destination nodes
- VM workloads may see increased latency during rebuild

### Monitoring Rebuild

```powershell
# Check active storage jobs (rebuild shows as "Repair" or "Regenerate")
Get-StorageJob | Format-Table Name, JobState, PercentComplete, BytesProcessed, BytesTotal

# Detailed view with estimated time
Get-StorageJob | Format-List *

# Check virtual disk repair status
Get-VirtualDisk | Where-Object OperationalStatus -eq "InService" |
    Select FriendlyName, OperationalStatus, HealthStatus
```

### Rebuild Priority

S2D prioritizes rebuild based on urgency:
- **Critical:** Virtual disk has zero redundancy remaining (one more failure = data loss)
- **Normal:** Virtual disk has reduced redundancy but can survive another failure
- Background I/O throttling adjusts based on workload pressure

## 5. Azure Shared Disk Troubleshooting

### Common Issues

| Issue | Symptom | Root Cause | Resolution |
|-------|---------|-----------|------------|
| **Disk not visible on 2nd VM** | Disk Management shows no shared disk | maxShares not set or VM not attached | Set maxShares ≥ 2, attach disk to VM |
| **SCSI PR failures** | Cluster resource fails to come online | Guest OS doesn't support SCSI PR | Use Windows Server 2019+ or supported Linux |
| **Reservation conflicts** | Disk goes offline on one node | Conflicting reservations from stale cluster state | Clear reservations, restart cluster service |
| **Exceeding maxShares** | Attach fails in Azure portal | Disk already attached to maximum VMs | Detach from one VM or increase disk size tier |

### SCSI Persistent Reservation Diagnostics

```powershell
# Check if SCSI PR is supported on the disk
Get-Disk | Select Number, FriendlyName, BusType, SerialNumber

# Verify cluster disk resource status
Get-ClusterResource | Where-Object ResourceType -eq "Physical Disk" |
    Get-ClusterParameter | Where-Object Name -like "*DiskSignature*"

# Check Event Viewer for SCSI PR errors
Get-WinEvent -FilterHashtable @{LogName='System'; ProviderName='disk'} -MaxEvents 20
```

### maxShares Configuration

```bash
# Azure CLI: Check current maxShares
az disk show -n MyDisk -g MyRG --query "maxShares"

# Azure CLI: Update maxShares
az disk update -n MyDisk -g MyRG --max-shares 3
```

## 6. Clustered Azure VM Storage

### Key Considerations

| Setting | Recommendation for Clusters | Why |
|---------|---------------------------|-----|
| **Host Caching** | None or ReadOnly for shared disks | ReadWrite caching can cause data inconsistency across nodes |
| **Temp Disk** | Do NOT use for cluster data | Temp disk is ephemeral; lost on VM redeployment |
| **Write Accelerator** | Only for M-series VMs with Premium storage | Reduces write latency for transaction logs |
| **Availability** | Use Availability Zones or Proximity Placement Groups | Ensures VMs can access shared disk with low latency |

```powershell
# Verify disk caching on Azure VM (from inside the VM)
Get-WmiObject -Class Win32_DiskDrive | Select Model, InterfaceType, MediaType

# Check disk type and caching via Azure CLI
az vm show -n MyVM -g MyRG --query "storageProfile.dataDisks[].{name:name, caching:caching, diskSizeGb:diskSizeGb}"
```

> **Common pitfall:** Setting Host Caching to **ReadWrite** on a shared disk causes
> silent data corruption because each VM's cache is independent — writes from one VM
> won't be visible in another VM's cache.

## 7. Diagnostic Commands

```powershell
# --- Storage Jobs (rebuild, repair, optimization) ---
Get-StorageJob | Format-Table Name, JobState, PercentComplete, ElapsedTime

# --- Fault Domains (S2D topology) ---
Get-StorageFaultDomain | Format-Table FriendlyName, Type, OperationalStatus, HealthStatus

# --- Comprehensive Diagnostics ---
# Captures logs, event data, and cluster health into a ZIP
Get-StorageDiagnosticInfo -DestinationPath "C:\StorageDiag" -IncludeLiveDump

# --- Debug Storage Subsystem (deep diagnostics) ---
$subsystem = Get-StorageSubSystem -FriendlyName "Clustered Windows Storage*"
Debug-StorageSubSystem -StorageSubSystem $subsystem

# --- Quick S2D Health Report ---
Get-HealthFault                                     # Current active faults
Get-StorageSubSystem -FriendlyName *Cluster* |
    Get-StorageHealthReport                         # Overall health metrics

# --- Event Log Queries ---
Get-WinEvent -FilterHashtable @{
    LogName = 'Microsoft-Windows-StorageSpaces-Driver/Operational'
    Level   = 2,3   # Error + Warning
} -MaxEvents 50 | Format-Table TimeCreated, Id, Message -Wrap
```
