## 1. Storage Spaces Overview

Storage Spaces is a Windows storage virtualization technology that pools physical disks
into logical storage constructs, providing flexibility, resiliency, and efficiency.

```
┌─────────────────────────────────────────────────────────────┐
│                     STORAGE SPACES ARCHITECTURE             │
│                                                             │
│   Physical Disks         Storage Pool        Virtual Disks  │
│  ┌──────────┐                                  (Spaces)     │
│  │ Disk 0   │──┐                             ┌───────────┐  │
│  └──────────┘  │      ┌──────────────┐       │ VDisk A   │──┼──► Volume E:
│  ┌──────────┐  ├─────►│              │──────►│ (Mirror)  │  │
│  │ Disk 1   │──┤      │  Storage     │       └───────────┘  │
│  └──────────┘  │      │  Pool        │       ┌───────────┐  │
│  ┌──────────┐  │      │              │──────►│ VDisk B   │──┼──► Volume F:
│  │ Disk 2   │──┤      │  (Aggregated │       │ (Parity)  │  │
│  └──────────┘  │      │   raw        │       └───────────┘  │
│  ┌──────────┐  │      │   capacity)  │       ┌───────────┐  │
│  │ Disk 3   │──┘      │              │──────►│ VDisk C   │──┼──► Volume G:
│  └──────────┘         └──────────────┘       │ (Simple)  │  │
│                                              └───────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**Key Concepts:**

| Concept | Description |
|---------|-------------|
| **Storage Pool** | Aggregation of physical disks into a single logical pool of raw capacity |
| **Virtual Disk (Space)** | Carved from a pool with a chosen resiliency type; appears as a disk to Windows |
| **Volume** | A formatted file system (NTFS/ReFS) created on top of a virtual disk |

> **Why it matters:** Support cases often confuse virtual disks with volumes. A degraded
> virtual disk affects all volumes on it, but a pool can have multiple healthy virtual disks
> even if one is degraded.

## 2. Resiliency Types

| Resiliency | How It Works | Space Efficiency | Fault Tolerance | Performance | Min Disks |
|------------|-------------|-----------------|-----------------|-------------|-----------|
| **Simple** | Striping across disks, no redundancy | 100% | None — any disk loss = data loss | Best read/write throughput | 1 |
| **Two-way Mirror** | Two copies of every data stripe | 50% | Survives 1 disk failure | Good read (2×), write overhead | 2 |
| **Three-way Mirror** | Three copies of every data stripe | 33% | Survives 2 simultaneous disk failures | Best read (3×), most write overhead | 5 (S2D) |
| **Single Parity** | Striping + 1 parity stripe (like RAID 5) | 67-80% (depends on columns) | Survives 1 disk failure | Good sequential read, slow random write | 3 |
| **Dual Parity** | Striping + 2 parity stripes (like RAID 6) | 50-67% | Survives 2 disk failures | Good sequential, slower random write | 7 (S2D) |

> **Tip:** For production workloads, Microsoft recommends **mirror** for most use cases.
> Parity is suitable for archival/sequential workloads where capacity efficiency matters.

## 3. Tiered Storage

Tiered storage automatically moves frequently accessed ("hot") data to faster media (SSD)
and infrequently accessed ("cold") data to slower, higher-capacity media (HDD).

**How Tiering Works:**
- A **heat map** tracks read/write frequency per 1 MB data slab
- A scheduled optimization task (default: 1 AM daily) moves slabs between tiers
- You can **pin** critical files permanently to the SSD tier

**Key Commands:**

```powershell
# Create storage tiers
New-StorageTier -StoragePoolFriendlyName "Pool1" -FriendlyName "SSD_Tier" -MediaType SSD
New-StorageTier -StoragePoolFriendlyName "Pool1" -FriendlyName "HDD_Tier" -MediaType HDD

# Create a tiered virtual disk
New-VirtualDisk -StoragePoolFriendlyName "Pool1" -FriendlyName "TieredDisk" `
    -StorageTiers (Get-StorageTier -FriendlyName "SSD_Tier"),
                  (Get-StorageTier -FriendlyName "HDD_Tier") `
    -StorageTierSizes 200GB, 1800GB

# Pin a file to the SSD tier
Set-FileStorageTier -FilePath "D:\Databases\critical.mdf" `
    -DesiredStorageTierFriendlyName "SSD_Tier"
```

## 4. Storage Spaces Direct (S2D)

Storage Spaces Direct (S2D) uses local attached disks in cluster nodes to create a
shared, software-defined storage pool — no external SAN/NAS required.

```
┌──────────────────────────────────────────────────────────────────┐
│                   STORAGE SPACES DIRECT (S2D)                    │
│                                                                  │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐            │
│  │   Node 1    │   │   Node 2    │   │   Node 3    │            │
│  │ ┌───┐ ┌───┐ │   │ ┌───┐ ┌───┐ │   │ ┌───┐ ┌───┐ │            │
│  │ │NVM│ │SSD│ │   │ │NVM│ │SSD│ │   │ │NVM│ │SSD│ │            │
│  │ └───┘ └───┘ │   │ └───┘ └───┘ │   │ └───┘ └───┘ │            │
│  │ ┌───┐ ┌───┐ │   │ ┌───┐ ┌───┐ │   │ ┌───┐ ┌───┐ │            │
│  │ │HDD│ │HDD│ │   │ │HDD│ │HDD│ │   │ │HDD│ │HDD│ │            │
│  │ └───┘ └───┘ │   │ └───┘ └───┘ │   │ └───┘ └───┘ │            │
│  └──────┬──────┘   └──────┬──────┘   └──────┬──────┘            │
│         │                 │                 │                    │
│         └────────┬────────┴────────┬────────┘                    │
│                  ▼                 ▼                              │
│        ┌──────────────────────────────────┐                      │
│        │     Software Storage Bus          │  ◄── SMB over RDMA  │
│        └──────────────┬───────────────────┘                      │
│                       ▼                                          │
│        ┌──────────────────────────────────┐                      │
│        │     Unified Storage Pool          │                     │
│        └──────────────┬───────────────────┘                      │
│                       ▼                                          │
│        ┌──────────────────────────────────┐                      │
│        │   Virtual Disks (CSV Volumes)     │                     │
│        └──────────────────────────────────┘                      │
└──────────────────────────────────────────────────────────────────┘
```

**Deployment Models:**

| Model | Description | Use Case |
|-------|-------------|----------|
| **Hyper-converged** | Compute + Storage on same nodes; VMs run directly on S2D cluster | Most common; simpler management |
| **Converged (Disaggregated)** | Separate S2D storage cluster provides storage via SMB3 to compute cluster | Scale compute and storage independently |

## 5. S2D Components

### Cache Drives vs Capacity Drives

| Component | Media Type | Role | Behavior |
|-----------|-----------|------|----------|
| **Cache** | NVMe or SSD | Absorbs writes, caches hot reads | Bound to capacity drives on same node |
| **Capacity** | SSD or HDD | Stores persistent data copies | Data is mirrored/paritied across nodes |

### Cache Behavior

| Capacity Media | Cache Media | Read Cache | Write Cache |
|---------------|-------------|------------|-------------|
| HDD | SSD | ✅ Yes | ✅ Yes (write-back) |
| HDD | NVMe | ✅ Yes | ✅ Yes (write-back) |
| SSD | NVMe | ❌ No | ✅ Yes (write-back) |
| All same type | (none) | N/A | Write-through (no caching) |

> **Key insight:** When all drives are the same media type (e.g., all NVMe), S2D does not
> designate cache drives. All drives serve as capacity with write-through semantics.

## 6. Azure Shared Disks

Azure Shared Disks allow a single managed disk to be attached to multiple Azure VMs
simultaneously, enabling lift-and-shift of clustered applications.

**Key Facts:**

| Feature | Details |
|---------|---------|
| **Protocol** | SCSI Persistent Reservations (SCSI PR) for coordinated disk access |
| **Supported Types** | Premium SSD, Ultra Disk, Premium SSD v2 |
| **maxShares** | Controls how many VMs can attach (max varies by disk size and type) |
| **Use Cases** | SQL FCI, SAP ASCS/SCS, Windows Failover Cluster in Azure |

**SCSI Persistent Reservations:**
Clusters use SCSI PR commands to coordinate which node owns a disk resource.
These are the same reservation commands used by on-premises SANs.

```powershell
# Verify shared disk is visible on both cluster nodes
Get-Disk | Where-Object { $_.BusType -eq 'SAS' -or $_.BusType -eq 'SCSI' }

# Check maxShares on the Azure disk (Azure CLI)
az disk show --name MySharedDisk --resource-group MyRG --query "maxShares"
```

## 7. Essential Commands

```powershell
# --- Pool Management ---
Get-StoragePool                                    # List all pools
New-StoragePool -FriendlyName "Pool1" `
    -StorageSubSystemFriendlyName "Windows Storage*" `
    -PhysicalDisks (Get-PhysicalDisk -CanPool $true)

# --- Virtual Disk Management ---
Get-VirtualDisk                                    # List all virtual disks
New-VirtualDisk -StoragePoolFriendlyName "Pool1" `
    -FriendlyName "Mirror1" -ResiliencySettingName Mirror `
    -Size 500GB

# --- Physical Disk Inspection ---
Get-PhysicalDisk                                   # List all physical disks
Get-PhysicalDisk | Select FriendlyName, MediaType, HealthStatus, OperationalStatus

# --- Enable S2D ---
Enable-ClusterS2D                                  # Enable S2D on a failover cluster
Enable-ClusterS2D -CacheState Disabled             # All-flash, no cache tier

# --- Quick Health Check ---
Get-StoragePool | Select FriendlyName, HealthStatus, OperationalStatus
Get-VirtualDisk  | Select FriendlyName, HealthStatus, OperationalStatus, ResiliencySettingName
```
