## 📌 WFP (Windows Filtering Platform) Architecture

```
User Mode                          Kernel Mode
┌────────────┐                    ┌─────────────────┐
│ Application│ ──── Winsock ────→ │  AFD.sys         │
└────────────┘                    │  (socket driver) │
                                  └───────┬─────────┘
┌────────────┐                            ↓
│  BFE       │ ←── Policy ──────→ ┌─────────────────┐
│  Service   │                    │  netio.sys       │
│ (manages   │                    │  (WFP engine)    │
│  filters)  │                    │                  │
└────────────┘                    │  ┌─────────────┐ │
                                  │  │ Filter Layer │ │
┌────────────┐                    │  │ - ALE layers │ │
│  MPSSVC    │ ── Registers ───→  │  │ - Transport  │ │
│  (Firewall │    Filters         │  │ - Network    │ │
│   service) │                    │  │ - Forward    │ │
└────────────┘                    │  └─────────────┘ │
                                  └─────────────────┘
```

**Key Layers in WFP:**
- **ALE (Application Layer Enforcement)**: Matches traffic to applications
- **Transport Layer**: TCP/UDP level filtering
- **Network Layer**: IP level filtering
- **Forward Layer**: Routing/forwarding decisions

## 📌 Advanced WFP Diagnostics

### Dump All WFP Filters
```cmd
netsh wfp show filters
# Creates: filters.xml — contains ALL active WFP filters (can be thousands)
```

### Dump Current WFP State
```cmd
netsh wfp show state
# Creates: wfpstate.xml — shows callout drivers, providers, current connections
```

### Capture Real-Time WFP Events
```cmd
netsh wfp capture start
# Reproduce the issue
netsh wfp capture stop
# Creates: wfpdiag.cab containing wfpdiag.xml
```

### Analyze WFP Capture
In `wfpdiag.xml`, look for:
```xml
<classifyDrop>
    <filterId>12345</filterId>        <!-- Which filter caused the drop -->
    <layerId>FWPM_LAYER_ALE_AUTH_RECV_ACCEPT_V4</layerId>  <!-- Which layer -->
    <direction>inbound</direction>
</classifyDrop>
```

Then cross-reference `filterId` with the filters.xml to find the exact rule.

## 📌 Complex GPO Scenarios

### GPO Priority and Merge Behavior
```
Priority (highest to lowest):
1. Local Group Policy (gpedit.msc)
2. Site-level GPO
3. Domain-level GPO  
4. OU-level GPO (most specific OU wins)

Within same level:
- "Block" rules take precedence over "Allow" rules
- More specific rules take precedence over less specific
```

### GPO Setting: "Apply local firewall rules"
```
Computer Configuration > Windows Settings > Security Settings > 
  Windows Defender Firewall > Domain Profile > 
    "Apply local firewall rules: No"
```
**Effect**: ALL locally created rules are **ignored**. Only GPO rules apply.
This is a **very common cause** of "I added a rule but traffic is still blocked."

### Diagnosing GPO Conflicts
```powershell
# Export effective firewall policy
netsh advfirewall export "C:\temp\firewall-policy.wfw"

# Compare with expected rules
Get-NetFirewallRule -PolicyStoreSource GroupPolicy |
    Select DisplayName, Direction, Action, Profile, Enabled |
    Export-Csv C:\temp\gpo-rules.csv

Get-NetFirewallRule -PolicyStoreSource PersistentStore |
    Select DisplayName, Direction, Action, Profile, Enabled |
    Export-Csv C:\temp\local-rules.csv
```

## 📌 Connection Security Rules (IPsec)

Connection Security Rules ≠ Firewall Rules. They enforce **authentication** between machines.

```powershell
# List connection security rules
Get-NetIPsecRule | Format-Table DisplayName, Enabled, InboundSecurity, OutboundSecurity

# Common issue: "Require authentication" rule blocks traffic from
# machines that don't support IPsec / don't have matching certificates
```

**Troubleshooting IPsec:**
```powershell
# Check IPsec Main Mode SA (Security Associations)
Get-NetIPsecMainModeSA

# Check Quick Mode SA
Get-NetIPsecQuickModeSA

# If empty → IPsec negotiation failed
# Check Event Log: Security > Event ID 4653, 4654 (IPsec audit)
```

## 📌 Edge Cases You'll See in Cases

### 1. Third-Party Security Software Conflict
- Products like antivirus may install their own WFP callout drivers
- These run at a **lower layer** than Windows Firewall
- Can block traffic even when WF rules say "Allow"
- **Diagnosis**: `netsh wfp show state` → look for non-Microsoft providers

### 2. Hyper-V Virtual Switch Filtering
- Hyper-V VMs have their own firewall context
- VMSwitch extension may interfere with firewall
- Port ACLs on VMSwitch are separate from WDF rules

### 3. Windows Firewall Reset
```powershell
# Nuclear option: reset firewall to factory defaults
netsh advfirewall reset
# ⚠️ This removes ALL custom rules (local + GPO will re-apply)
```

### 4. Corrupt Firewall Configuration
```powershell
# Symptoms: Firewall service crash, rules not loading
# Fix attempt 1: Reset
netsh advfirewall reset

# Fix attempt 2: Re-register DLLs
regsvr32 firewallapi.dll

# Fix attempt 3: SFC/DISM
sfc /scannow
DISM /Online /Cleanup-Image /RestoreHealth
```
